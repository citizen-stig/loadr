== Test Data and the Environment

_The two things that quietly wreck most load tests: unrealistic data and an unrepresentative environment._

A team I worked with once load-tested a product catalog API for three weeks
before a marketing launch. Every run came back green: 14ms at p99, headroom for
five times the projected traffic. Launch night, the API hit 1,200ms at p95
within twenty minutes and the on-call engineer spent the evening restarting
database replicas.

The test wasn't badly built. The scenario shape was right, the load model was
an honest open model, the SLO gate was exactly what <<ch04>> prescribes. The
test lied anyway, for two reasons that had nothing to do with load. Every
request in the test fetched one of *twelve* product IDs, so the entire working
set sat in the CDN, the application cache, and the database buffer pool
simultaneously — the test measured a system with a 100 percent cache hit rate,
and production, with 400,000 live SKUs and real customers, ran closer to 60
percent. And the test environment's database held 50,000 rows in the table
that production held 80 million in, so the query planner in the test chose a
plan that simply did not exist in production.

Nothing about more virtual users, cleverer ramp shapes, or better percentile
math would have caught this. The request rate was realistic; the *requests
themselves* were not, and neither was the system receiving them. This chapter
is about those two dimensions: making the data your test sends look like the
data production receives, and making the environment your test hits behave
like the environment your users hit. Between them they account for more
quietly wrong load tests than every other cause in this book combined.

=== The single-user lie

Start with the most common version of the problem: the test that logs in as
`testuser1` and does everything as `testuser1`, ten thousand times.

Modern systems are caches stacked on caches. A CDN caches responses. The
application caches sessions and rendered fragments. The ORM caches query
results. The database caches pages in its buffer pool and caches *query plans*
keyed on parameter statistics. Even the CPU is caching your one user's row in
L2. Send the same user and the same handful of IDs through all of that and
every layer converges to its best case at once. You are no longer testing the
system; you are testing the memo table in front of it.

There are second-order effects too. One user means one session row, so a
locking read on the session table serializes on a single hot row — you can
manufacture contention that production never sees, or hide contention it does.
One user means one rate-limit bucket, one shard, one tenant. Whatever your
system partitions by, a single-key test exercises exactly one partition.

The fix is *parameterization*: externalize the varying values into a data
source and have each iteration draw fresh ones. Every serious tool has this;
the vocabulary differs (JMeter calls the file a CSV Data Set Config, Gatling
calls it a feeder, k6 makes you build it from a `SharedArray`), the mechanism
is the same. In loadr it's a `data:` block:

[source,yaml]
----
# Each iteration takes the next row from users.csv.
data:
  users:
    type: csv
    path: data/users.csv
    mode: shared        # one cursor for the whole run; VUs pull rows atomically
    on_eof: recycle     # wrap at end-of-file (stop: retire the VU instead)

scenarios:
  login_flow:
    executor: per-vu-iterations
    vus: 5
    iterations: 20
    flow:
      - request:
          name: login
          method: POST
          url: /login
          body:
            form:
              username: ${data.users.username}
              password: ${data.users.password}
          assert:
            - { type: status, equals: 200 }
----

Two knobs on that block encode real modeling decisions, not plumbing.

*`mode`* decides who owns the cursor. `shared` gives the whole run one cursor:
rows are spread across VUs and each row is used once per lap — the right shape
when rows represent units of work to distribute (accounts to bill, orders to
process). `per_vu` gives every VU its own pass through the full file from the
top — the right shape when a row *is* an identity and each simulated user
should stay one person for their whole session.

*`pick`* decides the access pattern, and this is where realism lives:

[source,yaml]
----
data:
  users:
    type: csv
    path: data/users.csv
    mode: per_vu
    pick: shuffle       # sequential (default) | random | shuffle
    on_eof: recycle
  skus:
    type: json
    path: data/skus.json    # [ { "sku": "W-1", "name": "Widget" }, ... ]
    pick: random            # a uniformly random row every reference; never exhausts
----

`sequential` walks the file in order — deterministic, reproducible, and the
kindest possible pattern for every cache and B-tree in the path, because
consecutive requests touch adjacent keys. `random` draws uniformly — which is
usually *too* cruel, because real traffic is never uniform: real product views,
real user logins, real anything follow a heavily skewed distribution where a
small head of keys gets most of the traffic (the pattern is roughly Zipfian,
and it's why production caches work at all). `shuffle` sits between: full
coverage, no adjacency. None of these is "correct" in the abstract. The correct
choice is the one that reproduces *your* production key distribution — which
you can measure from access logs, and which we treated as part of workload
modeling in <<ch03>>. If your production cache hit rate is 85 percent and your
test's is 99 or 40, your latency numbers are fiction either way.

[TIP]
====
The single highest-leverage realism check you can run costs five minutes:
compare cache hit rates (CDN, app cache, DB buffer cache) between a test run
and a normal production hour. If they don't roughly match, fix the data before
you trust another number the test produces.
====

For values that don't need to correlate with anything — request IDs,
idempotency keys, random emails for a signup flow — you don't need a file at
all. Generate them inline:

[source,yaml]
----
- request:
    method: POST
    url: /signup
    headers:
      X-Request-Id: "${js: crypto.uuidv4()}"
    body:
      json:
        email: "user-${vu}-${iteration}@example.test"
----

And for data that lives in a database — the common enterprise case where the
only source of truth for "valid account numbers" is production's replica — the
robust pattern is to export at build time: a query runs in CI, writes a CSV,
and the test consumes the file. It keeps the load test itself free of database
credentials and makes every run's input set an inspectable artifact. Gatling
offers `jdbcFeeder` and JMeter a JDBC config element if you want the live
connection, but a checked-in-or-generated CSV is easier to reason about, diff,
and replay.

The same concept in k6, for portability — the data lives in a `SharedArray` so
all VUs share one parsed copy in memory:

[source,javascript]
----
import { SharedArray } from 'k6/data';
import papaparse from 'https://jslib.k6.io/papaparse/5.1.1/index.js';

const users = new SharedArray('users', () =>
  papaparse.parse(open('./data/users.csv'), { header: true }).data
);

export default function () {
  const user = users[Math.floor(Math.random() * users.length)];
  // ... use user.username, user.password
}
----

Note what k6 makes you decide in code (the pick strategy is that
`Math.random()` line) and JMeter makes you decide in checkboxes ("Recycle on
EOF?", "Stop thread on EOF?"). Same knobs everywhere. Learn the knobs, not the
spelling.

=== Correlation: the values you can't put in a file

Parameterization covers the values you know before the test starts. A second
class of values doesn't exist until the system under test creates them:
session tokens, CSRF tokens, cart IDs, order numbers, `Location` headers.
Capturing one of these from a response and feeding it into a later request is
called *correlation*, and it is the difference between a script that exercises
your system and a script that exercises your error handler.

The classic failure comes from record-and-replay tooling. You record a
checkout flow in a browser; the recording faithfully captures the CSRF token
the server issued *during the recording*. Replay it at 500 iterations per
second and every iteration submits that same, long-expired token. The server
rejects all of them — often *faster* than it would process a real checkout,
because token validation is cheap and order placement is not. The test reports
splendid latency, and what it measured was the speed of your rejection path.

[WARNING]
====
A load test whose requests are being rejected can look *better* than a
healthy one: error paths frequently skip the expensive work. Never trust a
latency number without its companion error rate and a positive functional
check on the response body. "Fast" and "fast because it's failing" are
indistinguishable on a latency chart.
====

In loadr, correlation is an `extract:` list on the producing request; the
named values become `${...}` variables for every later step in the iteration:

[source,yaml]
----
scenarios:
  checkout:
    executor: constant-vus
    vus: 3
    duration: 1m
    flow:
      - request:
          name: form page
          url: /checkout/start
          extract:
            # CSRF token hidden in the HTML form
            - { type: css, name: csrf, expression: "input[name=csrf]", attribute: value }
            # A response header
            - { type: header, name: request_id, header: X-Request-Id }
      - request:
          name: submit
          method: POST
          url: /checkout/submit
          headers:
            X-Parent-Request: ${request_id}
          body:
            form:
              csrf: ${csrf}
          extract:
            - { type: jsonpath, name: order_id, expression: "$.order.id" }
          assert:
            - { type: status, equals: 200 }
      - request:
          name: order status
          url: /orders/${order_id}
          checks:
            - { type: jsonpath, name: order is pending, expression: "$.status", equals: pending }
----

The extractor types map one-to-one onto what you'll find elsewhere: `jsonpath`
for JSON bodies, `regex` and JMeter-style `boundary` (left/right delimiters)
for text, `css` and `xpath` for markup, `header` for headers. JMeter's
Regular Expression Extractor and Boundary Extractor, Gatling's `css(...)` and
`jsonPath(...)` checks with `saveAs`, k6's `res.json()` and `res.html()` — all
the same move.

One default matters enormously: in loadr, an extraction that matches nothing
(and has no `default:`) *marks the request failed*. That is the behavior you
want. The alternative — a silently unset variable flowing into the next
request as a literal `${order_id}` — produces exactly the rejected-traffic
scenario above, at full speed, with green latency. If your tool of choice
fails soft here, add an explicit assertion after every extraction that the
variable is non-empty.

Two special cases of correlation are common enough to be solved features
rather than scripting exercises.

*Cookies.* Server-side sessions ride on `Set-Cookie`, and a per-VU cookie jar
handles them without any extraction at all — each virtual user holds its own
cookies, exactly like a browser. In loadr the jar is on by default:

[source,yaml]
----
defaults:
  http:
    base_url: https://api.example.com
    cookies: true          # default; set false to ignore cookies entirely

scenarios:
  session:
    executor: constant-vus
    vus: 5
    duration: 30s
    flow:
      - request:            # Set-Cookie from the login lands in this VU's jar…
          name: login
          method: POST
          url: /login
          body: { form: { username: demo, password: demo } }
      - request:            # …and is replayed automatically here.
          name: dashboard
          url: /dashboard
          checks: [ { type: status, equals: 200 } ]
----

The trap to check for: some tools (and some configurations) share one jar
across all VUs, which turns your thousand simulated users back into one
session — the single-user lie sneaking in through the side door.

*Auth tokens.* Token-based auth adds a lifecycle problem: tokens expire, and
a soak test (see <<ch01,the taxonomy>>) will outlive them. The clean shape is
per-VU acquisition plus refresh — log in once per virtual user, not once per
iteration (hammering your identity provider with a login per request is its
own unrealistic workload), and re-mint before expiry. loadr expresses the
per-VU part with an `on_start` hook, one of the lifecycle seams borrowed from
Locust:

[source,yaml]
----
js:
  file: scripts/auth.js

scenarios:
  authed_traffic:
    executor: constant-vus
    vus: 50
    duration: 2h            # a soak — tokens *will* expire during this
    on_start: login         # runs once per VU, before its first iteration
    exec: browse            # runs once per iteration
----

[source,javascript]
----
// scripts/auth.js
import http from 'k6/http';

export function login() {
  const res = http.post('/auth/login', JSON.stringify({ pw: __ENV.PW }));
  session.vars.token = res.json().token;   // per-VU state
}
export function browse() {
  http.get('/feed', { headers: { Authorization: `Bearer ${session.vars.token}` } });
}
----

=== Cardinality and volume: the data behind the API

So far we've talked about the data the test *sends*. The data the system
*holds* matters just as much, and it fails in a sneakier way, because the test
file can be perfect and the result still fiction.

Databases are the sharpest example. A query planner chooses plans based on
table statistics. Against the 50,000-row staging table, the planner in my
opening story chose a sequential scan — and a sequential scan of 50,000 rows
is genuinely fast, so the test was *correctly* reporting that the test
database was quick. Against 80 million rows the same query wanted an index
that could no longer fit its hot pages in memory. This isn't a linear
slowdown you can extrapolate through; it's a different algorithm. B-tree
depth grows with volume, joins flip strategies at cardinality thresholds,
vacuum and compaction behavior changes character. A performance test against
a near-empty database is a functional test with extra steps.

Cardinality — how many *distinct* values, not just how many rows — has its
own effects:

* *Too few distinct keys in the test* → cache hit rates soar, hot-row lock
  contention appears (or disappears) artificially, one shard takes all the
  traffic. Twelve SKUs in the feeder was exactly this.
* *Too many, too uniform* → every request is a cold miss, and you'll
  conclude you need triple the hardware production actually needs.
* *Unique-per-request values where production repeats them* → a cache keyed
  on that value grows without bound. I've watched a test with a fresh UUID in
  a query string flush a CDN's cache of real content — the test was a cache
  eviction attack on itself.

The discipline is straightforward and unglamorous: pull the real numbers.
Row counts per table from production (`n_live_tup` in Postgres's
`pg_stat_user_tables`, `information_schema.TABLES` in MySQL), distinct-value
counts for the keys your test varies, and the top-N key frequency from an
hour of access logs. Then seed the test environment to the same order of
magnitude and shape. You don't need production's 80 million rows to the
digit; you need the same query plans and a comparable buffer-cache hit
ratio, and you should *verify* both, not assume them — `EXPLAIN` the hot
queries in both environments and compare.

=== The environment: a scale model that stays honest

Suppose the data is right — the feeder has production cardinality, the
database has production volume. The machine the test runs against still has
to resemble the machines users hit.

The ideal is boring: same instance types, same topology, same configuration,
same TLS termination, same CDN in front (or deliberately not, with the test
aimed past it — decide, don't drift into it). Configuration parity is the
part teams skip: a staging environment with debug logging on, a 10-connection
database pool where production has 200, or half the feature flags in the
other position is a different system wearing the same code. Keep the
environment definition in the same infrastructure-as-code as production and
diff them mechanically; "we think it's the same" has cost the industry more
launch nights than any load generator bug.

Since one test will run against several targets over its life, keep the
scenario constant and vary only the target. loadr's `env:` overlays deep-merge
over the document, so the delta between environments is explicit and tiny:

[source,yaml]
----
defaults:
  http: { base_url: https://prod.example.com, timeout: 10s }

scenarios:
  api:
    executor: constant-vus
    vus: 50
    duration: 10m
    flow:
      - request: { url: /api/products }

env:
  staging:
    defaults:
      http:
        base_url: https://staging.example.com   # only this key changes
  ci:
    scenarios:
      api: { vus: 1, duration: 10s }            # tiny smoke in CI
----

[source,console]
----
$ loadr run -e staging test.yaml
----

==== The one-tenth-environment trap

The pressure you will actually face is cost: "we can't afford a copy of
production, so we built it at one-tenth scale — run the test at one-tenth
load and multiply." It sounds proportional. It almost never is, because
systems are not made of parts that all scale the same way:

[cols="1,2",options="header"]
|===
| Component | What happens at one-tenth scale
| Stateless app tier | Scales down honestly — ten pods behave like a hundred, per-pod
| Database | Usually still *one* primary in both environments; at one-tenth load it's loafing, so the component most likely to be production's bottleneck is the one you've exempted from the test
| Caches | Hit rate depends on working-set-size versus cache-size, a ratio you've changed twice over; it does not scale linearly in either direction
| Connection pools, file handles, rate limits | Fixed numbers someone configured; they don't know about your scaling factor
| Third-party APIs | Same external rate limit either way
| Network | Latency doesn't scale down; NAT tables, load-balancer connection limits, and ephemeral-port exhaustion appear at absolute thresholds
|===

The deeper problem is that the interesting behavior — the knee where latency
turns upward, the point where a queue starts growing — happens at *absolute*
thresholds, not relative ones. A system at one-tenth scale has its knees in
different places, and multiplying the results by ten tells you nothing about
where the full-scale knees are. Scaled-down results interpolate; they do not
extrapolate.

What to do instead, in order of preference: build the stateful spine
(database, cache) at full size even if the stateless tier is smaller — that's
where the money already went and where the surprises live; run short-lived
full-scale environments on demand (an hour of a production-sized environment
during a test window costs a fraction of a permanently idle replica, and
<<ch14>> automates exactly this); or test carefully against production
itself, which sounds reckless and is by <<ch15>> a mainstream, guard-railed
practice. What you should not do is run the one-tenth test, multiply by ten,
and put the product of those two fictions in a capacity plan (<<ch16>>).

=== Seeding and teardown: tests that mutate the world

Load tests write. They create orders, fill carts, register users, append to
event streams. Run the same test against the same environment every night
without resetting it and the environment *drifts*: the orders table grows by
two million rows a week, indexes deepen, and in month three someone files a
bug that "the API is getting slower" when the only thing getting slower is
the sediment your own tests deposited. Worse is the opposite error —
comparing Tuesday's run against Wednesday's as if they hit the same system
(the comparison discipline of <<ch12>> assumes they did).

The rule: *every run starts from a known state and either restores it or
throws the environment away.* The mechanisms, roughly in order of robustness:

* *Snapshot and restore.* Restore the datastore from a fixture snapshot
  before each run. Slowest to set up, most trustworthy, and it doubles as a
  rehearsal of your backup restore.
* *Ephemeral environments.* Create, seed, test, destroy — the natural fit
  for the CI-gated tests of <<ch14>>.
* *Idempotent seed plus scripted cleanup.* A seed script that converges the
  environment to the fixture state, and teardown that removes what the run
  created. Namespace everything the test writes (a run-ID prefix on emails,
  order references, tenant names) so cleanup is a targeted delete, not a
  guess.

The test tool itself gets you part of the way. loadr's exported `setup()`
function runs once before any VU and can create prerequisites; `teardown()`
runs once when the run ends — the right place for revoking test credentials
or deleting the run's namespace. And for seeding jobs themselves — "process each
of these 10,000 fixture rows exactly once" — a shared feeder with
`on_eof: stop` turns the load tool into a competent parallel seeder: VUs pull
rows until the file is exhausted, then the run winds down.

[source,yaml]
----
data:
  fixtures:
    type: csv
    path: data/seed-accounts.csv
    mode: shared
    on_eof: stop          # each row exactly once; run ends when data does
----

=== Privacy: realistic data without real people

The tempting shortcut for all this realism is to copy production. Real
volume, real cardinality, real skew — one `pg_dump` and every problem in this
chapter dissolves. It also puts real names, emails, addresses, and payment
details into an environment with weaker access controls, longer-lived
credentials, and — by this chapter's own advice — regular deliberate
overload. Test environments are breached disproportionately often precisely
because nobody hardens them, and under GDPR and its siblings, "it was only
staging" is not a sentence a regulator accepts. Load-test results also leak
by design: they get pasted into tickets, dashboards, and slide decks. Data
that must not appear in a screenshot must not be in the test.

So the copy is out, and you're left balancing two options:

*Anonymized production data* keeps the shape (volume, skew, referential
integrity) and hides the values — if the anonymization is done well, which is
the catch. Naive masking destroys the statistical properties you copied the
data to get: map every email to `user@example.test` and you've collapsed a
40-million-cardinality column to one, resurrecting the single-user lie inside
your "realistic" dataset (and breaking every unique constraint on the way).
And pseudonymization that preserves too much — consistent fake names over
real purchase histories — is often re-identifiable, which is why regulators
treat it as still-personal data. Good anonymization preserves *distributions*
(cardinality, length, frequency skew, referential links) while destroying
*values*, and that is a real engineering task, not a sed script.

*Fully synthetic data* is generated from scratch to match measured
statistics: N users, M orders per user following the observed distribution,
key popularity following the observed skew. It's more up-front work and it
will miss pathologies you didn't think to model — the customer with 40,000
addresses, the name that's 300 characters of combining diacritics — which is
an argument for periodically mining production for *statistics about* its
outliers (never the outliers themselves) and folding them into the
generator.

For the secrets a test legitimately needs — API keys, service credentials —
keep them out of the scenario file and out of the output. loadr's `secrets:`
block sources values from the environment or a file and redacts them from
logs, summaries, and error messages:

[source,yaml]
----
secrets:
  api_key: { env: API_KEY }        # never appears in console output or reports

scenarios:
  s:
    executor: constant-vus
    vus: 10
    duration: 5m
    flow:
      - request:
          url: /v1/accounts
          headers:
            X-Api-Key: ${secrets.api_key}
----

A load-test report gets forwarded more widely than almost any other artifact
your team produces. Assume every run's output will be screenshotted.

=== Adversarial inputs: production traffic includes garbage

One last data-realism gap, and it's the one polite test suites always have.
Look at an hour of real production traffic and a substantial slice of it is
junk: bots, scrapers, stale mobile clients sending last year's payload
schema, half-closed connections, requests with malformed JSON, security
scanners probing for `/wp-admin` on your Go service. Your load test, built
from the API documentation, sends none of this. It tests the system your
designers imagined, not the one your firewall logs describe.

That would be fine if garbage were cheap to reject. Often it isn't. Error
paths are the least-optimized code in any system: a malformed request that
triggers an exception, a stack-trace capture, and a five-line ERROR log entry
can cost more CPU than the happy path it failed to be — and logging
frameworks serialize on appenders in ways that only show up when the error
*rate* is high. I've seen a service comfortable at 3,000 good requests per
second brought to its knees by 300 bad ones per second, entirely inside its
logging code.

So put a realistic fraction of hostile and edge-case traffic into the mix.
An inline feeder is a natural home for a curated nasties list:

[source,yaml]
----
data:
  nasties:
    type: inline
    rows:
      - { q: "" }                                    # empty
      - { q: "a" }                                   # degenerate
      - { q: "Ω≈ç√∫˜µ≤≥÷ 🏳️‍🌈 ﷽" }                    # unicode stress
      - { q: "' OR '1'='1" }                         # injection-shaped
      - { q: "AAAAAAAAAA...8KB-of-this..." }         # oversized
    pick: random

scenarios:
  hostile_slice:                # run alongside the main scenario, ~5% of traffic
    executor: constant-vus
    vus: 2
    duration: 10m
    flow:
      - request:
          url: /search
          params: { q: "${data.nasties.q}" }
          checks:
            - { type: status, one_of: [200, 400, 422] }   # reject, don't die
----

The check is deliberate: for adversarial input the SLO isn't "succeeds," it's
"fails fast, fails cheap, and doesn't degrade the well-behaved traffic
sharing the process." Run the hostile scenario *alongside* the main workload
(<<ch06>> covers composing mixed scenarios) and watch the good traffic's
percentiles, not just the bad traffic's status codes.

Edge-case inputs have one more job: input *size and shape* is a load axis all
its own, because a request that's ten times larger can be a hundred times
more expensive if a code path is accidentally quadratic. That observation —
feeding systematically growing payloads and watching whether cost grows
linearly or explosively — turns performance testing into a hunt for
algorithmic denial-of-service, and it gets its own treatment in <<ch13>>.

<<ch03>> and <<ch04>> gave you a realistic load shape and a checkable
definition of success. This chapter supplied the two preconditions that make
those meaningful: requests carrying data with production's variety and skew,
aimed at an environment with production's volume and configuration. <<ch06>>
assembles all of it into full, multi-scenario workloads — the last design
step before Part III finally lets you press the button.

.Key takeaways
****
* A load test with unrealistic data measures your caches, not your system.
  Parameterize with feeders, and match production's *access pattern* — key
  cardinality and skew — not just its request rate.
* Correlate dynamic values (tokens, session IDs, created resources) and make
  failed extraction fail the request. Rejected traffic is often *faster* than
  real traffic; a latency number without an error rate is meaningless.
* Data volume changes algorithms, not just durations: query plans, index
  depth, and cache hit ratios all shift with scale. Seed test databases to
  production's order of magnitude and verify with `EXPLAIN`, not hope.
* One-tenth environments don't produce one-tenth behavior. Bottlenecks live
  at absolute thresholds; build the stateful spine full-size, use ephemeral
  full-scale environments, or test in production with guardrails —
  extrapolation is not an option.
* Every run starts from a known state: snapshot/restore, ephemeral
  environments, or idempotent seed plus namespaced teardown. An environment
  that drifts between runs invalidates every comparison you make on it.
* Never load-test with real PII. Anonymize without collapsing cardinality, or
  generate synthetic data that preserves measured distributions — and keep
  secrets redacted, because test output travels.
* Production traffic includes garbage, and error paths are expensive. Put a
  realistic slice of malformed, oversized, and adversarial input in the mix,
  and require it to fail *cheaply*.
****

[[ch13]]
== Finding and Fixing Bottlenecks

The run finishes and the number is ugly: p99 checkout latency, 2.7 seconds,
against a 300ms SLO. The dashboard is a wall of red. Somebody asks the question
that separates a performance engineer from a person who runs load tests: _why?_

Not "is it slow" — you already know it's slow. Not "can we throw more instances
at it" — maybe, but you don't yet know whether that helps or just moves the
queue. The job now is a hunt: from a symptom (a fat tail on one endpoint) to a
cause specific enough to fix (this pool has eight connections and you asked it
for eighty). Done well, it takes an afternoon and ends with a one-line diff and
a re-test that proves it. Done badly, it's a week of changing random things and
re-running until the graph looks better for reasons nobody understands.

This chapter is the method for doing it well. We'll work from the outside in —
system resources first, then the application, then the algorithm — because that
order finds the cause fastest and stops you profiling a service that's actually
starved of file descriptors. The load test from <<ch08>> generated the pressure;
the observability from <<ch11>> gave you the internal view; the statistics from
<<ch12>> told you the tail is real and not noise. Now we use all three to name
the bottleneck and kill it.

=== Start with a method, not a hunch

The failure mode of bottleneck hunting is _pattern-matching on your last
incident_. Last time it was the database, so you stare at the database. This time
it's the connection pool in front of the database, you never look, and you lose a
day. The cure is a checklist you run _regardless_ of your hunch.

The best one is Brendan Gregg's *USE method*. For every resource in the system,
check three things:

* *Utilization* — what fraction of the time is it busy? (CPU at 85%, disk at
  40%.)
* *Saturation* — how much extra work is queued because the resource can't keep
  up? (Run-queue length, pool wait time, GC pause time.)
* *Errors* — is it throwing errors? (Failed allocations, dropped packets,
  connection timeouts.)

The resources are the usual suspects: CPU, memory, disk I/O, network, and the
software equivalents — connection pools, thread pools, locks, file descriptors.
You walk the list. Most entries come back clean in seconds. The one or two that
don't are where you dig.

The insight that makes USE worth memorizing is that *saturation, not utilization,
is what users feel*. A CPU pinned at 100% is not automatically a problem — it
might be doing exactly the work you want at full efficiency. What hurts is the
_queue_ behind it: the requests waiting for a core, the threads blocked on a lock,
the connections parked waiting for a free slot in the pool. Utilization tells you
a resource is busy. Saturation tells you work is _piling up_, and piled-up work is
latency with a countdown timer on it — the on-ramp to "down" we met in <<ch01>>.

[TIP]
====
Run the USE checklist top to bottom even when you're sure you know the answer.
The whole point is to catch the case where your intuition is confidently wrong.
It costs five minutes and it's saved me days.
====

=== Was it the target, or your load generator?

Before you trust a single server-side number, rule out the dumbest and most
common cause of a bad result: *your load generator ran out of gas*. A saturated
generator inflates every latency you measure — the request was ready to send at
T, but the box was pinned at 100% CPU and didn't actually put it on the wire
until T+400ms. You record 400ms of "server latency" that happened entirely on
your side. This is coordinated omission's uglier cousin, and <<ch10>> is the
whole chapter on generating load at scale without hitting it.

The defense is to measure the generator with the same rigor as the target.
loadr's `observe` collector samples the local host's CPU, memory, disk, and
network every second and folds the series into the run timeline as ordinary
metrics — so they're valid threshold targets, and a starved generator _fails the
run_ instead of quietly poisoning your numbers:

[source,yaml]
----
observe:
  - type: system
    metrics: [ cpu, memory, disk, network ]
    interval: 1s

thresholds:
  # If the generator itself saturates, the numbers are worthless — so fail.
  system_cpu:    [ "max < 0.90" ]
  system_memory: [ "max < 0.85" ]
  http_req_duration: [ "p(95)<800" ]
----

If `system_cpu` crested 0.90 while your latency climbed, stop. You haven't found
a bottleneck in the target; you've found one in your test rig. Add generators
(<<ch10>>), re-run, and only _then_ believe the latency curve. Every serious tool
gives you this view one way or another — k6 exposes generator resource use
through its own metrics and the `k6 run` output; the principle is tool-neutral,
the discipline is not optional.

=== Reading saturation on the four resources

Once you trust the numbers, correlate the latency timeline against the target's
resource timeline — the load test on one axis, the server's telemetry on the
other, sharing a clock (that correlation is the payoff of <<ch11>>). Each
resource saturates with a recognizable signature.

*CPU.* Utilization climbs toward 100% and the *run queue* grows — more runnable
threads than cores to run them. On Linux the tell is load average rising past
your core count and `%usr`/`%sys` dominating in `top`. Latency rises smoothly
with load, then bends sharply upward at the knee. If `%sys` (kernel time)
dominates rather than `%usr`, suspect syscall overhead — too many small
reads/writes, connection churn — not your business logic.

*Memory.* Raw utilization matters less than the _consequences_ of pressure. Watch
for the page cache getting evicted (disk reads climb because data that used to be
in RAM isn't anymore), swap activity (catastrophic for latency — a swapped-out
page is a millisecond where you expected a nanosecond), and, in managed runtimes,
GC frequency and pause time. A memory bottleneck often disguises itself as a disk
or CPU bottleneck two hops downstream.

*Disk I/O.* Utilization near 100% with a growing I/O queue depth. The signature
is latency that's fine until a threshold and then _steps_ — because you fell out
of the page cache and every read now costs a seek. `iostat -x 1` and watch
`%util` and `aqu-sz` (average queue size). A p99 that's 50× the p50 with a p50
that's healthy is very often "most requests hit cache, the unlucky ones hit
disk."

*Network.* Bandwidth saturation (bytes/sec near the NIC's ceiling) or, more
often, _connection_ saturation — ephemeral port exhaustion, `TIME_WAIT`
accumulation, conntrack tables filling. The signature is connection-establishment
latency rising while request-processing latency stays flat, and errors that are
`ECONNREFUSED`/`connection reset` rather than slow 200s.

[NOTE]
====
The number that ends most arguments is the *client-side vs server-side latency
gap*. If your load test says p99 is 900ms and the server's own request histogram
says p99 is 120ms, the missing 780ms is in the network, a proxy, TLS
renegotiation, or a queue you haven't instrumented — _not_ in the code you were
about to profile. Always establish where the time is being spent before you
decide what to fix.
====

=== Profiling: where the time actually goes

USE tells you _which resource_ is saturated. Profiling tells you _which code_ is
saturating it. When CPU is the bottleneck, you want a *flame graph* — the single
most useful visualization in performance work.

A flame graph is a picture of stack samples. The x-axis is _not_ time; it's the
proportion of samples in which a given function was on the CPU. Width equals cost.
You read it by scanning across the top for _wide plateaus_: a function that's wide
at the top of the stack was itself on-CPU for a large share of samples, which
means that's where the cycles are going. A function that's wide but only because
of a wide child is just a caller — follow the width down to the leaf that's
actually burning time.

[source,console]
----
# Linux, whole-host, sampled at 99Hz for 30s while the load test runs
$ perf record -F 99 -a -g -- sleep 30
$ perf script | stackcollapse-perf.pl | flamegraph.pl > cpu.svg
----

Language runtimes have their own, and you should reach for them first when the
target is a single service: Go's `net/http/pprof` (`go tool pprof`), `async-profiler`
for the JVM, `py-spy` for Python, `perf` + Node's `--prof` for V8. The move that
catches people out: *profile under the load test, not at rest*. A profile of an
idle service tells you what's expensive when nothing is happening, which is
nothing. The interesting contention — the lock everyone's waiting on, the
allocation storm, the cache that only misses under concurrency — only shows up
when the system is _actually_ under the pressure your load test is generating. Run
the profiler for a window in the middle of the steady-state hold, not during the
ramp.

Two flame-graph variants earn their keep. An *off-CPU flame graph* samples where
threads are _blocked_ (waiting on I/O, locks, sleeps) rather than running — that's
where you find lock contention and pool waits, which a CPU flame graph is blind to
by construction. And a *differential flame graph* colors a flame graph by the
_delta_ between two profiles (before and after a change, or good version vs
regressed version), so the thing that got more expensive lights up red. When you
have a regression and a suspect commit, the differential is often a five-second
diagnosis.

=== The usual suspects

Most production bottlenecks are not exotic. After enough hunts you learn a suspect
lineup, and each suspect has a signature you can match against the USE readings
and the flame graph.

*Connection-pool exhaustion.* The single most common one, and the most
misdiagnosed. Your app holds a pool of N connections to the database (or a
downstream service). Under load, all N are checked out, and the N+1st request
_waits_ — not because the database is slow, but because it can't get a connection
to talk to it. The signature is unmistakable once you know it: the database's own
metrics look _healthy_ (low CPU, fast queries), but application latency is
terrible, and the latency is spent in "acquire connection," not "execute query."
People stare at a bored database for hours. Instrument pool wait time. If your p99
request spends 600ms waiting for a pool slot and 4ms running the query, the fix is
pool size, connection lifecycle, or fixing whatever holds connections too long —
not the database.

[WARNING]
====
The instinct on pool exhaustion is to make the pool bigger, and sometimes that's
right. But a pool exists to _protect_ the downstream from more concurrency than it
can handle. Enlarge it blindly and you can convert a fast-failing client-side wait
into a slow-dying database with 500 concurrent queries thrashing its own locks and
buffer pool. Size the pool to what the downstream can actually serve, then fix the
_demand_ — usually the next suspect.
====

*The N+1 query.* The application makes one query to fetch a list, then one more
query _per row_ to fetch each item's details: 1 + N round trips where 2 would do.
Each query is individually fast, which is why it hides — the flame graph shows a
thousand tiny identical database calls, no single one expensive. The signature is
latency that scales with _result-set size_ rather than request rate, and a
database log full of the same query shape repeated with different IDs. This is
algorithmic complexity wearing a database costume — and complexity is the suspect
that hides best of all.

*Lock contention.* A shared lock (a mutex, a hot row, a `synchronized` block)
serializes work that looks parallel. Throughput flatlines or _decreases_ as you
add concurrency — the tell that separates it from simple CPU saturation, where
throughput at least plateaus. Utilization can look _low_ (everyone's blocked, not
running), which fools the USE utilization check; you catch it on saturation (long
off-CPU waits) and on the off-CPU flame graph, where the contended lock is a wide
plateau of blocked time.

*GC pauses.* In managed runtimes, garbage collection stops the world for
milliseconds to occasionally _seconds_. The signature is a p50 that's excellent and
a p99/p99.9 that's periodically awful, with the slow requests clustering in _time_
(everything that arrives during a pause is slow at once) rather than by input.
Correlate your latency spikes against the GC log; if the fat tail lines up with
collection pauses, you're tuning heap and allocation rate, not query plans.

*Cold caches.* Everything is fast until a deploy, a cache flush, or a failover,
and then latency spikes for a few minutes while caches refill from cold. The
signature is a _transient_ tail after any restart that recovers on its own — which
is exactly why a soak test (<<ch01>>) and a proper warm-up matter, and why a load
test that starts measuring before the caches are warm reports a pessimism that
production only sees at deploy time. Decide which one you're testing and warm (or
deliberately don't) accordingly.

=== Complexity as denial-of-service

Every suspect so far shows up under _realistic_ load. The nastiest bottleneck of
all does not. It waits for a specific, _adversarial_ input and then bends your
response-time curve into a wall.

The mechanism is algorithmic complexity. A parser that's O(n) on input size is
fine; one that's O(n²) on nesting depth is a loaded gun. At typical input it's
indistinguishable from the linear one — a few hundred microseconds, lost in the
noise. Feed it an input crafted to hit the super-linear path — a JSON document
nested ten thousand levels deep, a regex input engineered for catastrophic
backtracking, an XML entity-expansion bomb — and a one-kilobyte request ties up a
core for ten seconds. A handful of such requests and the service is down. This is
a denial-of-service vulnerability that lives in your _performance_ profile, and no
amount of replaying realistic traffic will ever find it, because realistic traffic
never sends the adversarial input. You have to send it on purpose.

The technique is to make the input's severity a single number you can _scale_ —
a depth, a byte count, a level count — and then watch what the response time does
as that number grows. Linear growth is safe. Anything that bends upward is a
finding. loadr's `payload` generator produces exactly these adversarial inputs,
each parameterized by one magnitude:

[source,console]
----
# A JSON object nested 10,000 levels deep, straight at a parse endpoint.
$ loadr payload nested-json:10000 | curl -X POST http://localhost:3001/api/parse \
    -H 'Content-Type: application/json' --data-binary @-

# See the whole catalog — 18 kinds: nesting, amplification, ReDoS,
# hash-collision, unicode, bignum — each with its magnitude parameter and cap.
$ loadr payload --list
----

[WARNING]
====
These payloads are built to exhaust CPU and memory on the receiving end — that is
the point. Aimed at a shared or production target they are indistinguishable from
a real DoS attack. Only ever fire them at systems you own or are explicitly
authorized to test, start with small magnitudes, and use a disposable
environment. This is the one technique in the book that is genuinely dangerous
pointed the wrong way.
====

Firing a single payload proves the path is _reachable_. To prove it's a
_complexity_ problem — and to gate against it — you scale the magnitude across a
range and fit the curve. Embed the payload in the request body with the
`${payload:<kind>:$ENVVAR}` template, where the magnitude comes from a swept axis,
so one plan runs at every size with no edits:

[source,yaml]
----
scenarios:
  render:
    executor: per-vu-iterations
    vus: 1
    iterations: 4
    flow:
      - request:
          name: render
          method: POST
          url: /api/v1/markup
          headers: { Content-Type: application/json }
          # depth is read at request time from the swept axis (LOADR_SWEEP_DEPTH)
          body: '{"Text": "${payload:nested-markdown-blockquote:$LOADR_SWEEP_DEPTH}", "Mode": "markdown"}'
          checks: [ { type: status, equals: 200 } ]
----

Then sweep the depth axis and let `loadr sweep --complexity` fit the exponent `k`
in `response-time ≈ size^k` — a log-log regression of p95 latency against the size
axis — and fail CI when `k` crosses a bound (recall from <<ch07>> and the sweep
tooling that a size sweep and a load sweep are the same machinery pointed at
different axes):

[source,console]
----
$ loadr sweep examples/49-payload-complexity.yaml \
    --var depth=4000,8000,16000,32000,64000 \
    --complexity depth --max-exponent 1.2
----

Against a parser that walks the nesting quadratically, every 2× in depth roughly
4×s the latency, and the fit is damning:

[source,text]
----
  combo        p50      p95       p99       error rate  rps
  -----------  -------  --------  --------  ----------  -------
  depth=4000   41.20ms  52.90ms   61.00ms   0.00%       18.9/s
  depth=8000   150.4ms  178.2ms   190.1ms   0.00%       5.4/s
  depth=16000  590.7ms  651.0ms   690.4ms   0.00%       1.5/s
  depth=32000  2.35s    2.61s     2.74s     0.00%       0.4/s
  depth=64000  9.41s    10.30s    10.80s    0.00%       0.1/s

complexity (response time vs depth)
  O(n^1.99)  ≈ quadratic ⚠ DoS risk
✗ fitted exponent O(n^1.99) exceeds the --max-exponent 1.20 bound
----

`O(n^1.99)` — quadratic — past the 1.2 bound, exit 99, merge blocked. A healthy
endpoint fits near `O(n^1.03)`, passes, and says so. The reasoning is
tool-agnostic: run the same input at several sizes, plot response time against
size on log-log axes, and read the slope — a slope near 1 is linear and safe, a
slope near 2 is quadratic and a latent outage. You can do it by hand with a k6
scenario per size and a spreadsheet; loadr just fits the line and gates on the
slope for you. Point the same pattern at a `redos` body to guard a validator, a
`hash-collision` body to guard a map, a `nested-json` body to guard an API parser
— whichever code path you can't afford to have go super-linear.

[TIP]
====
Complexity probes are cheap — one VU walking a handful of magnitudes, no sustained
load — so unlike the overnight capacity sweep they belong on _every PR_. That turns
"our markdown renderer is O(n²) on nesting depth" from a 3am incident into a red
check on the pull request that introduced it. That's the whole game: move the
discovery left.
====

=== Prove the fix

Finding the bottleneck is half the job. The other half is proving you fixed it —
and proving you didn't just move it somewhere else or make three other things
worse. A performance fix without a re-test is a hypothesis, not a result.

The discipline is: capture a baseline _before_ the change, apply the smallest fix
that addresses the named cause, and re-run the _identical_ test. Then compare the
distributions — not the averages, the distributions, and especially the tail you
were chasing. loadr's `compare` diffs two summary exports and tells you, per
metric and per percentile, what moved and in which direction:

[source,console]
----
$ loadr run perf/checkout.yaml --summary-export before.json   # bottleneck present
# ... apply the fix: pool size 8 → 40, sized to what the DB can serve ...
$ loadr run perf/checkout.yaml --summary-export after.json     # fix applied
$ loadr compare before.json after.json
----

[source,text]
----
  metric             field   baseline  current   delta      delta %
  -----------------  ------  --------  --------  ---------  --------  -
  http_req_duration  p50     42.30ms   40.10ms   -2.20ms    ▼ 5.2%    ✓
  http_req_duration  p95     880.40ms  96.20ms   -784.20ms  ▼ 89.1%   ✓
  http_req_duration  p99     2.71s     138.00ms  -2.57s     ▼ 94.9%   ✓
  http_req_failed    rate    2.31%     0.02%     -2.29%     ▼ 99.1%   ✓
----

The p50 barely moved — it never was the problem — while the p99 collapsed from 2.7
seconds to 138ms. _That's_ what a real fix looks like: the tail comes home and the
median doesn't lie about it. Wire the same comparison into CI (<<ch14>>) with a
regression tolerance and the pipeline now defends the fix — the next change that
re-inflates that tail fails the build instead of reaching production. This is also
the honest guard against the most seductive mistake in the whole discipline:
declaring victory because the average improved while the tail — the part your
users actually feel, per <<ch01>> and <<ch12>> — got quietly worse.

That's the loop, and it's the same loop whatever's in the middle: generate the
load, read the resources with USE, profile the saturated one, match the signature
to a suspect, apply the smallest fix that names the cause, and re-test to prove
it. Do it enough times and the hunt that used to take a week takes an afternoon —
because you stopped guessing and started reading the system.

.Key takeaways
****
* Hunt with a *method*, not a hunch. Walk the USE checklist — Utilization,
  Saturation, Errors — across every resource, even the ones your intuition has
  already ruled out. *Saturation is what users feel*, not utilization.
* *Rule out your own load generator first.* A saturated generator inflates every
  latency you measure; make its CPU and memory threshold-gated so a starved rig
  fails the run instead of poisoning it.
* Establish the *client-side vs server-side gap* before you profile. If the missing
  time is in the network or a proxy, the code you were about to profile is
  innocent.
* *Profile under load, not at rest*, and reach for flame graphs — off-CPU for
  contention and waits, differential for regressions.
* Learn the suspect lineup by *signature*: pool exhaustion (bored database, slow
  app), N+1 (latency scales with result size), lock contention (throughput drops as
  concurrency rises), GC (periodic tail clustered in time), cold cache (transient
  tail after restart).
* *Algorithmic complexity is a DoS you won't find with realistic traffic.* Scale one
  adversarial magnitude, fit response-time-vs-size, and gate on the exponent —
  linear passes, super-linear is a latent outage. It's cheap enough for every PR.
* A fix isn't done until a *re-test proves it*. Compare distributions and watch the
  tail you were chasing — a fix that only moves the average is a fix that lied.
****

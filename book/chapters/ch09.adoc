== Beyond HTTP: Advanced Protocols and Patterns

A trading platform I worked with had one of the best REST load-test suites I've
ever audited. Every endpoint covered, arrival-rate models, SLO gates in CI, the
works. Their worst outage of the year had nothing to do with any of it. A
routine deploy restarted the WebSocket gateway, 40,000 connected clients
reconnected within nine seconds, each reconnect triggered a token validation
against the identity provider, the identity provider fell over, and now the
retry logic in 40,000 browsers was in charge. Total time from "healthy deploy"
to "nothing works": under a minute. Total coverage of that failure mode in the
load-test suite: zero.

Grep your production ingress and you'll find the same shape almost everywhere.
Plain request/response HTTP is the *minority* of what a modern system does.
There are persistent WebSocket and SSE connections, gRPC between services,
GraphQL in front of the mobile app, Kafka and AMQP carrying the events, a
database and a cache underneath all of it, and an identity provider that every
single one of those flows depends on. If your load tests only speak REST,
you've tested the lobby of the building and declared the structure sound.

The good news: everything you've learned so far transfers. Open versus closed
workload models (<<ch03>>), explicit machine-checkable objectives (<<ch04>>),
honest percentiles (<<ch12>>) — none of that changes. What changes is *what a
"request" even is*, *which metric carries the truth*, and *which failure modes
are worth provoking*. This chapter walks the protocols you'll actually meet and
shows you, for each one, the question worth asking and the trap worth avoiding.

=== HTTP/2 and HTTP/3: same verbs, different physics

Start with the deceptively familiar case. HTTP/2 keeps the semantics of
HTTP/1.1 — same methods, same status codes — and completely changes the
transport underneath: one TCP connection per origin, many concurrent streams
multiplexed over it. That one change quietly invalidates several assumptions a
load test might be making.

Under HTTP/1.1, a virtual user that fires ten concurrent requests opens (or
borrows from a pool) up to ten connections. Under HTTP/2 it opens *one*. Your
server's connection count, its TLS-handshake rate, its file-descriptor
pressure, its load-balancer distribution — all of these look radically
different under h2 than under h1 at the same request rate. I've watched a team
"reproduce" a production connection-exhaustion incident with a load tool that
silently spoke HTTP/1.1 against a server whose real clients were all h2. The
test showed the failure beautifully. Production could never have failed that
way. They spent a week fixing a bug they didn't have.

So the first rule: *know which protocol version your tool is negotiating, and
pin it deliberately.* In loadr that's one key:

[source,yaml]
----
defaults:
  http:
    version: http2      # auto | http1 | http2 | http2-prior-knowledge
    tls:
      min_version: "1.2"
      max_version: "1.3"
----

`auto` (the default) uses ALPN and takes h2 whenever the server offers it —
which is what real browsers do, so it's usually the honest choice. But run the
same scenario pinned to `http1` and pinned to `http2` and compare: the delta
tells you how much of your capacity story is really a connection story. The
`http2-prior-knowledge` mode skips negotiation entirely, which is how gRPC and
many internal service meshes actually run.

The second rule: multiplexing moves head-of-line blocking, it doesn't remove
it. HTTP/2 fixes HTTP-level HOL blocking but inherits TCP's: one lost packet
stalls *every* stream on the connection. On clean data-center networks you'll
never see it; on lossy last-mile networks it's why h2 can be *slower* than six
parallel h1 connections. HTTP/3 exists precisely to fix this — QUIC gives each
stream independent loss recovery — but at the time of writing, HTTP/3 support
across load-testing tools (loadr and k6 included) is essentially absent. If h3
performance matters to you, measure it from real browsers (see
"Browser-level versus protocol-level" later in this chapter) rather than
trusting a generator that silently downgraded.

=== gRPC: deadlines, status zero, and streams

gRPC is HTTP/2 with a schema and an attitude. Three things make load-testing
it different from testing REST.

First, *the status model*. A gRPC call can return HTTP 200 and still have
failed — the real verdict is the gRPC status code in the trailers, where `0`
means OK. Any check that looks at HTTP status alone will happily green-light a
service returning `RESOURCE_EXHAUSTED` on every call. Assert on the gRPC
status, always.

Second, *deadlines*. Well-behaved gRPC clients send a deadline with every call
and servers shed work that can't finish in time. A load test that omits
deadlines measures a client no one actually runs — and misses the most
interesting production behavior, which is what your service does when
deadlines start expiring mid-call under load.

Third, *the schema gets in the way of quick tests*. You need the `.proto`
files, or a server with reflection enabled. Here's both variants in loadr —
unary calls at a fixed arrival rate, and a server-streaming call via
reflection:

[source,yaml]
----
scenarios:
  unary:
    executor: constant-arrival-rate
    rate: 200
    duration: 3m
    pre_allocated_vus: 30
    max_vus: 100
    flow:
      - request:
          name: say hello
          url: grpc://greeter.example.com:50051
          grpc:
            proto_files: [ protos/helloworld.proto ]
            service: helloworld.Greeter
            method: SayHello
            message: { name: "vu-${vu}" }
            metadata: { x-api-key: "${env.GRPC_API_KEY}" }
          assert:
            - { type: status, equals: 0 }        # gRPC OK, not HTTP 200
            - { type: jsonpath, expression: "$.message", exists: true }

  streaming:
    executor: constant-vus
    vus: 5
    duration: 3m
    flow:
      - request:
          name: stream replies
          url: grpc://greeter.example.com:50051
          grpc:
            reflection: true                     # no proto files needed
            service: helloworld.Greeter
            method: LotsOfReplies
            message: { name: "stream-${vu}" }

thresholds:
  grpc_req_duration: [ "p(95)<150" ]
----

The same unary call in k6, which has first-class gRPC support:

[source,javascript]
----
import grpc from 'k6/net/grpc';
import { check } from 'k6';

const client = new grpc.Client();
client.load(['protos'], 'helloworld.proto');

export default () => {
  client.connect('greeter.example.com:50051');
  const res = client.invoke('helloworld.Greeter/SayHello', { name: 'k6' });
  check(res, { 'status is OK': (r) => r && r.status === grpc.StatusOK });
  client.close();
};
----

Notice the two scenarios use different executors, and that's deliberate. Unary
calls are request/response, so an open, arrival-rate model applies just as it
did for HTTP. A server stream is a long-lived thing — what you're modeling is
*concurrent streams held open*, which is a closed, VU-count question. Getting
this mapping right is most of the craft in this chapter.

=== WebSocket and SSE: when the connection is the workload

With WebSocket, the request/response frame collapses entirely. There is no
"requests per second" in any meaningful sense. What exists instead:

* *Concurrent connections held* — the number your gateway's memory and file
  descriptors actually care about.
* *Connection establishment rate* — the handshake-plus-auth storm, the thing
  that killed the trading platform.
* *Message latency* — how long a message takes to make the round trip once the
  connection exists.
* *Session lifetime behavior* — does a connection that lives for an hour leak
  anything?

These are four different tests, and the vocabulary of <<ch01>> maps onto them
cleanly: concurrent connections is a load test, establishment rate is a spike
test, session lifetime is a soak. Here's a loadr scenario exercising the
conversational middle — connect, speak the protocol, assert on what comes
back:

[source,yaml]
----
scenarios:
  chat:
    executor: constant-vus         # closed model: N connections held
    vus: 20
    duration: 2m
    flow:
      - request:
          name: chat session
          url: wss://chat.example.com/ws
          protocol: ws
          headers:
            Origin: https://chat.example.com
          ws:
            subprotocols: [ "chat.v2" ]
            send:
              - '{"type":"hello","room":"load-${vu}"}'
              - { text: '{"type":"message","body":"ping ${iteration}"}', delay: 500ms }
            receive_count: 2
            session_duration: 10s
          checks:
            - { type: body_contains, name: got ack, value: '"type":"ack"' }
      - think_time: { type: constant, duration: 1s }

thresholds:
  ws_session_duration: [ "p(95)<11000" ]
  ws_msgs_received: [ "count>100" ]
----

Note the thresholds: they're on *session duration* and *messages received*,
not on anything shaped like `http_req_duration`. When the protocol changes,
insist that your success criteria change with it. A WebSocket test whose only
gate is "connections succeeded" is a number generator.

Server-Sent Events is WebSocket's simpler cousin — one-directional, plain
HTTP, trivially proxyable — and it hides the single most underrated failure
mode in this chapter: *the reconnect storm*. SSE clients reconnect
automatically and send a `Last-Event-ID` header so the server can replay what
they missed. Now imagine a deploy that drops every stream at once. Thousands
of clients reconnect nearly simultaneously, and each one asks the server to
replay history. Reconnect load plus replay load, precisely when the service is
coldest. Test it on purpose:

[source,yaml]
----
scenarios:
  order_stream:
    executor: constant-vus
    vus: 25
    duration: 2m
    flow:
      - request:
          name: order updates
          url: sse://events.example.com/orders/stream
          headers:
            Authorization: Bearer ${token}
            Last-Event-ID: "${iteration}"    # simulate replay-from-offset
          sse:
            events: 10          # stop after 10 events…
            until: done         # …or an `event: done`…
            duration: 30s       # …or 30s — whichever comes first
          checks:
            - { type: body_contains, name: got update, value: '"status"' }
----

[WARNING]
====
Long-lived connection tests will find *your generator's* limits before the
server's if you're not watching: file descriptors, ephemeral ports, and
per-connection buffers on the load-injection machine are all finite. Holding
50,000 idle WebSockets is cheap in CPU and expensive in everything else.
<<ch10>> covers how to tell when the thing you're measuring is your own test
rig.
====

=== GraphQL: one endpoint, a thousand costs

To an HTTP-level load tool, GraphQL looks like the easiest target imaginable:
one endpoint, `POST /graphql`, done. That appearance is the trap, twice over.

Trap one: *GraphQL returns 200 for almost everything.* Resolver blew up?
`200 OK`, with an `errors` array in the body. A test that checks HTTP status
will report a flawless run against a service that failed every query. Your
checks must look inside the response — either for the data you expected or for
the absence of `errors`. (loadr's `graphql` protocol does this classification
for you and fails the request when no data comes back; with a generic HTTP
tool you must write the body check yourself, and most people don't.)

Trap two: *the endpoint has no cost; the query does.* `{ product(id: 1) {
name } }` and a five-level nested query that fans out into ten thousand
resolver calls hit the same URL. "We load-tested /graphql" is a meaningless
sentence until you say *which queries, in what mix*. The workload-modeling
discipline from <<ch03>> applies with a vengeance here: pull your top queries
by frequency *and* by cost from production telemetry, and test that corpus.
Pay special attention to list fields backed by per-item resolvers — the N+1
query pattern is GraphQL's signature performance bug, invisible at one virtual
user and catastrophic at a hundred.

[source,yaml]
----
scenarios:
  product_search:
    executor: ramping-vus
    start_vus: 0
    stages: [ { duration: 30s, target: 20 }, { duration: 3m, target: 20 } ]
    flow:
      - request:
          name: search products
          url: /graphql
          protocol: graphql
          graphql:
            query: |
              query Search($term: String!, $first: Int!) {
                products(search: $term, first: $first) {
                  edges { node { id name price } }
                  totalCount
                }
              }
            variables: { term: "widget", first: 20 }
            operation_name: Search
          extract:
            - { type: jsonpath, name: first_id, expression: "$.data.products.edges[0].node.id" }
          checks:
            - { type: jsonpath, name: has results, expression: "$.data.products.totalCount" }
      - request:
          name: product detail
          url: /graphql
          protocol: graphql
          graphql:
            query: "query One($id: ID!) { product(id: $id) { id name description } }"
            variables: { id: "${first_id}" }

thresholds:
  graphql_req_duration: [ "p(95)<400" ]
----

Two details worth stealing from that example even outside GraphQL: the second
request uses an ID *extracted from the first response*, so the flow behaves
like a client rather than a random-number generator; and the per-query `name`
means the results break down by operation, not by URL — essential when every
operation shares one URL.

[TIP]
====
If your API enforces persisted queries or query-complexity limits in
production, run your load tests through the same enforcement. Testing raw
query strings against a server that will only accept persisted hashes in
production is testing a configuration you don't ship.
====

=== Queues and events: latency becomes a supply chain

The moment traffic goes through Kafka or RabbitMQ, the question changes shape.
Nobody's SLO says "the produce call acknowledged in 8ms." The SLO that matters
is *end-to-end*: an order event was published, and the fulfillment service
acted on it within two seconds. Between those points sit a broker, partitions,
a consumer group, and — the number that actually predicts your incidents —
*consumer lag*.

A load test against an event-driven system therefore has two layers. The layer
your load tool measures directly is the broker interaction: produce
acknowledgment latency, fetch latency, publish and get against queues. In
loadr, brokers are runtime-installable protocol plugins (`loadr plugin install
kafka`), and the operations look like any other request:

[source,yaml]
----
plugins:
  - name: kafka

scenarios:
  producers:
    executor: constant-vus
    vus: 5
    duration: 15s
    flow:
      - request:
          name: produce event
          url: kafka://broker.example.com:9092/loadr-demo
          plugin:
            operation: produce
            key: "vu-${vu}"
            value: "event from vu ${vu} iter ${iteration}"
          assert:
            - { type: status, equals: 1 }     # 1 = ok, 0 = client error

  consumers:
    executor: constant-arrival-rate
    rate: 40
    duration: 15s
    pre_allocated_vus: 5
    flow:
      - request:
          name: fetch from head
          url: kafka://broker.example.com:9092/loadr-demo
          plugin:
            operation: fetch
            offset: 0
            max_wait_ms: 500
          checks:
            - { type: duration, name: fetch is fast, max: 1s }

thresholds:
  kafka_req_duration: [ "p(95)<1s" ]
----

(RabbitMQ is the same pattern with `amqp://` URLs and `publish`/`get`
operations against a queue.)

The second layer — end-to-end propagation — your load tool *cannot* measure
alone, because the consumer that matters is your real service, not the test.
The technique: embed a timestamp in every produced payload (the `value` line
above is where it goes), have the consuming side record the delta as a metric,
and watch consumer lag on the broker while the test runs. This is the clearest
case in the book of the partnership from <<ch01>>: the load test creates the
conditions; observability (<<ch11>>) carries the verdict.

And the test worth running above all others here is the *backlog drain*: pause
consumers, produce for ten minutes, resume, and time the recovery. Queues
don't fail by being slow. They fail by growing a backlog faster than consumers
can drain it — and a system that can only just keep up at steady state has no
headroom to recover from any interruption at all.

=== Hitting the data layer directly

Why load-test a database by itself when every HTTP test already exercises it?
Isolation. When the API's p99 doubles under load, "it's the database" is a
hypothesis, not a finding. A direct test against the data layer gives you a
clean baseline for what the database can do *without* the application in the
way — which turns "it's the database" into a number you can compare.

[source,yaml]
----
plugins:
  - name: postgres

scenarios:
  postgres_reads:
    executor: constant-vus
    vus: 10
    duration: 30s
    flow:
      - request:
          name: pg list products
          url: postgres://loadr:loadr@db.example.com:5432/loadr
          sql:
            query: SELECT id, name, price FROM products WHERE price < $1 ORDER BY price
            params: ["50"]
          checks:
            - { type: status, equals: 1 }
            - { type: duration, name: query is fast, max: 250ms }

thresholds:
  postgres_req_duration: [ "p(95)<300ms" ]
----

Two rules keep tests like this honest. *Parameterize like production does* —
`$1` placeholders bound by the driver, not string-interpolated SQL, because
prepared-statement caching is part of what you're measuring. And *watch the
connection pool first*, because it is nearly always the first thing to
saturate: ten VUs against a pool of ten is a queueing system (Little's Law,
<<ch02>>, will tell you exactly how it degrades). JMeter deserves a mention
here — its JDBC sampler has been the workhorse of direct database load
testing for two decades, and for SQL-only work it remains a perfectly good
choice.

Caches get the same treatment with one extra twist: *key distribution is the
whole test.* Redis will do hundreds of thousands of uniformly distributed GETs
per second and then fall over on one hot key, because a single key lives on a
single shard and a single CPU. Test both shapes deliberately — per-VU keys for
churn, one shared key for the hot spot:

[source,yaml]
----
plugins:
  - name: redis

scenarios:
  cache_churn:
    executor: constant-vus
    vus: 20
    duration: 1m
    flow:
      - request:
          name: set session
          url: redis://cache.example.com:6379
          plugin:
            command: ["SET", "session:${vu}", "active"]   # per-VU key
      - request:
          name: get session
          url: redis://cache.example.com:6379
          plugin:
            command: ["GET", "session:${vu}"]
          checks:
            - { type: duration, name: cache is fast, max: 50ms }

  hot_counter:
    executor: constant-arrival-rate
    rate: 200
    duration: 1m
    pre_allocated_vus: 30
    flow:
      - request:
          name: increment counter
          url: redis://cache.example.com:6379
          plugin:
            command: ["INCR", "page:views"]               # one shared hot key
----

Search stores (Elasticsearch and friends) follow the same isolation logic with
one addition: index writes and searches contend with each other, so test the
*mix* — a bulk-indexing scenario running concurrently with a search scenario —
not each side alone.

Finally, sometimes there's no protocol at all — a custom TCP wire format, a
UDP metrics firehose, a legacy socket service. Most tools have an escape
hatch: Locust lets you wrap any Python client and report its timings as tasks,
and loadr speaks raw sockets directly (`url: tcp://echo.example.com:7000` with
`socket: { send_text: "PING\n", read_bytes: 64, read_timeout: 2s }`, or
`udp://` with hex payloads). The measurement discipline is unchanged; only the
framing is yours to define.

=== Browser-level versus protocol-level

Everything so far has measured the *server*. None of it measures what a user
*sees*, because what a user sees also includes DNS, TLS, thirty subresources,
a megabyte of JavaScript, and however long the main thread chews on it. That's
what browser-level testing is for: drive real headless Chrome, render the
page, and capture Web Vitals — First Contentful Paint, Largest Contentful
Paint — instead of just time-to-last-byte.

The catch is cost. A protocol-level VU is a few kilobytes of connection state;
a browser VU is a Chrome tab — tens to hundreds of megabytes and a real slice
of CPU. You will not run ten thousand of them, and you shouldn't try. The
pattern that works is the *hybrid*: generate the crowd at the protocol level,
and let a handful of browser VUs experience the site while the crowd is
present.

[source,yaml]
----
metrics:
  plugin_req_duration: { kind: trend, time: true }

scenarios:
  crowd:                      # thousands of cheap protocol-level users
    executor: constant-arrival-rate
    rate: 500
    duration: 5m
    pre_allocated_vus: 100
    flow:
      - request: { name: api browse, url: https://example.com/api/products }

  experience:                 # five real browsers riding on top
    executor: constant-vus
    vus: 5
    duration: 5m
    flow:
      - request:
          name: load homepage
          protocol: browser   # real headless Chrome via CDP
          url: https://example.com
          timeout: 30s
          checks:
            - { type: body_contains, name: rendered, value: "</html>" }
      - think_time: { type: constant, duration: 2s }

thresholds:
  plugin_req_duration: [ "p(95)<5000" ]   # page render under load
----

k6 offers the same hybrid through its `k6/browser` module — a Playwright-style
API in the same script as your HTTP scenarios — and the pattern matters far
more than the tool: *the browser VUs are your measurement instrument; the
protocol VUs are your load.* Five browsers tell you what LCP looks like for a
real user during the rush. Five browsers do not *create* the rush.

=== Auth at scale: minting, refreshing, and the herd

Every protocol in this chapter shares one dependency, and it's the one load
tests most consistently get wrong. Authentication under load fails in three
distinct ways, and you should decide — explicitly, per test — which ones you're
testing and which you're bypassing.

*The accidental IdP test.* Ten thousand VUs that each begin by calling
`POST /oauth/token` will load-test your identity provider whether you meant to
or not — and if it's a third-party IdP, "load test" may be spelled "rate
limited," or worse, "please see the attached invoice." When auth isn't the
system under test, mint tokens cheaply: a shared long-lived test token, or one
grant performed centrally and distributed to every VU.

*The deliberate login storm.* Sometimes the IdP *is* the system under test —
Monday 9 a.m. is a login spike at every enterprise on earth, and password
hashing is deliberately, tunably expensive CPU work. That test is worth
running, on purpose, with the IdP team watching.

*The refresh storm.* The subtle one. A load test starts all its VUs within
seconds of each other; each VU mints a token; every token carries the same
TTL. One TTL later, every VU refreshes *simultaneously* — a periodic thundering
herd your test manufactured, hammering the token endpoint in lockstep every
hour on the hour. Production traffic, with logins smeared across the day,
never does this. Jitter your token lifetimes, or your steady-state test has a
metronome in it.

The mechanics of doing this well: keep token handling out of the scenario flow
and in a hook, so the YAML stays about the workload. Each loadr VU here holds
its own token and transparently re-mints it before expiry:

[source,javascript]
----
// scripts/auth.js — fired before every YAML `request:` step
const TTL_MS = 60_000, SKEW_MS = 5_000;

export function beforeRequest(req) {
  let auth = session.vars.auth;
  if (!auth || Date.now() >= auth.expiresAt - SKEW_MS) {
    auth = mint();                              // real: POST to your IdP
    session.vars.auth = auth;
    session.counterAdd('token_refreshes', 1);   // make refresh visible
  }
  req.headers = req.headers || {};
  req.headers['Authorization'] = `Bearer ${auth.token}`;
  return req;
}
----

[source,yaml]
----
js: { file: scripts/auth.js }

metrics:
  token_refreshes: { kind: counter }

scenarios:
  authed_traffic:
    executor: constant-vus
    vus: 10
    duration: 1m
    flow:
      - request: { name: profile, url: /me, checks: [ { type: status, equals: 200 } ] }
      - think_time: { type: uniform, min: 200ms, max: 800ms }
      - request: { name: orders, url: /orders, checks: [ { type: status, equals: 200 } ] }
----

That `token_refreshes` counter is the point of the example: auth overhead is
part of your workload, so measure it like one. If refreshes spike in lockstep,
you've found the metronome. The same per-VU-token pattern exists in k6 (mint
in the `setup()` or per-VU init context) and Artillery (its
`generating-vu-tokens` recipe); at very high VU counts, prefer the centralized
variant — one client-credentials grant, refreshed in one place, shared by the
fleet — so your auth server sees one token request every few minutes instead
of thousands competing with the traffic you're actually trying to measure.

One last cost that hides in plain sight: *validation*. Every authenticated
request costs the server a JWT signature check, and RSA verification is
meaningfully more expensive than the HMAC or EdDSA alternatives. At tens of
thousands of requests per second, token verification alone can be a visible
slice of your CPU flame graph. You'll only see it if your test sends real
tokens — one more reason not to stub auth out of existence.

=== Choosing your altitude

You now have the same system testable at four altitudes: the browser above,
HTTP/GraphQL/gRPC at the API, queues in the middle, databases and caches
below. Resist the instinct to always test at the top. The top-level test is
the most realistic and the least diagnostic — when it's slow, everything below
is a suspect. The skill is picking the altitude that matches your question:
user experience → browser; service SLO → API protocol; propagation and
backlog → broker; "is it the database?" → the database. And when a top-level
number goes bad, the lower-level tests are how you bisect — which is exactly
the systematic hunt <<ch13>> takes up.

.Key takeaways
****
* Real traffic is mostly not plain request/response HTTP. A load-test suite
  that only speaks REST has tested the lobby, not the building.
* Pin and verify the HTTP version your tool negotiates. h1 and h2 produce
  different connection, TLS, and load-balancer behavior at identical request
  rates — and a version mismatch can make you "reproduce" failures you can't
  actually have.
* Match the workload model to the protocol: arrival rate for request/response
  (HTTP, gRPC unary), VU counts for held-open things (WebSocket, SSE, gRPC
  streams). And change your metrics with the protocol — session duration and
  messages received, not `http_req_duration`.
* Status codes lie off-HTTP: gRPC fails inside trailers with HTTP 200, GraphQL
  fails inside an `errors` array with HTTP 200. Check the protocol's real
  verdict.
* For event-driven systems the SLO is end-to-end propagation and backlog
  drain, not produce latency. The load tool creates the conditions; broker
  and consumer observability carry the verdict.
* Test the data layer directly to turn "it's the database" from a hypothesis
  into a comparison — and remember key distribution (hot keys) and connection
  pools are usually the real limits.
* Browsers are instruments, not load: a handful of real-browser VUs measuring
  Web Vitals on top of a protocol-level crowd beats either approach alone.
* Decide per test whether auth is under test or bypassed. Beware the
  accidental IdP load test, and beware the refresh storm your own
  synchronized VUs manufacture.
****

[.fieldcard]
.Field Card — Testing non-HTTP protocols
****
Match the workload model to the protocol, and change your metric with it — REST habits lie off-HTTP.

Request/response (HTTP, gRPC unary):: open model, `constant-arrival-rate`.
Held-open (WebSocket, SSE, gRPC streams):: closed model, `constant-vus`; gate on `ws_session_duration` / `ws_msgs_received`, never `http_req_duration`.
gRPC / GraphQL verdict:: HTTP 200 is not success — assert gRPC `status: 0` (trailers); for GraphQL check for data / absence of an `errors` array.
Event-driven (Kafka, AMQP):: the SLO is end-to-end propagation + backlog drain, not produce-ack latency; watch consumer lag.
"Is it the database?":: test the data layer directly for a clean baseline; watch the connection pool and hot keys first.

*Remember:* GraphQL returns 200 for almost everything — check inside the body. Browsers are the instrument, not the load: 5 real Chrome VUs (tens–hundreds of MB each) ride on a protocol-level crowd. Jitter token TTLs or synchronized VUs manufacture an hourly refresh storm; and holding 50k idle sockets exhausts *your* file descriptors and ephemeral ports before the server's.

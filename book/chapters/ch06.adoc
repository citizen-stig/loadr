[[ch06]]
== Designing Realistic Scenarios

_From a single request to a believable population of users._

The test was forty lines of YAML and it passed beautifully. A UK retailer, two
weeks before their biggest sale of the year, ran `GET /products` at 3,000
requests per second for an hour: p95 of 80ms, error rate zero, champagne emoji
in the Slack channel. On the day, the site went down at roughly 400 checkouts
per minute — perhaps a fiftieth of the traffic the test had "cleared."

The post-mortem was not subtle. Checkout writes to the orders table, decrements
inventory inside a transaction, and calls a payment provider with a 2-second
timeout. Under real traffic, row locks on the hot SKUs queued up, connections
from the pool sat pinned waiting on the payment call, the pool exhausted, and
every request behind it — including the product listings that had tested so
well — began to starve. The load test never touched any of that machinery,
because the load test didn't do anything a customer does. It did one cheap,
cacheable read, very fast, and called it a day.

<<ch03>> gave you the workload model: which journeys exist, in what proportions,
at what rates. <<ch04>> turned "fast enough" into checkable objectives, and
<<ch05>> got the data and environment honest. This chapter is where those inputs
become an artifact — a scenario that behaves like a believable population of
users rather than a metronome hitting your cheapest endpoint. And a scenario is
code: it will be read, reviewed, misunderstood, and maintained, so much of the
craft here is keeping it legible while making it realistic.

=== A journey is not a list of URLs

The unit of scenario design is the *journey*: an ordered sequence of actions a
real user performs to accomplish something, with realistic pauses between them
and state carried from step to step. Not "the endpoints," but "what a person
does": land on the home page, search, poke at two or three products, maybe add
one to the cart. Your access logs and product analytics already know these
funnels; <<ch03>> showed you how to extract them. Here we encode one.

Three ingredients separate a journey from a URL list, and each one changes what
the system under test experiences:

*Sequence.* Real requests arrive in dependent order — you can't add to a cart
that doesn't exist, and the session, cache entries, and database pages the first
request warms are exactly what the second request hits. A random spray of
endpoints exercises none of that.

*Think time.* Humans pause — to read, to compare, to find their card. Those
pauses determine how long each virtual user holds its session, its cookies, and
(in some server designs) its resources, and they determine the concurrency that
a given user population produces. A journey with no think time is not "the same
test but faster"; it's a different population — recall from <<ch03>> that in a
closed model, halving think time roughly doubles offered load.

*State.* The token from login goes into the next request's header. The product
ID extracted from the search response is the one whose page gets fetched. Skip
this and you test a parade of strangers hitting error paths.

Here's a browse journey in loadr, built from those three ingredients:

[source,yaml]
----
name: browse-journey
description: land -> search -> view a product -> sometimes add to cart

defaults:
  http:
    base_url: https://shop.example.com

scenarios:
  browsers:
    executor: ramping-vus
    start_vus: 0
    stages:
      - { duration: 2m, target: 50 }   # ramp up
      - { duration: 5m, target: 50 }   # hold
      - { duration: 1m, target: 0 }    # ramp down
    think_time: { type: gaussian, mean: 2s, std_dev: 500ms }
    flow:
      - request: { name: home, url: / }
      - request:
          name: search
          url: "/search?q=widget"
          extract:
            - { type: jsonpath, name: sku, expression: "$.results[0].sku" }
      - request:
          name: product page
          url: "/products/${sku}"
          checks: [ { type: status, equals: 200 } ]
----

Note the shape: the *executor* says how many users and how they arrive, the
*think_time* says how humans pace themselves (a Gaussian around two seconds, not
a robotic constant — real pauses have variance, and the variance matters for how
requests clump), and the *flow* is the journey itself, with the SKU extracted
from the search response and reused. Every request has a `name`; hold that
thought for the next section.

How faithful should a journey be? Faithful to what the *server* can distinguish.
The server can't see the user scrolling, so don't model scrolling. It very much
can see the difference between "search then product page" and "product page
cold," between two seconds of think time and zero, between a real SKU and a
404. Model to that resolution and stop; every step past it is maintenance cost
with no measurement benefit.

TIP: Steal journeys, don't invent them. Your top ten user paths are sitting in
your analytics tool or derivable from session-stitched access logs. When a
scenario review asks "why these steps?", the answer should be "because that's
what the October logs say users do," never "it seemed reasonable."

=== Groups and transactions: naming the things you'll graph

When the run finishes, you will not be staring at YAML. You'll be staring at
metrics, and the only handles you'll have are the names you chose here. This is
worth more care than anyone gives it.

Two levels of naming matter. Every *request* gets a name — and this is
non-negotiable for parameterized URLs, because otherwise `/products/17`,
`/products/942`, and ten thousand friends each become their own metric series,
and your results are an unreadable smear of one-sample "endpoints" (your metrics
backend will also hate you; this is the cardinality explosion we return to in
<<ch11>>). Naming the request `product page` collapses them into one
distribution you can actually reason about.

Above requests sit *groups* — what JMeter calls Transaction Controllers and most
tools call transactions. A group brackets several steps under one
business-meaningful name and gives you an aggregate timing for the bundle:

[source,yaml]
----
    flow:
      - group:
          name: storefront
          steps:
            - request: { name: home, url: / }
            - request: { name: search, url: "/search?q=widget" }
      - group:
          name: cart
          steps:
            - request:
                name: add to cart
                method: POST
                url: /cart
                body: { json: { sku: "W-1" } }
                checks: [ { type: status, equals: 201 } ]
----

Groups earn their keep in conversations with people who don't read YAML. "The
`cart` transaction's p95 regressed from 400ms to 1.1s" lands in a product
meeting; "three of the seven requests got slower" doesn't. They're also the
natural place to hang SLOs from <<ch04>>: users experience "add to cart," not
your four internal calls, so gate on the group.

The concept is identical across tools — only the spelling changes. In k6 it's a
function call inside the script:

[source,javascript]
----
import { group } from 'k6';

group('cart', function () {
  http.post('https://shop.example.com/cart', JSON.stringify({ sku: 'W-1' }));
});
----

Gatling has `group("cart") { ... }`, Locust names each request explicitly via
`name="product page"` on the client call — same collapse-the-cardinality trick.

Naming rules that pay off later: low cardinality (a few dozen names, not
thousands); business vocabulary at the group level (`checkout`, not
`post_txn_v2`); stable across releases, because <<ch12>>'s regression
comparisons and <<ch15>>'s CI gates join on these names, and renaming one
silently breaks the comparison's memory.

=== Weighting the mix

A single journey, however faithful, is still a monoculture. Real traffic is a
mix, and the mix is measurable: in a typical retail workload something like 70
percent of sessions only browse, 20 percent search, and single-digit percent
buy. Get the proportions wrong and you get the retailer from the opening — a
test dominated by cheap reads that never generates enough of the expensive
writes to find the lock contention.

The lightweight way to encode a mix is a weighted branch inside one scenario:
each iteration rolls the dice and follows one path.

[source,yaml]
----
scenarios:
  mixed_traffic:
    executor: constant-vus
    vus: 50
    duration: 10m
    flow:
      - random:
          strategy: weighted
          choices:
            - weight: 70
              name: browse
              steps:
                - request: { name: home, url: / }
                - think_time: { type: uniform, min: 1s, max: 3s }
                - request: { name: category, url: "/c/widgets" }
            - weight: 20
              name: search
              steps:
                - request: { name: search, url: "/search?q=widget" }
            - weight: 10
              name: checkout
              steps:
                - request: { name: cart, url: /cart }
                - request:
                    name: checkout
                    method: POST
                    url: /checkout
                    checks: [ { type: status, equals: 200 } ]
----

Locust spells the same idea as weighted tasks — the concept survives the change
of tool:

[source,python]
----
class Shopper(HttpUser):
    wait_time = between(1, 3)

    @task(7)
    def browse(self):
        self.client.get("/")
        self.client.get("/c/widgets", name="category")

    @task(2)
    def search(self):
        self.client.get("/search?q=widget", name="search")

    @task(1)
    def checkout(self):
        self.client.get("/cart")
        self.client.post("/checkout")
----

(Gatling calls this `randomSwitch`; JMeter has the Throughput Controller.)

Weighted branches suit one *population* whose members behave differently on
different visits. But some workloads are genuinely different populations —
humans in browsers, API integrations, a batch job — with different pacing,
different load models, and different SLOs. Cramming those into one flow makes a
mess. Run them as separate scenarios, concurrently, each with its own executor:

[source,yaml]
----
scenarios:
  browsers:                       # closed model: a population of humans
    executor: ramping-vus
    stages: [ { duration: 1m, target: 30 }, { duration: 4m, target: 30 } ]
    think_time: { type: gaussian, mean: 2s, std_dev: 500ms }
    flow:
      - request: { name: home, url: / }

  api_clients:                    # open model: machines don't slow down
    executor: constant-arrival-rate
    rate: 50
    duration: 5m
    pre_allocated_vus: 20
    max_vus: 60
    flow:
      - request: { name: api inventory, url: /api/v1/inventory }

  nightly_export:                 # the batch job that always coincides with peak
    executor: per-vu-iterations
    vus: 1
    iterations: 1
    start_time: 2m                # kicks off two minutes into the test
    flow:
      - request: { name: export, method: POST, url: /api/v1/export, timeout: 120s }

thresholds:
  "http_req_duration{scenario:api_clients}": [ "p(99)<250" ]
  "http_req_duration{scenario:browsers}": [ "p(95)<800" ]
----

Notice the thresholds at the bottom: filtered per scenario, because "fast
enough" means different things for a human page load and a machine-to-machine
API call. One aggregate p95 across both populations is a number that describes
neither — the tyranny of the average, back in a new hat. And notice
`nightly_export` with its `start_time`: the most instructive incidents I've
worked involved a batch job landing on peak traffic, and this is how you find
out *before* production does.

WARNING: Wherever your weights come from, write the source next to them — a
comment like `# mix from prod access logs, 2026-05, checkout = 8.4%`. Unsourced
weights drift into fiction: someone bumps checkout to 25 percent "to be safe,"
and now the test over-stresses the payment path, under-stresses search, and its
capacity numbers apply to a workload that doesn't exist.

=== Shaping load over time: ramps, stages, and arrival-rate schedules

The mix says *what* the population does; the schedule says *how much of it
arrives, when*. You've met the two load models in <<ch03>> — closed (a fixed
population of VUs that each wait for responses) and open (work arrives at a rate
that doesn't care how slow you are) — and the executor names above encode the
choice: `ramping-vus` is closed, `constant-arrival-rate` and
`ramping-arrival-rate` are open.

Two rules of thumb carry most of the weight. First, *never step-function into
target load* unless a spike is precisely what you're testing. Real traffic ramps
— and so do JITs, caches, connection pools, and autoscalers. A test that goes
0-to-peak in one second measures cold-start behavior and calls it steady state.
Second, *hold long enough to trust the plateau*: minutes at target, not seconds,
so the transient warm-up washes out of your percentiles (<<ch12>> shows how to
check).

The `stages` list is the schedule notation, and the same list works for both
models. Closed, it ramps a user population (the `browse-journey` example above:
2m up, 5m hold, 1m down). Open, it ramps a rate — here's a spike test, since
spikes are where the open model is essential (real spikes don't politely slow
down when you do):

[source,yaml]
----
scenarios:
  spike:
    executor: ramping-arrival-rate
    start_rate: 20
    time_unit: 1s
    pre_allocated_vus: 100
    max_vus: 500
    stages:
      - { duration: 2m, target: 20 }    # baseline
      - { duration: 10s, target: 200 }  # spike!
      - { duration: 1m, target: 200 }   # sustained spike
      - { duration: 10s, target: 20 }   # recover
      - { duration: 2m, target: 20 }    # verify recovery

thresholds:
  http_req_failed:
    - { threshold: "rate<0.05", abort_on_fail: true, delay_abort_eval: 30s }
  dropped_iterations: [ "count<10" ]
----

Three details in there repay attention. The two minutes of baseline *after* the
spike is not padding — recovery is half of what a spike test measures, and
plenty of systems take the punch fine and then never quite come back (retry
queues, poisoned pools). The `abort_on_fail` threshold is a circuit breaker:
past 5 percent errors the run stops rather than flogging a corpse — kind to
shared environments, and it keeps the results file from being 20 minutes of
garbage. And `dropped_iterations` is the open model's honesty meter: when the
target can't absorb the arrival rate and the VU pool is exhausted, loadr counts
the iterations it *couldn't* start instead of silently slowing down. If that
counter is climbing, your latency percentiles describe only the requests that
got through — the test has found saturation, and this metric, not the latency
chart, is where it shows first.

Gatling's injection DSL spells the same schedule
`rampUsersPerSec(20).to(200).during(10.seconds)`; k6's `ramping-arrival-rate`
executor takes a `stages` array nearly identical to loadr's. The notation
converges because the underlying idea — a piecewise-linear rate schedule — is
the same everywhere.

=== Checks: making the journey prove it happened

Under load, systems lie in a specific way: they return HTTP 200 with an error
in the body. The search service times out, the front end catches it and renders
"something went wrong" with a cheerful 200, and your load test — which only
looks at status codes — reports success, faster than ever. Which points at the
deeper trap:

WARNING: *Failure is usually faster than success.* An error page skips the
database; a rejected login skips the bcrypt hash; an empty search result skips
ranking. A load test whose checks are weak degrades, under pressure, into a
benchmark of your error paths — and its latency numbers *improve* as the system
falls apart. If a run's p95 drops while load rises, check the checks before you
celebrate.

So checks inside the flow do double duty: they're assertions ("this step
worked"), and they're the evidence that your latency numbers describe
successes. Check semantics, not just status — that the token exists, that the
search returned results, that the order landed in a sane state:

[source,yaml]
----
      - request:
          name: login
          method: POST
          url: /login
          body: { form: { username: "${data.users.username}",
                          password: "${data.users.password}" } }
          checks:
            - { type: status, equals: 200 }
            - { type: jsonpath, expression: "$.token", exists: true }
      - request:
          name: place order
          method: POST
          url: /orders
          extract:
            - chain: order_status          # extract -> validate -> save, fused
              jsonpath: "$.status"
              transform: [lowercase]
              check:
                one_of: [pending, paid, shipped]
                on_failure: continue       # record the failure, keep the VU going
              default: pending
----

The second request shows a fused extract-and-check chain: pull the status,
normalize it, assert it's a state we recognize, save it for the next step, and
— via `on_failure: continue` — record a failed check without killing the VU's
iteration. That last choice is a policy decision worth making deliberately: in
a load test you usually want the population to keep behaving (a real user who
hits one glitch doesn't vanish mid-session), with the failure *counted*; in a
functional smoke pass you want the opposite.

Checks connect to <<ch04>>'s gates through one aggregate metric: every check
outcome feeds a `checks` success rate, and a threshold turns it into a verdict —
`checks: [ "rate>0.99" ]` for a load test, `rate>=1.0` when you're running the
same scenario as a CI contract test. The division of labor is clean: checks
judge individual responses, thresholds judge the run.

=== Per-VU state: making each user a user

A virtual user should be a *user* — its own identity, its own session, its own
memory. The moment a thousand VUs share one login, you're testing a system
where one cache-hot account does everything: the auth cache never misses, the
user-row lock becomes a bottleneck that doesn't exist in production, and
per-user rate limits either all trigger or never do. <<ch05>> made the case for
realistic data; per-VU state is where the scenario spends it.

State enters a VU three ways. *Extraction* you've seen: `extract` pulls values
from responses into the VU's variables, and `${token}` interpolates them later —
each VU carrying its own token, its own SKU. (Cookies get the same per-VU
treatment automatically: each VU has its own jar, so sessions don't bleed.)
*Feeders* hand each VU its own identity from external data, and the
`shared`-vs-`per_vu` distinction controls who sees what:

[source,yaml]
----
data:
  users:
    type: csv
    path: data/users.csv
    mode: per_vu        # each VU gets its own cursor over the file...
    pick: shuffle       # ...in its own shuffled order
    on_eof: recycle     # wrap around rather than retiring the VU
----

`mode: shared` (one cursor, VUs interleave rows) suits a case table you want
covered exactly once; `per_vu` suits identities, where each VU should *be* one
user rather than a different user every iteration.

The third mechanism is the *lifecycle*: some state should be established once
per user, not once per iteration. Real users log in once and then browse for
twenty minutes; a scenario that logs in every iteration hammers your auth
system at browse-rate, which is either an accidental stress test of bcrypt or a
wildly unrealistic one, depending on your mood. Lifecycle hooks split the flow:

[source,yaml]
----
js:
  file: scripts/lifecycle.js

scenarios:
  users:
    executor: constant-vus
    vus: 50
    duration: 10m
    on_start: login     # once per VU, before its first iteration
    exec: browse        # once per iteration
    on_stop: logout     # once per VU, when it retires
----

k6 draws the same lines with `setup()`/`default`/`teardown()` plus per-VU
initialization code; JMeter with Once Only Controllers; Gatling with what you
place before the loop in the chain. Every mature tool has this seam, because
every realistic scenario needs it. The design question is always the same:
*for each piece of state, what's its natural lifetime — request, iteration, or
user?* — and the auth-rate bug above is what happens when you answer it wrong
by one level.

=== Scenarios as code

Here's the test I apply before merging any scenario: *could a colleague who
wasn't in the room read this file and answer "what traffic does this simulate,
and why do we believe it?"* A scenario will outlive its author's memory of why
the checkout weight is 10, and unlike most code it makes *empirical claims*
that silently rot as your traffic evolves.

Concretely, the habits that keep scenario files reviewable:

*Factor out what repeats.* Base URLs, common headers, and default tags live in
`defaults`, not copy-pasted into forty requests. Environment differences
(staging vs. perf cluster) come in from outside — <<ch05>> covered the
mechanics — so the scenario file states behavior, not addresses.

*Keep the flow scannable; banish logic to named files.* A flow should read like
the user story it encodes. Loops and conditionals (loadr has `while`, `if`, and
`repeat` steps; the flow-control appetite is universal — Gatling's
`doIf`/`during`, Locust's plain Python) are sometimes exactly right — "browse 1
to 5 product pages" is more faithful than "browse exactly 2" — but three levels
of nesting in YAML is a smell. More than a line or two of JavaScript goes in a
referenced file (`js: { file: ... }`) where it can be linted and diffed like the
code it is.

*Tag scenarios so one file serves many purposes.* The same journeys, correctly
tagged, are your smoke test, your read-only soak, and your full mixed run:

[source,console]
----
$ loadr run --tags smoke scenarios/shop.yaml          # 30-second PR gate
$ loadr run --tags full --exclude-tags write scenarios/shop.yaml  # read-only soak
$ loadr run scenarios/shop.yaml                        # the whole population
----

with each scenario carrying tags like `tags: { suite: smoke, kind: read }`.
One reviewed artifact, many runs — instead of four drifting copies that agree
in January and diverge by June. This is the seam <<ch15>> hangs CI on.

*Review the claims, not just the syntax.* A scenario PR review should ask:
where did the weights come from, and are they dated and sourced in a comment?
Does every request have a low-cardinality name? Does every step that can fail
carry a check that would notice? Is any state shared between VUs that
production users wouldn't share? Is the ramp long enough to warm the system,
and the hold long enough to trust? None of these is a syntax question. All of
them are the difference between the forty lines that passed and the Black
Friday that didn't.

The scenario is now believable. <<ch07>> surveys the tools you might run it
with, and in <<ch08>> we finally take a scenario like this one end to end —
execution, live observation, and the first honest read of the results.

.Key takeaways
****
* A journey is sequence + think time + state, not a list of URLs. Model what
  the server can distinguish; steal the journeys from your logs and analytics
  rather than inventing them.
* Name every request and group steps into business-meaningful transactions —
  the names you choose here are the metric handles, SLO subjects, and
  regression-comparison keys you'll live with for years.
* Weight the mix from measured traffic and write the source next to the
  weights. Use weighted branches for one population's varied behavior; use
  separate concurrent scenarios — each with its own executor and thresholds —
  for genuinely different populations.
* Ramp, hold, recover: schedules are piecewise rate plans, and the choice of
  closed (`ramping-vus`) vs. open (`arrival-rate`) executor decides what the
  test can even measure. Watch `dropped_iterations` — it's where saturation
  shows first.
* Failure is faster than success. Check semantics, not just status codes, or
  your test will degrade into an ever-faster benchmark of your error paths.
* Give state its natural lifetime — request, iteration, or user — via
  extraction, per-VU feeders, and lifecycle hooks. A thousand VUs sharing one
  login is a different (and easier) system than the one you ship.
* A scenario is code that makes empirical claims. Factor out repetition, tag
  for selective runs, and review the claims — weights, names, checks, ramps —
  not just the YAML.
****

[.fieldcard]
.Field Card — Building believable scenarios
****
Reach for this when a passing test doesn't move under real traffic: a journey is *sequence + think time + state*, not a list of URLs.

One population, varied behavior:: weighted `random` branch inside one scenario.
Genuinely different populations:: separate concurrent scenarios, each its own executor + threshold.
Humans:: `ramping-vus` (closed) with `think_time: { type: gaussian, mean: 2s }`.
Machines / spikes:: `constant-arrival-rate` or `ramping-arrival-rate` (open) — watch `dropped_iterations`.
State's lifetime:: request → `extract`; user identity → per_vu feeder; login → `on_start` hook.

----
loadr run --tags smoke scenarios/shop.yaml                       # 30s PR gate
loadr run --tags full --exclude-tags write scenarios/shop.yaml   # read-only soak
loadr run scenarios/shop.yaml                                    # full mix
----

*Remember:* name every parameterized request (`product page`, not `/products/17`) or cardinality explodes. Failure is faster than success — check semantics (`jsonpath ... exists: true`), not just `status: 200`. Ramp minutes, not seconds. Date your weights in a comment: `# mix from prod logs 2026-05, checkout=8.4%`.

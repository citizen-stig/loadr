# Design Spec — AI Copilot (`loadr explain` + NL→scenario)

Status: draft · Owner: perf-tooling · Crates touched: `loadr-ai` (new), `loadr-cli`, `loadr-config`, `loadr-core`

## 1. Goal & user story

Two LLM-backed authoring/analysis affordances, one binary, no new services:

- **`loadr explain <summary.json>`** — a root-cause narrative over a completed run.
  Grounds the model in the real `loadr-core::Summary` (metrics, checks, thresholds),
  the per-interval `timeline` (RPS, latency percentiles, error rate) and the
  `observe:` external system metrics carried on `TimelinePoint::external`
  (CPU/mem from the `System` collector, PromQL series from the `Prometheus`
  collector). Optionally a `--baseline` for regression-aware explanations.
- **`loadr scenario "500 rps on checkout for 10m"`** — natural language → a valid
  loadr YAML plan, schema-constrained and self-repaired, mirroring the existing
  desktop AI flow.

One-command experience:

```
$ loadr run checkout.yaml --summary-export run.json
$ loadr explain run.json
  p95 climbed from 180ms to 1.4s after ~90s. The knee coincides with system_cpu
  crossing 95% (observe: host). checkout POST error_rate hit 4.2%, breaching
  http_req_failed<0.01. Likely CPU saturation on the target, not the client…

$ loadr scenario "500 rps on checkout for 10m, ramp over 1m" -o checkout.yaml
  ✓ wrote checkout.yaml (0 warning(s))
```

## 2. CLI / API surface

New subcommands in `crates/loadr-cli/src/main.rs` (`Command` enum + `run` match),
styled after `Convert`/`Compare`:

```
loadr explain <SUMMARY>            Summary JSON from `loadr run --summary-export`
  --baseline <PATH>                Baseline summary → regression-aware narrative
  --observe                        Include external/observe series in the digest (default: auto)
  --provider <bedrock|anthropic|openai>   Override LOADR_AI_PROVIDER
  --model <ID>                     Override the default model
  --format <text|markdown|json>    Narrative rendering (default text)
  --offline                        Skip the LLM; deterministic rule-based narrative
  -o, --output <PATH>              Write narrative to a file (default stdout)

loadr scenario <PROMPT>            Natural-language description of the test
  -o, --output <PATH>              Output YAML (default stdout) — mirrors `loadr convert`
  --provider / --model             As above
  --base-url <URL>                 Seed defaults.http.base_url
  --strict                         Exit non-zero if the plan still fails `loadr validate`
```

Both reuse `loadr convert`'s output convention: a `# Generated by loadr scenario`
header, warnings to stderr, plan to stdout/`-o`. `explain --format json` emits a
typed `Explanation` struct (snake_case, per the project's structured-output rule).

## 3. Architecture

New in-tree crate **`crates/loadr-ai`** — *not* a runtime plugin. The runtime
plugin ABI (`loadr-plugin-api`) exists for protocols and per-VU hot-path work;
the copilot is a build/analysis-time tool that needs direct access to in-tree
types (`loadr_config::TestPlan`, `loadr_config::json_schema()`, `loadr_config::validate`,
`loadr_core::Summary`) and has no hot path. An installable plugin would force an
ABI boundary and re-marshalling of exactly those types for no benefit. It is
gated behind a default-on cargo feature `ai` so `--no-default-features` still
produces the lean offline binary.

`loadr-ai` contains:

- `provider.rs` — the `LlmProvider` seam (below) with `bedrock.rs`, `anthropic.rs`,
  `openai.rs` implementations. HTTP uses **hyper** (the repo's chosen client, per
  architecture notes — not reqwest); Bedrock uses `aws-sdk-bedrockruntime`.
- `scenario.rs` — NL→plan. This is a direct Rust port of `desktop/src/shared/ai.ts`
  + `desktop/src/main/ai.ts::generatePlan`: build system+user messages, embed
  `loadr_config::json_schema()`, `extract_yaml()`, `loadr_config::validate`, one
  repair round. Returns `loadr_convert::Conversion` (reused verbatim — `plan` +
  `warnings`) so `loadr scenario` shares `convert.rs`'s YAML-writing tail.
- `explain.rs` — builds a compact `RunDigest` from `Summary` + `timeline` +
  `external` + optional baseline `Comparison`, prompts the model, returns
  `Explanation`.
- `prompt.rs` — system prompts + digest serialization.

`loadr-cli/src/commands/explain.rs` and `scenario.rs` are thin: load inputs
(reuse `report.rs`/`compare.rs`'s `load_summary`), call `loadr-ai`, render.
`explain.rs` reuses `commands::compare::diff` for `--baseline`; that requires
promoting `compare::diff` and `Comparison` from `pub(crate)` to `pub` (or
lifting them into `loadr-core`, preferred — the digest builder wants them too).

## 4. Key data structures & algorithms

```rust
// provider.rs — the seam (mirrors the desktop ChatFn injection).
#[async_trait]
pub trait LlmProvider: Send + Sync {
    async fn complete(&self, req: &ChatRequest) -> Result<String, AiError>;
}
pub struct ChatRequest { pub system: String, pub messages: Vec<ChatMessage>,
                         pub model: String, pub max_tokens: u32 }
pub struct ChatMessage { pub role: Role, pub content: String } // User | Assistant

// Resolution: LOADR_AI_PROVIDER (or --provider) selects; each provider self-checks
// credentials and returns Provider::Unconfigured if absent.
pub fn resolve(cfg: &AiConfig) -> Result<Box<dyn LlmProvider>, AiError>;
```

**explain digest** — the anti-hallucination core. Rather than dump raw JSON, emit
a token-bounded digest: thresholds (`ThresholdStatus`), per-metric p50/p95/p99 +
error_rate, a *downsampled* `timeline` (≤40 points, keeping the max-latency and
max-error inflection points), and every `external` series aligned to those
points. If `--baseline`, append the `Comparison` rows flagged `regression:Some(true)`.
The model is told: "explain only what the data shows; cite metric names and
elapsed_secs." This makes the narrative auditable against the digest.

**scenario algorithm** (ported): `messages=[user]` → `complete` → `extract_yaml`
(fenced-block regex, else bare-plan heuristic from `ai.ts`) → `loadr_config::validate`.
On error, one repair turn feeding `diagnostics` back, exactly as `generatePlan`.
Schema-constrained generation = the authoritative `json_schema()` embedded in the
user message (already the desktop mechanism) so output validates against the same
`deny_unknown_fields` types the runtime parses.

**Offline degradation** — `explain --offline` (and auto-fallback when
`resolve` returns `Unconfigured`) runs a deterministic analyzer over the same
`RunDigest`: report failed thresholds, the largest error-rate/latency interval
from `timeline`, and any `external` series whose rise correlates (Pearson over
aligned points) with the latency knee. No network, no key. `scenario` cannot
degrade — with no provider it errors with setup guidance and exits non-zero.

## 5. Reuse map

| Capability | Reuse (existing) | Net-new |
|---|---|---|
| Plan types + JSON Schema | `loadr_config::{TestPlan, json_schema}` | — |
| Plan validation | `loadr_config::validate` | — |
| NL→plan orchestration | logic ported from `desktop/src/{shared,main}/ai.ts` | Rust port in `scenario.rs` |
| YAML output tail | `commands/convert.rs` pattern + `loadr_convert::Conversion` | shared helper |
| Run inputs | `loadr_core::{Summary, TimelinePoint, ThresholdStatus}`, `report.rs::load_summary` | — |
| Baseline regressions | `commands/compare.rs::{diff, Comparison}` (promote to `pub`) | digest wiring |
| Observe series | `TimelinePoint::external`, `plan::ObserveConfig` | digest alignment |
| CLI plumbing | `main.rs` `Command` enum + `run` match | 2 subcommands |
| HTTP / Bedrock | `hyper`, `aws-sdk-bedrockruntime` | 3 provider impls |

Net-new is small: the provider seam (~3 transports), the digest builder, and two
thin command files. Everything domain-specific already exists.

## 6. Testing plan

Mirrors repo conventions (70% coverage gate, no real network in unit tests, the
`newX = || ...` injection seam from CLAUDE.md §7):

- **Unit (`loadr-ai`)**: `MockProvider` implementing `LlmProvider` returns canned
  completions. Test `scenario` happy path, extract-failure error, and the
  repair round (first response invalid YAML → second valid) — parallels
  `ai.test.ts`. Test `explain` digest downsampling keeps inflection points, and
  the offline analyzer flags a seeded threshold failure + CPU correlation.
- **Provider body mappers**: pure functions (`anthropic_body`, `openai_body`,
  `bedrock_body`) tested like the desktop `openAiBody`/`googleBody` — assert
  request JSON shape, `anthropic-version` header, model id, no secret leakage.
- **Integration (`loadr-cli/tests`)**: `loadr scenario --offline`-style with a
  stub provider via `LOADR_AI_PROVIDER=mock` (env-selected test transport) →
  assert emitted YAML passes `loadr_config::validate`. `loadr explain --offline`
  over a fixture `run.json` → assert narrative names the failing threshold.
- **Redaction test**: assert API keys never appear in `tracing` output or errors.
- Run with `-race`-equivalent `cargo test` + clippy in the existing quality gate.

## 8. Milestones

- **M1 (smallest shippable, ~3d)** — `loadr scenario` with the **Anthropic**
  provider only, ported from desktop `generatePlan` (system prompt + schema +
  validate + one repair). Reuses `Conversion` and the convert output tail. Ships
  NL→YAML end to end. *Effort: S.*
- **M2 (~2d)** — `loadr explain` over `Summary` + `timeline` (no observe, no
  baseline) with the deterministic **offline** analyzer as the default, LLM
  narrative when a key is present. Establishes the digest + `Explanation` type.
  *Effort: S–M.*
- **M3 (~3d)** — Provider abstraction hardening: **Bedrock** (`aws-sdk-bedrockruntime`,
  AWS cred chain / aws-vault) + **OpenAI-compatible** transports; `--provider`/`--model`;
  secret redaction; `--baseline` via promoted `compare::diff`. *Effort: M.*
- **M4 (~2d)** — Observe correlation in `explain` (external series alignment +
  Pearson knee detection), `--format json/markdown`, desktop Explain tab, book
  chapter + demo. *Effort: M.*

### Risks & hard parts

- **Hallucination in `explain`** — mitigated by the grounded, downsampled digest
  and "cite the data" prompting; the offline analyzer doubles as a factual floor.
- **Model-id drift** — keep ids in one `models.rs` table; Bedrock needs the
  `anthropic.`-prefixed id (`anthropic.claude-opus-4-8`) or a region inference
  profile — a common 400 source; validate at request build.
- **Secret handling** — never log request bodies; source keys only from env / AWS
  chain (per constitution §8); redact in `AiError`. Easy to leak via `tracing`.
- **Determinism of NL→plan** — LLMs drift; the `validate`+repair loop is the
  guardrail, but `--strict` may still fail. Keep the loop to one repair (as
  desktop does) to bound latency/cost.
- **Coverage gate** — network transports are the hard-to-cover part; the seam +
  pure body-mapper split (proven in `ai.test.ts`) keeps unit coverage above 70%
  without real calls.

== The Tooling Landscape

_An honest, opinionated survey. What each major tool is good at, what it costs
you, and how to choose._

A bank I worked with spent eleven weeks selecting a load-testing tool. There was
a steering committee, a weighted scoring matrix with thirty-one criteria, a
forty-page comparison document, and two vendor bake-offs. The winning tool was
perfectly fine. The first test the team wrote with it simulated 10,000
concurrent users looping as fast as they could against an endpoint whose real
production traffic peaked at 40 requests per second — a closed-model hammer test
against a system that would never see anything like it (<<ch03>> explains
exactly why that matters). Eleven weeks on the tool, one afternoon on the
workload model. Any of the eight tools on their shortlist could have run that
test, and every one of them would have produced the same wrong answer.

That's the first thing to understand about load-testing tools: the choice is
the most over-weighted decision in this field. The chapters you've already read
— workload modeling, objectives, data, scenario design — determine whether your
test tells the truth. The tool mostly determines how pleasant it is to write,
run, and automate.

Mostly. Not entirely. A tool can quietly sabotage honest work: it can report
only averages, it can offer only a closed load model, it can fall victim to
coordinated omission (<<ch12>>) by design, or it can be so hostile to version
control and CI that your tests rot in a shared folder instead of running on
every commit. So the choice deserves an afternoon of clear thinking — just not
eleven weeks. This chapter is that afternoon.

=== Seven questions before any tool name

Evaluate tools against your situation, not against each other. These are the
seven questions that actually discriminate, roughly in the order they should
eliminate candidates.

*1. Protocol coverage.* Can it speak what your system speaks — natively, not
through a plugin of unknown quality? Plain HTTP is table stakes. If you need
gRPC, WebSockets, Kafka, JDBC, JMS, or MQTT, the field narrows fast, and this
question alone often makes the decision (<<ch09>> covers testing beyond HTTP).
A tool that covers 90 percent of your protocols leaves you running two tools,
which means two skill sets, two report formats, and two CI integrations.

*2. Scripting model.* Who writes the tests, and in what? A YAML file, a
JavaScript module, a Python class, a Java DSL, or a GUI-generated XML document
are very different artifacts to write, review, and maintain. Match this to the
team you actually have, not the team you wish you had.

*3. Load model fidelity.* Can it express an open model — arrivals per second,
independent of response time — or only a closed pool of looping users? You
learned in <<ch02>> and <<ch03>> why this is not cosmetic: a closed-only tool
cannot run the most important class of test honestly, because its own workers
slow down in sympathy with the system under test. And ask how the tool records
latency: does it track full distributions (HDR histograms or equivalent) or
does it average as it goes? A tool that can only tell you the mean is a tool
that can only mislead you.

*4. Scale-out story.* One generator machine tops out — somewhere between
hundreds and hundreds of thousands of requests per second depending on the tool
and the payload (<<ch10>>). When you outgrow it, what happens? Built-in
distribution, a paid cloud, a DIY pile of scripts, or nothing? And critically:
when results come from ten machines, does the tool *merge the histograms* or
average the percentiles? Averaged percentiles are statistically meaningless,
and more than one popular setup produces them.

*5. Reporting and analysis.* Does the output include real percentiles, error
breakdowns, and time series you can correlate with server-side telemetry
(<<ch11>>)? Can it compare this run against the last one? Can a machine parse
the result and fail a build with it?

*6. CI citizenship.* Can it run headless from one command, exit nonzero when a
threshold fails, and live in a container without dragging a runtime along? A
tool you can't put in a pipeline gives you performance testing as an event, not
a practice — and <<ch14>> will argue that the practice is the whole point.

*7. Cost and skills.* Almost every tool below is open source at the core, with
a paid cloud for distributed execution and dashboards. The real cost is rarely
the license; it's the learning curve multiplied by everyone who has to touch
the tests, plus the paid tier you'll hit the moment you need serious scale.
Price that honestly up front.

[TIP]
====
Weight the questions by your actual bottleneck. If your team is three Python
developers testing a REST API, question 2 dominates and Locust probably wins
regardless of anything else on the matrix. If you're testing a JMS-fronted
mainframe adapter, question 1 has already decided for JMeter. Most bake-offs
are elaborate rituals for discovering which question was decisive all along.
====

=== GUI, code, or hybrid

Underneath the logos, there are three philosophies of what a load test *is*.

*GUI-first* tools (JMeter is the archetype; the commercial LoadRunner lineage
lives here too) treat a test as a document you assemble in an editor: drag in a
thread group, add samplers, attach listeners. The strength is accessibility —
a tester who doesn't program can build a working test in an hour, and there's
usually a recorder that captures a browsing session into a runnable script. The
costs show up at exactly the moments this book cares about. The saved artifact
is machine-generated XML that no human can meaningfully code-review; a
one-field change produces a fifty-line diff. Recorded tests capture one
session's cookies, timestamps, and session IDs and rot the moment the
application changes. And a test nobody can review is a test nobody trusts
enough to gate a release with.

*Code-first* tools (k6, Gatling, Locust) treat a test as a program. Tests live
in the repository next to the code they exercise, go through pull-request
review, get refactored into shared libraries, and run in CI like any other
test suite. The cost is the obvious one: someone has to program, and the load
test inherits every failure mode of software — including bugs in the test
itself, which now require the same discipline to catch.

*Hybrid* tools (Artillery, loadr, and Taurus — a YAML front-end that drives
JMeter and others) bet that most of a load test is declarative — endpoints,
rates, durations, thresholds — and only a sliver is logic. You describe the 90
percent in YAML and drop into JavaScript (or another language) for the 10
percent that needs computation: signing a request, generating a payload,
correlating a token. When the bet pays off, you get code-review-friendly diffs
*and* an escape hatch. When it doesn't — when your scenario is mostly logic —
you end up programming in YAML, which is nobody's idea of a good time.

None of these is "right." But notice that the axis correlates with question 6:
GUI-first tools all have headless modes, yet teams that build tests by
clicking tend not to run them on every commit, because the artifact resists
the workflows that automation depends on. Choose the philosophy that matches
where you want the tests to *live*.

=== The JVM tax

Two of the most capable tools in this survey — JMeter and Gatling — run on the
JVM, and it's worth being precise about what that costs, because the folklore
("Java is slow") is wrong and the real tax is elsewhere.

Throughput is not the problem. A warmed-up JVM generates load extremely well;
Gatling in particular is among the most efficient generators per core in this
chapter. The tax is operational:

* *Startup and warm-up.* JVM start plus JIT warm-up adds tens of seconds
  before measurements stabilize. In a long soak test, irrelevant. In a
  90-second CI smoke test (<<ch14>>), it's a meaningful slice of the budget,
  and the un-warmed early samples can pollute your numbers.
* *Memory floor.* A JVM generator wants hundreds of megabytes before it does
  anything. Ten containerized agents at 1 GB each is a real bill; the same
  fleet of static Go or Rust binaries runs in a tenth of that.
* *GC pauses on the measuring side.* A garbage-collection pause in the *load
  generator* stops the clocks mid-measurement. Modern collectors keep pauses
  to milliseconds, but under memory pressure a paused generator both distorts
  recorded latencies and momentarily stops offering load — a measurement
  artifact you must rule out before believing a weird spike (<<ch10>>).
* *Dependency weight.* "Install Java 17, then the tool, then the plugins" is a
  heavier ask of a CI image than "copy one binary." Container images tell the
  story: a JMeter image is hundreds of megabytes; single-binary tools ship
  images a tenth the size that cold-start in milliseconds.

The tax is real and usually affordable. Pay it knowingly, and when you're
running distributed agents or tight CI loops, remember that it compounds.

=== The field guide

What follows is one practitioner's honest read of the landscape as it stands.
Tools evolve; the shape of each one's trade-offs has been stable for years.

==== JMeter

Apache JMeter is the industry's default answer, in continuous development
since 1998, and its superpower is coverage: HTTP, JDBC, JMS, LDAP, FTP, SMTP,
and a plugin ecosystem that has an answer for nearly everything — if you need
to load-test something strange behind a corporate firewall, JMeter probably
already does it. It's GUI-first with a solid headless mode (`jmeter -n -t
plan.jmx`), and its HTML dashboard report is genuinely decent, percentiles
included.

The costs: test plans are unreviewable XML; the thread-per-user engine makes
each simulated user expensive, so large closed-model tests need big machines
or its (aging) distributed mode; and the load model is closed by default —
open-model behavior requires the Precise Throughput Timer or the experimental
Open Model Thread Group added in 5.5, both of which most JMeter users have
never touched. JMeter will do almost anything, but it makes the honest thing
the effortful thing.

==== Gatling

Gatling treats a load test as a program in a fluent DSL (originally Scala; the
Java and Kotlin DSLs are now first-class) built on an async engine that makes
individual virtual users nearly free. Its injection profiles are the cleanest
expression of open-versus-closed in any tool — `constantUsersPerSec(50)`
versus `constantConcurrentUsers(50)` forces the modeling decision into the
test's first line, which after <<ch03>> you'll recognize as a feature. Reports
are excellent, and assertions gate CI runs properly.

The costs: the JVM tax above; a build-tool ecosystem (Maven/Gradle/sbt) that
web developers find heavy; and distribution beyond one machine is where
Gatling wants to sell you Gatling Enterprise. If your team lives on the JVM
already, Gatling is probably the strongest code-first choice there is.

==== k6

k6 (now Grafana k6) is a single Go binary that runs test scripts written in
JavaScript — an embedded interpreter, not Node.js, so the language is familiar
but the npm ecosystem mostly isn't. Its scenario executors
(`constant-arrival-rate`, `ramping-vus`, and friends) put open and closed
models on equal footing, thresholds give you machine-checkable gates in one
line, and the whole thing containerizes and drops into CI beautifully. The
extension mechanism (xk6) compiles custom protocols into the binary.

The costs: JavaScript-only scripting; anything beyond bundled protocols means
xk6 builds or waiting on extensions; and native distributed execution is the
paid Grafana Cloud story (the open-source workarounds exist but you own the
result-merging problem — see question 4). k6 is the current default
recommendation for HTTP-centric teams, and deservedly so.

==== Locust

Locust is Python all the way down: a user is a class, a request is a method,
and anything you can do in Python — custom protocols, weird auth, business
logic — you can do in a test. Distributed mode (one master, many workers) is
built in, free, and unusually easy, and the live web UI is pleasant for
exploratory testing.

The costs cluster around measurement. The model is closed by default — users
with `wait_time` — and the standard trick for approximating an arrival rate
(`constant_throughput` pacing, shown below) holds only while responses stay
fast, which is precisely the regime you're testing your way out of. The GIL
means one worker process per core. And Locust trades percentile precision for
memory, rounding recorded response times to about two significant figures —
fine for a p95 around 200ms, blunt at the p99.9 tail. Locust is the right
tool when your team is Python-native and test *logic* is the hard part; treat
its tail percentiles as estimates.

==== Vegeta and wrk

Two small, sharp knives. *Vegeta* is a Go binary with exactly one idea: attack
a list of HTTP targets at a constant rate and report the latency distribution.
No scenarios, no sessions, no scripting — targets in, histogram out, composable
with pipes:

[source,console]
----
$ echo "GET https://api.example.com/products" | \
    vegeta attack -rate=50 -duration=2m | \
    vegeta report -type=text
----

*wrk* is the opposite trade: a C tool of brutal efficiency, closed-model,
scriptable in Lua, capable of saturating a 10 Gb link from one box. Its
history carries a lesson worth the price of this chapter: classic `wrk`
measures in a closed loop and therefore under-reports latency the moment the
server stalls — coordinated omission in the wild. Gil Tene's fork, *wrk2*,
exists specifically to fix that: it holds a fixed throughput (`-R`) and
back-fills the latency the stall actually caused. The full story is in
<<ch12>>; the moral is that even a benchmarking tool embeds a statistical
worldview, and one of the two available worldviews is wrong.

Use these for calibration and quick answers — "what's this endpoint's floor?",
"is the load balancer the bottleneck?" — not for scenario testing. In the same
family: `hey`, `oha`, and the ancient `ab`, with the same trade-offs.

==== Artillery

Artillery is the hybrid philosophy on Node.js: YAML scenarios with arrival-rate
phases as the native load model, JavaScript hooks for logic, first-class
Playwright integration for driving real browsers under load, and a clever
serverless distribution mode that fans out across AWS Lambda or Fargate. The
costs: a Node runtime per worker puts a lower ceiling on per-machine
throughput than the compiled tools, so you scale out earlier; and the
sharpest features pull toward the paid cloud. A natural fit for
JavaScript-native teams, especially ones that need browser-level load.

==== Tsung

Tsung is the old Erlang heavyweight: multi-protocol (HTTP, XMPP, MQTT,
PostgreSQL, MySQL, LDAP), natively distributed, and capable of millions of
concurrent connections from modest hardware — the Erlang VM was built for
exactly this shape of problem. The costs: XML configuration, sparse
maintenance, dated reporting, and extending it means writing Erlang. It
survives in this survey because for massive-connection-count protocols — chat,
messaging, MQTT fan-in — it still does things few others can. A niche tool
that owns its niche.

==== loadr

loadr is the tool this book uses for most worked examples, and — full
disclosure, as in the preface — I wrote it, so audit this entry hardest. It's
a single static binary; tests are YAML with a JavaScript escape hatch; the
executor model puts open and closed on equal footing (deliberately borrowing
k6's executor vocabulary so the concepts transfer); latency is recorded in HDR
histograms end to end; and distribution is built in and free — a controller
partitions rates across agents and merges histograms centrally rather than
averaging percentiles. Comparison of runs (`loadr compare`) and threshold
gates are first-class because <<ch12>> and <<ch14>> argue you should demand
them from any tool.

The honest costs: it's young, so the community, Stack Overflow answer supply,
and plugin ecosystem are a fraction of JMeter's or k6's; some protocols are
install-time plugins rather than built-ins; and a heavily logic-driven
scenario pushes you into the JavaScript escape hatch, at which point a
code-first tool may serve you better. Nothing in this book depends on
choosing it.

=== One test, four spellings

The fastest way to see that the concepts outrank the tools is to write the
same test in several of them. The test: 50 requests per second — an open
model — against `/products` for two minutes, failing the run if the p99
exceeds 300ms or more than 1 percent of requests error.

In *loadr*:

[source,yaml]
----
# products-baseline.yaml
name: products-baseline
description: 50 req/s open-model baseline against /products

defaults:
  http:
    base_url: https://api.example.com

scenarios:
  products:
    executor: constant-arrival-rate
    rate: 50                  # iterations per time_unit
    time_unit: 1s
    duration: 2m
    pre_allocated_vus: 25
    max_vus: 100
    flow:
      - request:
          name: products
          url: /products
          checks:
            - { type: status, equals: 200 }

thresholds:
  http_req_duration: [ "p(99)<300" ]
  http_req_failed: [ "rate<0.01" ]
----

[source,console]
----
$ loadr run products-baseline.yaml
----

In *k6* — notice the executor vocabulary is deliberately shared, so the file
is a near-transliteration:

[source,javascript]
----
import http from 'k6/http';
import { check } from 'k6';

export const options = {
  scenarios: {
    products: {
      executor: 'constant-arrival-rate',
      rate: 50, timeUnit: '1s', duration: '2m',
      preAllocatedVUs: 25, maxVUs: 100,
    },
  },
  thresholds: {
    http_req_duration: ['p(99)<300'],
    http_req_failed: ['rate<0.01'],
  },
};

export default function () {
  const res = http.get('https://api.example.com/products');
  check(res, { 'status 200': (r) => r.status === 200 });
}
----

In *Gatling* (Java DSL), where the open model is an injection profile and the
gate is an assertion:

[source,java]
----
public class ProductsSimulation extends Simulation {

  HttpProtocolBuilder httpProtocol =
      http.baseUrl("https://api.example.com");

  ScenarioBuilder scn = scenario("products")
      .exec(http("products").get("/products")
          .check(status().is(200)));

  {
    setUp(scn.injectOpen(
            constantUsersPerSec(50).during(Duration.ofMinutes(2))))
        .protocols(httpProtocol)
        .assertions(
            global().responseTime().percentile(99.0).lt(300),
            global().failedRequests().percent().lt(1.0));
  }
}
----

And in *Locust*, where the closed model shows through: there is no arrival-rate
executor, so the idiom is pacing — enough users, each throttled to a fixed
personal throughput:

[source,python]
----
from locust import HttpUser, task, constant_throughput

class ApiUser(HttpUser):
    # 500 users x 0.1 req/s each ~= 50 req/s -- but only while the
    # server keeps up. If responses slow past 10s, offered load sags:
    # the closed-model trap from Chapter 3, visible in the code.
    wait_time = constant_throughput(0.1)

    @task
    def products(self):
        with self.client.get("/products", catch_response=True) as r:
            if r.status_code != 200:
                r.failure(f"got {r.status_code}")
----

The JMeter equivalent exists — a Thread Group, an HTTP Sampler, a Precise
Throughput Timer, a Duration Assertion — but the saved `.jmx` runs to well
over a hundred lines of generated XML, which is itself the datapoint: the test
is fine, the *artifact* is hostile to review.

Four tools, one hypothesis. The parts that carry the meaning — open model, 50
per second, two minutes, p99 under 300ms — survive every translation. That's
the skill to build: fluency in the model, with the tool as an accent.

=== Single binaries, containers, and CI

A pattern in the table above deserves a name: the compiled single-binary tools
(k6, Vegeta, wrk, loadr) have quietly won the CI war, for mechanical reasons.
A static binary has no runtime dependency graph — no JVM version, no
`node_modules`, no Python environment to reconstruct in a pipeline image. It
cold-starts in milliseconds, which matters when a smoke test's whole budget is
two minutes. And it produces small containers, which matters when <<ch10>>
has you launching twenty ephemeral generator pods.

Containerization flattens some of this — `docker run` makes even JMeter a
one-liner — but it doesn't flatten image size, startup time, or memory floor,
and in a pipeline you pay those on every single run. The practical CI
checklist, previewing <<ch14>>: one command to run headless, thresholds
evaluated in-process, nonzero exit on failure, machine-readable results out.
Every tool in this chapter can be made to do this; the single binaries do it
without ceremony, and ceremony is what erodes practices.

[NOTE]
====
Distribution deserves the same scrutiny. When one generator machine isn't
enough, the free built-in options (Locust, Tsung, loadr, JMeter's remote mode)
and the paid clouds (k6, Gatling, Artillery) differ most in one detail from
question 4: whether cross-machine percentiles come from merged histograms or
averaged summaries. Ask the vendor. If the answer is a blank look, assume
averaged, and treat the fleet's p99 as fiction. <<ch10>> covers this in anger.
====

=== When the "worse" tool is the right call

Everything above is about the tool in isolation. Tools are never in isolation,
and there are four situations where the objectively weaker choice is the
correct one.

*The team already knows it.* A tool your team is fluent in beats a better tool
they'll fight. Three years of JMeter muscle memory, a library of working test
plans, and an engineer who knows exactly why the Precise Throughput Timer is
configured that way are assets a migration destroys and a scoring matrix never
prices. The switching cost is real even when the destination is better.

*The tests already exist.* Five hundred maintained JMeter plans encoding years
of accumulated knowledge about your system's quirks are worth more than a
greenfield k6 suite of six. Rewrite when the old tests are wrong or rotten,
not because the new tool is nicer.

*The protocol decides.* If your load has to speak JMS to a message broker or
exercise a JDBC pool directly, JMeter's plugin shelf settles the argument
regardless of how you feel about XML. A tool that can't reach your system is
worth exactly nothing; see question 1.

*The organization constrains.* Regulated environments, procurement rules,
support-contract requirements, an enterprise standard already bought and paid
for. Running the sanctioned tool adequately beats running the perfect tool in
the shadows, where its results gate nothing and outlive nobody's tenure.

The uncomfortable summary: a mediocre tool driving a well-modeled, honestly
measured, CI-gated test is worth more than the best tool in the survey driving
the bank's 10,000-user fantasy from the top of this chapter. Spend your
disagreement budget on the workload model and the gates. The tool will
forgive you; the model won't.

.Key takeaways
****
* Tool choice is the most over-weighted decision in performance testing. The
  workload model, objectives, and analysis determine whether a test tells the
  truth; the tool determines how easily.
* Evaluate against seven questions — protocols, scripting, load-model
  fidelity, scale-out, reporting, CI citizenship, cost and skills — and weight
  them by *your* bottleneck. Most bake-offs are decided by one question.
* Demand two non-negotiables from any tool: an open (arrival-rate) load model
  and honest latency recording — full distributions, merged (never averaged)
  across machines.
* GUI-first, code-first, and hybrid are philosophies about where tests live.
  Tests that live in version control and CI stay alive; tests that live in a
  GUI rot.
* The JVM tax is operational — startup, memory floor, GC pauses on the
  measuring side — not throughput. Pay it knowingly; it compounds across
  agents and CI runs.
* The concepts outrank the tools: the same open-model, threshold-gated test
  survives translation across loadr, k6, Gatling, and (with a visible
  workaround) Locust. Build fluency in the model, not the accent.
* Sometimes the "worse" tool is right: team fluency, existing test
  investment, protocol reach, and organizational reality are legitimate,
  decisive inputs.
****

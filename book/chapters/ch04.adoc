== Objectives, Baselines, and Success Criteria

A few years ago I sat in a readiness review for a retail platform two weeks
from its biggest sales day of the year. The performance team presented a
fourteen-page PDF: throughput graphs, latency graphs, a heat map, all of it
rendered in reassuring blues and greens. The room nodded. Someone said "looks
fine." The meeting ended eleven minutes early.

Nobody in that room — including the engineers who ran the test — could have
answered the only question that mattered: _did it pass?_ There was no
criterion to pass. The test had produced numbers, the numbers had been
formatted attractively, and the humans had pattern-matched "graphs without red
on them" to "we're ready." Six weeks later, when a dependency upgrade made the
checkout call 40 percent slower, the same test ran in the same pipeline,
produced a slightly different PDF, and nobody noticed. Why would they? The job
exited zero. It always exited zero.

A test with no pass/fail criterion is a science-fair project. It might be
interesting. It might even be beautiful. But it cannot protect anything,
because protection requires the possibility of failure, and failure requires a
line that a machine — not a tired human skimming a PDF — can check. This
chapter is about drawing that line: deriving it from what users actually need,
expressing it in numbers you can defend, anchoring it against a baseline so
regressions surface, and wiring it so that when the line is crossed, something
_stops_.

=== From "fast enough" to a number

Ask a product manager how fast the checkout needs to be and you'll get "fast."
Push, and you'll get "as fast as possible," which is worse, because it sounds
like a requirement and is actually a wish with no upper bound on spend. Your
job is to run a derivation — from user experience down to a number a load test
can check — and to write the derivation down, because the number will be
challenged and "I made it up in a meeting" is not a defense.

The derivation has three steps.

*Step 1: Start from perception, not from the code.* <<ch01>> gave you the
thresholds: under ~100ms feels instantaneous, ~1 second keeps the user's flow
of thought, ~10 seconds loses them entirely. Pick the experience the product
needs. For a checkout confirmation, "flow of thought" is the honest target —
users will tolerate a visible beat, but not a pause long enough to wonder
whether their card was charged. That gives you roughly one second, _end to
end, as the user experiences it_.

*Step 2: Decompose the budget.* The user's one second is not your API's one
second. Subtract what you don't control or must spend anyway: DNS and TLS on a
cold connection (call it 100–150ms on a decent mobile network), network
round-trip (50–100ms), client-side render (150–300ms depending on how much
your frontend does after the response lands). What's left — perhaps 400–500ms
— is the budget for the server side. If the page fans out to multiple backend
calls, the budget divides again, and not evenly: the sequential calls on the
critical path share it, the parallel ones each get almost all of it. This is
latency budgeting, and it's the same discipline as a financial budget: the
totals must add up, and someone has to own each line.

*Step 3: Attach a percentile and a population.* "The checkout API responds in
400ms" is still not checkable, because it doesn't say _which_ requests. All of
them? On average? The finished form is a *service level objective*, and it has
a rigid grammar: __N__ percent of __which requests__, measured __where__,
complete __successfully__ within __bound__, over __window__. For example:

[quote]
____
99% of `POST /checkout` requests, measured at the load balancer, return a
2xx within 400ms, over a rolling 28 days.
____

Every clause is load-bearing. "Measured at the load balancer" versus "measured
inside the service" can differ by tens of milliseconds of queueing — precisely
the queueing that load testing exists to find, so measure on the user's side
of the queue. "Return a 2xx" matters because an instant 500 is very fast and
entirely useless; latency SLIs that don't require success reward crashing
quickly. And the window turns a metric into a contract: it defines how much
history a violation is judged against, which — as we'll see shortly — is where
your error budget comes from.

[TIP]
====
Write the derivation into the test file as comments, next to the threshold it
produced. Eighteen months from now, someone will want to loosen `p99 < 400ms`
to make a red build green. "This number is the user's one-second budget minus
measured network and render costs, see the derivation" wins that argument.
An unexplained number loses it.
====

=== Choosing the percentile

The SLO grammar demands a percentile, and the choice is not a style
preference. Two forces determine it, and they usually push in the same
direction: upward.

The first force is *fan-out arithmetic*, which you met in <<ch01>>. If a page
makes 40 backend calls and you want 95 percent of _page loads_ to stay inside
budget, each _call_ needs to hit the budget with probability
`0.95^(1/40) ≈ 0.9987` — you need roughly p99.9 per call to deliver p95 per
page. The more your architecture fans out, the deeper into the tail your
per-service objectives must live. This is why platform teams at large
microservice shops obsess over p99.9 while a team running a monolith can often
live happily at p95: it's not machismo, it's multiplication.

The second force pushes back, and it's brutally practical: *a percentile is
only as trustworthy as the number of samples beyond it.* In a run of 5,000
requests, p99.9 is determined by the five slowest samples — five data points,
at the mercy of one GC pause, one connection reset, one noisy neighbor on the
CI runner. Gate a pipeline on that and you've built a coin flip. A workable
rule of thumb: don't gate on a percentile unless the run puts at least a few
hundred samples past it. That means p99 wants 30,000+ requests; p99.9 wants
300,000+. At 100 requests per second, that's five minutes of steady state for
a p99 gate and nearly an hour for p99.9. The percentile you choose sets the
duration of the test that can honestly check it — most teams get this
backwards, choosing a ten-minute test and then decorating it with a p99.9
threshold it cannot support.

(And gate on `max` almost never. The maximum of any sufficiently long run is
an outlier by definition — a TCP retransmit, a scheduler hiccup — and a gate
that fails on events you can't control trains everyone to ignore it. <<ch12>>
covers when the extreme tail _is_ the story; in a CI gate, it's noise.)

=== Choosing the window

The window clause does different work in production and in a test, and
conflating them causes a subtle mistake.

In production, the window (typically a rolling 28 or 30 days) defines the
SLO's memory: how long a bad hour haunts you. In a load test, you can't verify
a 28-day objective — a 30-minute test is a *sample* of the SLI, an estimate of
whether the system, under this load, produces latency and error distributions
compatible with the SLO. That's the honest claim, and it's plenty: a system
that can't hold `slo(99%) < 400ms` for thirty minutes at expected peak
certainly won't hold it for a month.

The practical windowing decision inside the test is simpler: *measure the
steady state, not the ramp.* The first seconds of a run are polluted by cold
caches, JIT warmup, connection-pool fill, and autoscaler lag. Ramp to target,
give the system time to settle, and evaluate your thresholds against the held
plateau. <<ch03>>'s load shapes exist partly for this reason — the shape
creates a window worth measuring.

=== Three gates or none

Here is the most common failure of otherwise well-built success criteria: the
team writes one gate, the system fails around it, and the gate stays green.

A *latency-only* gate is the classic. The service starts shedding load under
pressure — returning 503s in two milliseconds. Latency _improves_. Nothing
returns errors faster than a server that has given up. Your p99 threshold
passes while half your users see an error page.

An *error-only* gate has the mirror problem: everything returns 200,
eventually. Fourteen seconds is still a 200.

And both together can pass while *throughput* quietly collapses — particularly
in a closed-model test, where slow responses mean each virtual user simply
issues fewer requests, so the system faces less load exactly when it's
struggling (<<ch03>>'s open-versus-closed distinction, and <<ch12>>'s
coordinated omission, are both lurking here). In an open model the symptom is
different but visible: the generator can't start iterations on schedule
because every worker is stuck waiting.

So the rule is: *latency, error rate, and throughput, gated together, or you
have no gate at all.* Each one covers the escape routes of the other two. In
loadr, a complete set looks like this — note the `slo()` syntax, which reads
the way the reliability document is written, and the tag selector that holds
one endpoint to a stricter standard:

[source,yaml]
----
scenarios:
  storefront:
    executor: constant-arrival-rate   # open model: load doesn't back off
    rate: 100
    time_unit: 1s
    duration: 10m
    pre_allocated_vus: 50
    max_vus: 200
    flow:
      - request:
          name: browse
          url: /products
          checks:
            - { type: status, equals: 200 }
      - request:
          name: checkout
          method: POST
          url: /checkout
          body:
            json: { sku: "A-1", qty: 1 }
          checks:
            - { type: status, equals: 200 }

thresholds:
  # Latency: the service SLO, stated as an SLO.
  http_req_duration:
    - "slo(99%) < 400ms"        # == p(99) < 400
    - "slo(99.9%) < 1s"
  # The revenue path is held to a tighter budget than the browse path.
  "http_req_duration{name:checkout}":
    - "slo(95%) < 250ms"
  # Errors: 99.9% availability, as a rate bound.
  http_req_failed: [ "rate<0.001" ]
  # Throughput: the open model must actually deliver its arrival rate.
  dropped_iterations: [ "count<10" ]
----

The `dropped_iterations` gate deserves a highlight because almost nobody
writes it. In an open model, when the system slows down so much that all
allocated VUs are busy, the generator drops the iterations it can't start.
That drop *is* the throughput collapse — the moment demand exceeded what the
system could absorb — and gating on it means saturation fails the run instead
of merely dimpling a graph.

The same three-legged stool in *k6*, whose threshold syntax loadr deliberately
stays compatible with:

[source,javascript]
----
export const options = {
  scenarios: {
    storefront: {
      executor: 'constant-arrival-rate',
      rate: 100, timeUnit: '1s', duration: '10m',
      preAllocatedVUs: 50, maxVUs: 200,
    },
  },
  thresholds: {
    http_req_duration: ['p(99)<400', 'p(99.9)<1000'],
    'http_req_duration{name:checkout}': ['p(95)<250'],
    http_req_failed: ['rate<0.001'],
    dropped_iterations: ['count<10'],
  },
};
----

And in *Gatling*, as assertions on the simulation:

[source,scala]
----
setUp(storefront.inject(constantUsersPerSec(100).during(10.minutes)))
  .assertions(
    global.responseTime.percentile(99).lt(400),
    details("checkout").responseTime.percentile(95).lt(250),
    global.failedRequests.percent.lt(0.1)
  )
----

Same contract, three spellings. The syntax is the least of it — the portable
idea is that no run is allowed to end without a machine ruling on all three
axes.

=== Defensible baselines

Absolute thresholds catch catastrophe. They do not catch erosion. A checkout
that was 180ms at p95 last quarter and is 240ms today still sails under a
400ms gate — you've lost a third of your headroom and every dashboard is
green. The instrument for erosion is a *baseline*: a recorded run that
subsequent runs are diffed against, so the question shifts from "is it under
the line?" to "is it worse than it was?"

A baseline is defensible when you can answer four questions about it:

* *Same plan, same shape?* Comparing a 10-VU run against a 100-VU run tells
  you nothing; the diff is only meaningful when scenario, load shape, and
  duration are identical. Version the test plan with the code so this is true
  by construction.
* *Same environment?* A baseline captured on staging-large compared against a
  run on staging-small measures your infrastructure catalog, not your code.
  <<ch05>> is about making environments comparable; the baseline inherits all
  of that.
* *Produced by a machine?* A baseline someone hand-picked ("that run looked
  good") encodes their optimism. Generate it from a scheduled job — nightly on
  `main` is the common pattern — so it's boringly reproducible.
* *Refreshed deliberately?* A baseline that silently updates to "whatever ran
  last" ratchets in every regression the moment it lands. A baseline frozen
  forever drifts out of relevance. The workable policy: baselines advance
  automatically when the run passes all gates _and_ someone can see the trend
  (<<ch15>> shows the CI wiring; <<ch18>> the habit).

Mechanically, this is a diff of two summary exports. In loadr:

[source,console]
----
$ loadr run perf/checkout.yaml --summary-export baseline.json   # on main
$ loadr run perf/checkout.yaml --summary-export current.json    # on the PR
$ loadr compare baseline.json current.json
----

[source,console]
----
  metric             field       baseline  current   delta     delta %
  -----------------  ----------  --------  --------  --------  --------  -
  http_req_duration  p50         41.00ms   42.30ms   +1.30ms   ▲ 3.2%
  http_req_duration  p95         98.40ms   131.20ms  +32.80ms  ▲ 33.3%   ✗
  http_req_duration  p99         180.00ms  184.10ms  +4.10ms   ▲ 2.3%    ✓
  http_req_failed    error_rate  0.20%     0.21%     +0.01%    ▲ 5.0%    ✓
  checks             pass_rate   99.80%    99.75%    -0.05%    ▼ 0.1%    ✓

✗ 1 regression(s) beyond tolerance
----

Notice the comparison is *direction-aware* — an improvement is never flagged,
only movement in the worse direction — and *tolerance-gated*. That tolerance
is where teams stumble. Two identical runs against an identical build are not
identical: network jitter, CPU contention on the runner, cache luck. Before
you pick a tolerance, measure your noise floor — run the same plan five times
against the same build and look at the spread of p95. If your run-to-run
variation is 8 percent, a 5 percent tolerance is an alarm that cries wolf, and
within a month everyone will have learned to re-run the job until it passes,
which is the same as having no gate. Set the tolerance above the noise floor,
and if the noise floor is too high to catch the regressions you care about,
fix the environment (<<ch05>>) or lengthen the run — don't widen the gate
until it's decorative.

[source,console]
----
$ loadr compare baseline.json current.json --assert \
    --max-regression p95=10% \
    --max-regression error_rate=0.5 \
    --max-regression rps=5%
----

Percentage limits are relative to the baseline; bare numbers are absolute in
the field's unit (milliseconds for latency, percentage points for rates). And
`--assert` turns a regression beyond tolerance into a non-zero exit — the same
failure contract as `loadr run` itself, which is what lets a pipeline treat
"slower than yesterday" exactly like "slower than the SLO."

=== Error budgets are testing budgets

An SLO of 99.9 percent availability is also, read from the other side, a
*budget*: 0.1 percent of requests are allowed to fail. Over a month, that's
about 43 minutes of full outage, or a steady trickle of one failure per
thousand requests. The SRE literature treats this budget as a spending
account for risk — deploys, experiments, migrations all draw on it — and the
same frame does two useful jobs in testing.

First, it tells you *what error rate your gates should allow*. A load-test
gate of `rate<0.001` isn't an arbitrary bar; it's the availability SLO applied
under load. Teams instinctively write "zero errors" gates, and for a
functional smoke pass they're right — one iteration over a case table with
`checks: ["rate>=1.0"]` is a contract test, and any failure is a bug. But a
zero-error gate on a ten-minute run at 100 RPS demands _better_ than
five-nines from a system you've only promised three-nines for, and it will
fail on the background radiation of real networks. Match the gate to the
promise: functional tests gate at 100 percent; load tests gate at the SLO.

Second, the budget frame sets *testing priorities*. A team burning its
production error budget every month has no business adding chaos experiments
to CI — the budget is spoken for. A team with months of unspent budget can
afford aggressive testing in and against production (<<ch16>>), because the
budget is exactly the license to take that risk. The budget converts "should
we run this scary test?" from a debate about courage into a query against a
number.

=== Cost and resource budgets

One more axis, chronically ignored because latency gates hide it: *what did
meeting the SLO cost?* A service that holds p99 under 400ms by running on
sixteen oversized instances has passed your gate and failed your CFO. The
criteria that catch this are resource-shaped: requests per second per core,
memory per instance at target load, and — once you can price your
infrastructure — cost per million requests. <<ch17>> turns these into
capacity models; here the point is only that they belong _in the success
criteria_, because "meets the SLO at a cost we can afford" is the actual
requirement and "meets the SLO" is the abridged version that gets
misquoted.

Resource metrics also guard the test's own validity. The most common way a
load test lies is that the load *generator* saturates — its CPU pegs, its
timers drift, and it reports the target as slow when the truth is that the
observer blinked. loadr can sample the generator host and make its health a
threshold, so a saturated generator fails the run instead of silently skewing
it:

[source,yaml]
----
observe:
  - type: system
    metrics: [ cpu, memory ]
    interval: 1s

thresholds:
  # If the generator itself is unhealthy, no other number means anything.
  system_cpu: [ "max < 0.90" ]
  system_memory: [ "max < 0.85" ]
----

The same pattern extends to the system under test through your observability
stack (<<ch11>>): scrape CPU and memory during the run, and treat "held the
SLO but only at 97 percent CPU" as the near-miss it is — that system has no
headroom, and headroom is what absorbs the traffic spike you didn't model.

=== Criteria that fail loudly

Everything above is wasted if failure arrives as a PDF. The readiness review
from the top of this chapter had criteria, in a sense — they lived in the
heads of the people squinting at graphs. Criteria protect you only when
they're wired into a machine that can say _no_ and make it stick. Four
properties make that real.

*Failure is an exit code, not a color.* Load tools converge on the same
contract: `0` for success, a distinct code for threshold failure (loadr and
k6 both use `99`), something else for "the test itself broke." That
distinction matters — a threshold failure means "the system is too slow,"
exit `1` means "the test never validly ran," and a pipeline should treat them
differently: the first blocks the merge, the second pages whoever owns the
test. A CI job that runs the test and then exits zero regardless — the
"report-only" pattern, usually installed as a temporary measure — is the
science-fair project with extra steps. Temporary report-only jobs are how
that retail platform shipped a 40 percent regression.

*Failure arrives where decisions happen.* An exit code fails the job; a JUnit
export (`--junit junit.xml`) puts the failed thresholds in the PR's test
panel next to the unit tests; `loadr compare --markdown compare.md` produces
a diff table ready to paste as a PR comment, regressed cells bolded. The
reviewer should see "p95 +33.3%, regression" in the same pane where they'd
see a failing assertion — performance results that live in a separate
dashboard are results the reviewer never sees. <<ch15>> assembles the full
pipeline.

*Runaway tests stop themselves.* A gate evaluated only at the end of the run
means a system that collapses in minute one gets pounded for the remaining
twenty-nine — bad for a staging environment, potentially catastrophic for a
test in production. An abort threshold is the circuit breaker:

[source,yaml]
----
thresholds:
  http_req_failed:
    - { threshold: "rate<0.05", abort_on_fail: true, delay_abort_eval: 30s }
----

The `delay_abort_eval` grace period matters: without it, the two requests
that fail during connection-pool warmup can kill the run before it starts
telling the truth.

*Criteria live with the code.* Thresholds in a YAML file in the repo are
reviewed, versioned, and blamed like any other code. When someone loosens a
gate, that's a diff in a PR with a name attached — which is exactly the
friction you want on the act of lowering a standard. Criteria configured by
hand in a testing dashboard are criteria that will be quietly edited the
week before a deadline, and no one will remember by whom.

None of this is sophisticated. That's rather the point. The distance between
a science-fair project and a performance _test_ is not better graphs or a
smarter tool — it's a handful of numbers, honestly derived, checked by a
machine, compared against yesterday, and allowed to say no.

.Key takeaways
****
* Derive thresholds, don't declare them: user perception budget, minus
  network and render costs, decomposed across the critical path, with the
  derivation written down next to the number.
* A checkable objective has a grammar — _N% of which requests, measured
  where, succeed within bound, over window_ — and every clause is
  load-bearing.
* Fan-out pushes your percentile up; sample counts push back. Don't gate on
  a percentile the run can't populate — the percentile you choose dictates
  the test's minimum duration.
* Gate latency, error rate, and throughput *together*. Each one alone has an
  escape route: fast errors, slow successes, or a quietly collapsing request
  rate.
* Absolute gates catch catastrophe; baselines catch erosion. A defensible
  baseline is same-plan, same-environment, machine-produced, and refreshed
  deliberately — with tolerances set above your measured run-to-run noise.
* An error budget is a testing budget: gate load tests at the SLO's error
  rate (not zero), and let unspent budget license riskier testing.
* "Meets the SLO" is the abridged requirement; the full version includes
  what it cost — and whether the load generator itself stayed healthy enough
  for the numbers to mean anything.
* Failure must be an exit code that blocks a pipeline, delivered where
  decisions happen, with criteria versioned in the repo. Report-only jobs
  are science-fair projects with extra steps.
****

[.fieldcard]
.Field Card — Gates That Say No
****
A test with no pass/fail line is a science-fair project. Reach for these to draw the line and make a machine enforce it.

Writing the objective:: _N% of which requests, measured where, succeed within bound, over window_ — every clause load-bearing. Write the derivation next to the number.
Percentile and samples:: fan-out pushes it up (p95/page ≈ p99.9/call at 40 calls); sample count pushes back — p99 wants 30k+ requests, p99.9 wants 300k+. Never gate on `max`.
Gate together:: latency, error rate AND throughput, or you have no gate — each covers the others' escape route (fast 503s, slow 200s, collapsing rps). In YAML: `slo(99%) < 400ms` and `slo(99.9%) < 1s`, per-endpoint `http_req_duration{name:checkout}: slo(95%) < 250ms`, `http_req_failed: rate<0.001` (the SLO, not zero), `dropped_iterations: count<10`.
Catch erosion:: absolute gates miss slow drift; diff against a same-plan, same-env, machine-made baseline with tolerance set ABOVE your measured noise floor — run the plan 5× first.

[source,console]
----
loadr run perf/checkout.yaml --summary-export current.json --junit junit.xml
loadr compare baseline.json current.json --assert \
  --max-regression p95=10% --max-regression error_rate=0.5 --max-regression rps=5%
----

*Remember:* threshold failure exits `99`, a broken test exits `1` — page different people. `%` limits are relative, bare numbers absolute (ms / points). `abort_on_fail` needs `delay_abort_eval` so warmup blips don't kill the run.

== Generating Load at Scale

_When one machine isn't enough, and how to make sure the machine isn't the thing you're actually measuring._

A team I worked with spent eight days optimizing an API that wasn't slow.

Their load test reported a p99 of 2.1 seconds at 5,000 requests per second, well
outside their 500ms objective. So they did what good engineers do: profiled the
service, tuned the connection pool, added a cache, shaved allocations. The p99
barely moved. On day eight, someone ran the exact same scenario from a second,
idle machine at a tenth of the rate — and measured a p99 of 180 milliseconds.
The API had been fine the whole time. The load generator was a 4-core VM running
at 100 percent CPU with 30,000 sockets stuck in `TIME_WAIT`, and every number it
had produced for a week was a measurement of its own suffering.

This is the most expensive failure mode in load testing that nobody warns you
about: *the generator becomes the system under test*. It doesn't fail loudly. It
doesn't error out. It just quietly adds its own queueing, its own scheduling
delays, and its own resource exhaustion to every latency sample, and hands you a
report that blames the server. This chapter is about generating load at scales
where that can't happen — first by making one machine honest, then by
coordinating many, and finally by merging what they measure without corrupting
it in the merge.

=== The generator is part of the experiment

Every measurement apparatus perturbs what it measures, and a load generator is a
measurement apparatus under more stress than the thing it's measuring. It has to
open the connections, drive the TLS handshakes, serialize the requests, parse
the responses, run the checks, and record the timings — for *every* virtual
user, on one box, while the target spreads the same work across a fleet.

The generator lies to you through a handful of specific mechanisms. Learn to
recognize each one.

*CPU saturation.* When the generator's cores are pegged, requests that are
"in flight" are often actually sitting in a run queue waiting for the event loop
to get scheduled. The recorded latency includes that wait. The rule of thumb
that has served me for years: *don't trust any number produced by a generator
running above 70 percent CPU.* Past that, scheduling delay starts leaking into
your latency samples, and the leak grows non-linearly. Measure the generator's
CPU during every serious run — it's one `top` away, and it's the cheapest
insurance in this book.

*Ephemeral port exhaustion.* Every outbound TCP connection consumes a local
port from the ephemeral range. Linux's default range is 32768–60999 — about
28,000 ports — and a closed connection holds its port in `TIME_WAIT` for 60
seconds. Do the division: without connection reuse, one machine can sustain
roughly *470 new connections per second* to a single destination before the
port pool runs dry. Hit the wall and you'll see `EADDRNOTAVAIL` errors, connect
timeouts, and — nastier — latency spikes as the kernel hunts for a free port.
Teams routinely misread this as the server refusing connections.

*File descriptor limits.* The default `ulimit -n` on many distributions is a
prehistoric 1024. A test with 2,000 concurrent connections will start failing at
descriptor 1025 with errors that, again, look superficially like a server
problem.

*Garbage collection.* JVM-based generators (JMeter, Gatling on default
settings) stop the world when they collect. A 200ms GC pause does two bad
things at once: it delays the *sending* of scheduled requests — which is
coordinated omission by another name, the statistical trap <<ch02>> introduced
and <<ch12>> dissects — and it inflates the recorded latency of every request
that happened to be in flight when the collector fired. Both distortions blame
the server for the generator's memory management.

One diagnostic settles the question definitively; run it whenever a result
matters:

[TIP]
====
*The halving test.* Split the same total load across two generators instead of
one and re-run. If the latency numbers improve, the generator was part of your
measurement, and every prior result from the single machine is suspect. If they
don't move, the generator was clean. This costs one extra VM and thirty
minutes, and it has invalidated more "server-side regressions" than any
profiler I've used.
====

Open-model tools give you a second tripwire for free. Because an arrival-rate
executor must start iterations *on schedule* regardless of how the target
responds (<<ch03>>), it knows when it can't keep up — it has to drop
iterations. Make that a failing condition:

[source,yaml]
----
defaults:
  http:
    base_url: https://api.example.com

scenarios:
  steady_api_load:
    executor: constant-arrival-rate
    rate: 100
    time_unit: 1s
    duration: 10m
    pre_allocated_vus: 50
    max_vus: 200          # the pool the executor may grow into
    flow:
      - request:
          name: health
          url: /health

thresholds:
  http_req_duration: [ "p(99)<300" ]
  dropped_iterations: [ "count<10" ]   # generator fell behind => fail the run
----

A rising `dropped_iterations` count means the generator exhausted its VU pool
trying to hold the rate — either the target is so slow that in-flight requests
pile up (interesting!) or the generator itself is out of capacity (a broken
experiment). Either way you want the run to fail loudly rather than hand you a
polluted p99. k6 emits the same metric under the same name; watch it there too.

=== Vertical first: make one machine honest

Distribution is a tax — more moving parts, more coordination, more ways to
aggregate wrong. Before you pay it, extract everything a single machine has to
give. A tuned box running an efficient engine (loadr, k6, Gatling) will sustain
tens of thousands of requests per second; most teams reach for a fleet long
before they've earned it.

The tuning that matters, in descending order of payoff:

*Reuse connections.* HTTP keep-alive is the single biggest lever. A reused
connection skips the TCP handshake, the TLS handshake, and the ephemeral port
consumption entirely — turning the 470-connections-per-second wall into a
non-issue. Every serious tool reuses connections by default; the design
question is whether reuse matches reality. If your real traffic is a million
distinct mobile clients each making two requests, connection reuse makes the
test *unrealistically cheap* for the server, which now sees a fraction of the
handshakes it would in production. That's a workload-modeling decision
(<<ch03>>), not a tuning one — but make it consciously.

*Raise the kernel limits.* Two minutes of setup on any Linux generator:

[source,console]
----
$ ulimit -n 65536                                        # file descriptors
$ sudo sysctl -w net.ipv4.ip_local_port_range="1024 65535"   # ~64k ports
$ sudo sysctl -w net.ipv4.tcp_tw_reuse=1                 # reuse TIME_WAIT ports
----

Bake these into the machine image or the container entrypoint so nobody
rediscovers them at 2 a.m.

*Stop writing per-request logs to disk.* Recording every request's full detail
to a local file turns your load test into a disk benchmark. Stream aggregated
metrics instead; keep full request logging for functional debugging at low
rates.

*Warm up.* JITs compile, connection pools fill, DNS caches populate. The first
thirty seconds of any run are measuring a cold generator as much as a cold
server. Ramp in, and exclude the warm-up window from the analysis — <<ch08>>
covered the mechanics.

*Never generate load from a GUI.* JMeter's own documentation says GUI mode is
for test creation only, and it's right: the Swing interface can cost you an
order of magnitude in throughput. `jmeter -n` or nothing.

Once the machine is tuned, *find its ceiling on purpose*. Point it at a
deliberately overprovisioned target (or a stub that answers instantly) and step
the rate up until `dropped_iterations` appears or CPU crosses 70 percent. A
parameter sweep does this in one command:

[source,console]
----
$ loadr sweep ceiling.yaml --var rate=5000,10000,20000,40000 --duration 60s
----

(`vus` and `duration` map straight onto run overrides; any other swept axis,
like `rate` here, reaches the plan as an environment variable —
`ceiling.yaml` reads its `rate` from `LOADR_SWEEP_RATE`.)

The resulting table tells you, say, that this instance type is clean up to
22,000 req/s. Write that number down. It converts "how many agents do I need?"
from a guess into arithmetic — and it's the input the cost section at the end
of this chapter depends on.

=== When one machine isn't enough

Suppose the target is 600 requests per second of heavyweight,
TLS-terminated, JSON-parsing traffic, held for fifteen minutes, and your
per-node ceiling math says one box would run hot. Every mature tool converges
on the same architecture for this: a *control plane* that partitions the plan
and aggregates results, and *agents* that are deliberately dumb muscle. In
loadr the control plane is a `controller` process; agents dial in over gRPC,
receive their slice of the plan plus any data files it references, start on a
synchronized barrier, and stream metric deltas back every second.

The scenario file doesn't know it will be distributed. This one asks for 600
req/s *in total*; the controller divides the rate across however many agents
have joined:

[source,yaml]
----
# distributed.yaml — 600 req/s split across the whole agent fleet
name: distributed
description: 600 req/s split across the agent fleet

defaults:
  http:
    base_url: https://api.example.com

scenarios:
  fleet_load:
    executor: constant-arrival-rate
    rate: 600                 # total across ALL agents
    duration: 15m
    pre_allocated_vus: 300
    max_vus: 900
    flow:
      - request:
          name: search
          url: "/search?q=stress"
          checks: [ { type: status, equals: 200 } ]

thresholds:
  http_req_duration: [ "p(95)<400", "p(99)<900" ]
  http_req_failed: [ "rate<0.01" ]
----

Running it is three commands — one on the controller host, one on each
generator, one from wherever you submit tests:

[source,console]
----
$ loadr controller --bind 0.0.0.0:7625 --ui-bind 0.0.0.0:6464   # control plane
$ loadr agent --join controller-host:7625 --name agent-$(hostname)  # per generator
$ loadr run --controller controller-host:6464 distributed.yaml  # submit
----

With three agents joined, each runs 200 req/s. Add a fourth and resubmit; each
runs 150. VU-based executors split their VU counts the same way, and stage
*timings* stay identical on every agent — only magnitudes divide — so a global
ramp is exact rather than approximately staggered. Two operational details
generalize to every tool's distributed mode:

* *Synchronized start.* Agents start on a shared barrier, not "whenever the
  message arrives." Without it, a 30-second ramp across ten agents smears into
  ten slightly offset ramps and the aggregate load shape isn't the one you
  designed.
* *Agent-loss policy.* Machines die mid-run. You must decide in advance whether
  the run continues with the surviving agents carrying their original share
  (total load quietly drops — fine for a soak, misleading for a capacity test)
  or aborts everywhere. Defaulting to "continue and note it in the summary" is
  reasonable; not knowing which behavior you'll get is not.

The same shape exists everywhere, with different trade-offs. *Locust* has the
cleanest equivalent — a master process and workers that connect to it:

[source,console]
----
$ locust -f loadtest.py --master                          # control plane
$ locust -f loadtest.py --worker --master-host=10.0.0.5   # on each generator
----

*JMeter* ships remote testing (one controller driving worker JVMs over RMI),
but it predates modern expectations: every worker funnels sample results back
to the controller, which becomes its own bottleneck at high rates unless you
aggregate downstream. *k6*'s open-source binary has no native multi-node mode;
the supported paths are the k6-operator on Kubernetes or Grafana Cloud k6.
*Gatling*'s distributed story lives in its Enterprise product. None of this is
a reason to switch tools — it's a reason to know, before the week you need
50,000 req/s, which path your tool expects you to take.

=== Load from cloud and Kubernetes

Agents are stateless by design — they hold no results, no configuration, no
identity beyond a name — which makes them a perfect fit for the most
disposable compute you can buy. On Kubernetes the whole fleet is three
manifests: a controller pod, an agent `Deployment` you scale like any other,
and a `Job` that submits the plan:

[source,console]
----
$ kubectl apply -f controller.yaml
$ kubectl apply -f agents.yaml
$ kubectl scale deployment/loadr-agents --replicas=10   # size the fleet
$ kubectl apply -f run-job.yaml                         # kick off the run
$ kubectl logs -f job/loadr-run                         # merged summary
----

The Job's exit code reflects the thresholds — zero on pass, non-zero on breach
— so the same manifest doubles as an in-cluster CI gate, a thread we pick up
properly in <<ch14>>. Locust's Helm chart and the k6-operator's `TestRun`
custom resource give you the same pattern with the same economics.

Kubernetes also introduces two generator-specific traps:

[WARNING]
====
*CPU limits throttle your generator invisibly.* Kubernetes CFS throttling
enforces CPU limits by *pausing* the container for the rest of a 100ms
accounting period once it exceeds its quota. A throttled agent is a generator
that periodically freezes — delaying scheduled requests (coordinated omission
again) and inflating in-flight latencies, exactly like a GC pause. Give agent
pods generous or no CPU limits, use guaranteed QoS, and watch the
`container_cpu_cfs_throttled_periods_total` metric during runs.

*Don't co-locate the generator with the target.* Agents scheduled onto the same
nodes as the system under test compete with it for CPU and network, and you end
up measuring the fight. Use node selectors or a separate node pool — ideally a
separate cluster.
====

=== Geographic distribution

Your users are not in `us-east-1`. A service that shows a 40ms p95 from a
generator in the same region can be a 190ms p95 from Sydney, and if your SLO is
written from the user's point of view (<<ch04>>), the Sydney number is the one
that counts. Geographic distribution is also the only way to exercise the parts
of the stack that only exist geographically: CDN edge behavior, GeoDNS,
anycast routing, regional failover.

Mechanically it's the same fleet — agents just live in different places and
carry a label saying so:

[source,console]
----
$ loadr agent --join ctrl.example.com:7625 --name eu-west-1a
$ loadr agent --join ctrl.example.com:7625 --name ap-southeast-2a
----

Every sample an agent emits is tagged with its instance name, which means you
can hold different regions to different objectives in one run:

[source,yaml]
----
thresholds:
  "http_req_duration{instance:eu-west-1a}":      [ "p(95)<120" ]
  "http_req_duration{instance:ap-southeast-2a}": [ "p(95)<350" ]
  http_req_failed:                               [ "rate<0.01" ]
----

One number you should refuse to report from a geo-distributed run: the *global*
percentile, on its own. Merge a 30ms European mode with a 180ms Australian mode
and the combined distribution is bimodal — its p95 lands somewhere between the
two humps and describes the experience of *no actual user in either region*.
The merged data isn't wrong (the merge machinery below keeps it exact); it's
the summary that's meaningless. Report per region, always.

One more decision belongs to test design rather than execution: crossing the
public internet puts the internet *inside your experiment* — its jitter, its
peering moods, its bad evenings. That's the point when you're validating the
user experience end to end, and pure noise when you're hunting a service-side
regression. Match the generator's location to the question, the same way
<<ch05>> matched the environment to it.

=== Merging results without lying

Here is where most homegrown distributed setups quietly fall apart. Ten agents
each produce a summary; someone writes a script that averages the ten p99s into
a "fleet p99." *That number is fiction.* Percentiles are not averageable — not
weighted, not any way. If agent A saw a p99 of 100ms and agent B saw a p99 of
1,000ms, the fleet's true p99 is not 550ms; it depends on the full shape of
both distributions, which the two summary numbers have already thrown away.

Make it concrete: two agents, same request count. Agent A's
latencies are spread over 1–1,000ms; agent B's over 1,001–2,000ms. The true p99
of the combined traffic is about 1,980ms — 99 percent of the way through the
union. Naive averaging of the per-agent p99s reports roughly 1,485ms. That's a
half-second of tail latency, invented by arithmetic, in the direction that
makes you ship something you shouldn't.

The correct construction, which you should demand from any distributed tool or
build into any homegrown one:

. Each agent records latencies into a *mergeable* data structure — HDR
  histograms are the standard (loadr uses them at three significant figures;
  Gatling and modern JMeter backends can too; t-digests are the common
  alternative).
. Agents ship histogram *deltas* to the aggregation point — compact, every
  second, cheap on the wire.
. The aggregator *merges the histograms*, which is lossless: the merged
  histogram is bit-for-bit what a single generator observing all the traffic
  would have recorded.
. Every percentile anyone ever sees — live dashboard, threshold evaluation,
  final report — is computed from the merged histogram only.

Counters and rates are easier: sums merge exactly (add the passes, add the
totals, divide once at the end — don't average the per-agent rates, which
breaks the moment agents handle unequal shares). Gauges keep the latest value
with min/max envelopes.

The aggregation point is also where pass/fail must live. Thresholds evaluated
per agent produce split-brain outcomes — agent 3 aborts the run because *its*
slice breached while the fleet as a whole was healthy. Evaluate centrally
against merged data, then fan stop commands out. loadr disables local threshold
evaluation on agents in distributed runs for exactly this reason; if you're
wiring up JMeter workers or a custom harness, you have to impose the same
discipline yourself. And the trap has a longer reach than load testing: the
moment your *observability* pipeline pre-computes percentiles per host and
averages them on a dashboard, it's telling the same lie — a thread we pull in
<<ch11>>.

[WARNING]
====
If a distributed load-testing setup — commercial, open source, or homegrown —
cannot explain how it combines per-node latency data, assume it averages
percentiles until proven otherwise. Ask the vendor. Read the aggregation code.
This single question separates tools that measure from tools that decorate.
====

=== The bill, and how to keep it small

The pleasant surprise about load generation at scale is that the generators are
the cheap part. Twenty 4-core cloud VMs at roughly $0.17 an hour is $3.40 an
hour of fleet — a fifteen-minute capacity test costs less than the coffee you
drink while watching it. The *expensive* parts are the production-scale
environment under test (<<ch05>>) and the engineer-hours spent on runs that
answer nothing. Cost discipline at scale is mostly about those, but the fleet
side has real levers:

*Size by arithmetic, not anxiety.* You measured the per-node ceiling earlier
with a sweep. Size the fleet so each agent runs at 50–60 percent of that
ceiling: a 100,000 req/s test against a 22,000 req/s ceiling wants nine or ten
agents, not the forty a nervous engineer provisions "to be safe." Headroom
below 70 percent CPU is a correctness requirement; headroom below 50 percent is
budget on fire.

*Exploit statelessness.* Agents hold nothing, so run them on spot/preemptible
instances at a 60–90 percent discount — a reclaimed agent is precisely the
agent-loss case the controller already has a policy for. Scale the fleet to
zero between runs; an idle load-generation cluster is the most embarrassing
line on a platform team's bill.

*Match the fleet to the question.* The everyday CI gate (<<ch14>>) needs one
small generator and five minutes. The full fleet exists for the questions that
need it — the quarterly capacity validation, the pre-launch breakpoint hunt,
the geo test before an international rollout. Teams that run the big fleet
nightly out of ritual get numb to it, and numbness is how a real regression
sails through. The capacity-and-cost decisions those big runs feed into get a
full treatment in <<ch16>>.

The through-line of this chapter is plain: scale is not the
goal. *Valid measurement at the required load* is the goal, and every
technique here — the 70 percent rule, the halving test, the synchronized
barrier, the histogram merge — exists to protect validity while the numbers
get big. A fleet of fifty agents producing an averaged percentile is a very
expensive way to be wrong. One honest machine, and the knowledge of exactly
where its honesty ends, beats it every time.

.Key takeaways
****
* The load generator is part of the experiment. Above ~70 percent CPU — or
  with exhausted ports, descriptors, or a pausing GC — its numbers describe its
  own queueing, not your server's. Monitor the generator like you monitor the
  target.
* The *halving test* (same load, twice the generators) is the definitive
  check: if latency improves, every previous single-node result is suspect.
  Open-model `dropped_iterations` is the always-on tripwire.
* Tune vertically before distributing: connection reuse, raised
  `ulimit`/port-range, no per-request disk logging, no GUI mode. Then measure
  the per-node ceiling on purpose — fleet sizing is arithmetic from that
  number.
* Distributed generation is a control plane plus stateless agents everywhere
  — loadr controller/agents, Locust master/worker, JMeter remote hosts, the
  k6-operator. Demand a synchronized start and an explicit agent-loss policy.
* On Kubernetes, CPU limits throttle agents into lying (CFS pauses look like
  server latency), and co-locating agents with the target measures the fight
  between them. Isolate and unconstrain the fleet.
* *Never average percentiles.* Merge histograms (HDR or similar) and compute
  every percentile from the merged data; evaluate thresholds centrally. Report
  geo-distributed results per region — a global percentile over a bimodal
  distribution describes nobody.
* Generators are cheap; invalid runs are expensive. Spot instances, scale to
  zero between runs, and reserve the big fleet for the big questions.
****

== Tool Quick-Reference and Cheat Sheets

_Side-by-side "how do I do X" recipes for loadr, k6, JMeter, Gatling, and
Locust. <<ch07>> is where the tools are compared and chosen; this appendix is
for the day you already chose and just need the spelling._

Every recipe here is the same eight tasks, once per tool: install and run,
choose a load model, shape a ramp, check a response, gate on an SLO, feed in
test data, correlate values across requests, and get the results out into CI.
If a cell says "plugin" or "not native," that's not a value judgment — it's a
warning about where you'll spend the extra hour.

=== Install and run

[cols="1,3,3",options="header"]
|===
| Tool | Install | Headless run
| loadr | Download the release binary and put it on `PATH` (or `cargo install --git https://github.com/levantar-ai/loadr loadr-cli`) | `loadr run test.yaml`
| k6 | `brew install k6` / OS packages / `docker run grafana/k6` | `k6 run script.js`
| JMeter | Download from apache.org (needs a JRE) or `brew install jmeter` | `jmeter -n -t plan.jmx -l results.jtl`
| Gatling | Maven/Gradle/sbt plugin, or the standalone bundle | `mvn gatling:test` (or `./bin/gatling.sh` from the bundle)
| Locust | `pip install locust` | `locust -f locustfile.py --headless -u 100 -r 10 -t 5m`
|===

TIP: `loadr convert plan.jmx` and `loadr convert script.js` translate existing
JMeter and k6 tests to loadr YAML — useful for migration, and as a Rosetta
stone when you're learning one tool's vocabulary from another's.

=== Open (arrival-rate) vs. closed (VU) model

The single most consequential setting in any tool. <<ch03>> explains why; here
is where each tool hides the switch.

[cols="1,3,3",options="header"]
|===
| Tool | Closed (N concurrent users) | Open (N arrivals/sec)
| loadr | `executor: constant-vus` / `ramping-vus` | `executor: constant-arrival-rate` / `ramping-arrival-rate`
| k6 | `executor: 'constant-vus'` / `'ramping-vus'` | `executor: 'constant-arrival-rate'` / `'ramping-arrival-rate'`
| JMeter | Thread Group (threads + ramp-up) | Open Model Thread Group (5.5+), or Precise Throughput Timer
| Gatling | `constantConcurrentUsers(n)` / `rampConcurrentUsers` | `constantUsersPerSec(n)` / `rampUsersPerSec`
| Locust | Users + spawn rate (the native model) | Not native; approximate with `wait_time = constant_throughput(x)` pacing per user
|===

The open model in loadr, with the saturation tell-tale you should always gate
on (a growing `dropped_iterations` means the target can't absorb the offered
rate — see <<ch12>>):

[source,yaml]
----
scenarios:
  steady_api_load:
    executor: constant-arrival-rate
    rate: 100              # iterations per time_unit
    time_unit: 1s
    duration: 10m
    pre_allocated_vus: 50
    max_vus: 200
    flow:
      - request: { name: health, url: /health }

thresholds:
  dropped_iterations: [ "count<10" ]
----

k6 uses the same executor names and the same `preAllocatedVUs`/`maxVUs`
knobs — the vocabulary was deliberately shared, so this table transfers
one-to-one.

=== Ramps and stages

[cols="1,4",options="header"]
|===
| Tool | Idiom
| loadr | `stages: [ {duration: 2m, target: 50}, {duration: 5m, target: 50}, {duration: 1m, target: 0} ]` on a `ramping-vus` or `ramping-arrival-rate` executor
| k6 | Identical `stages` array on the ramping executors
| JMeter | Thread Group ramp-up (up only); Ultimate Thread Group plugin for real profiles
| Gatling | Chain injection steps: `rampUsersPerSec(0).to(200).during(30), constantUsersPerSec(200).during(120)`
| Locust | Subclass `LoadTestShape` and return `(user_count, spawn_rate)` per tick
|===

A spike profile in loadr — baseline, 10× surge, recovery — with an abort
threshold as a circuit breaker so a dying target doesn't eat an hour of CI:

[source,yaml]
----
scenarios:
  spike:
    executor: ramping-arrival-rate
    start_rate: 20
    time_unit: 1s
    pre_allocated_vus: 100
    max_vus: 500
    stages:
      - { duration: 2m,  target: 20 }    # baseline
      - { duration: 10s, target: 200 }   # spike
      - { duration: 1m,  target: 200 }   # sustain
      - { duration: 10s, target: 20 }    # recover
      - { duration: 2m,  target: 20 }    # verify recovery
    flow:
      - request: { name: checkout, method: POST, url: /checkout,
                   body: { json: { sku: "A-1", qty: 1 } } }

thresholds:
  http_req_failed:
    - { threshold: "rate<0.05", abort_on_fail: true, delay_abort_eval: 30s }
----

=== Checks and assertions (per response)

[cols="1,4",options="header"]
|===
| Tool | Idiom
| loadr | `checks:` on a request: `{ type: status, equals: 200 }`, `{ type: jsonpath, expression: "$.status", equals: pending }`, `{ type: duration, max: 800ms }`. Use `assert:` instead of `checks:` to abort the iteration on failure.
| k6 | `check(res, { 'status 200': (r) => r.status === 200 })` — records a rate, never aborts
| JMeter | Response Assertion, JSON Assertion, Duration Assertion elements
| Gatling | `.check(status.is(200), jsonPath("$.status").is("pending"))`
| Locust | `catch_response=True`, then `resp.failure("...")` / `resp.success()`
|===

WARNING: In loadr and k6, a failed *check* is a recorded measurement, not a
stop sign — the run continues and the failure rate feeds your thresholds.
That's what you want: under load, some failures are data, not exceptions.
Reserve loadr's `assert:` (and Gatling's implicit check-fails-the-user
behavior) for cases where continuing the flow is meaningless, like a login
that returned no token.

=== Thresholds and SLO gates (pass/fail for the whole run)

A test without a machine-checkable gate is a number generator (<<ch04>>).
Where each tool puts the gate:

[cols="1,4",options="header"]
|===
| Tool | Idiom
| loadr | Top-level `thresholds:`; k6 syntax (`"p(99)<300"`) and SLO syntax (`"slo(99%) < 300ms"`) mix freely; per-endpoint via tag selector; exit code 99 on failure
| k6 | `options.thresholds: { http_req_duration: ['p(99)<300'] }`; `abortOnFail`; exit code 99
| JMeter | Not native — gate in CI by parsing the JTL (Taurus and the Jenkins Performance plugin are the usual wrappers)
| Gatling | `setUp(...).assertions(global().responseTime().percentile(99.0).lt(300))` — fails the build via the Maven/Gradle plugin
| Locust | Process exits nonzero on request failures; latency gates via locust-plugins (`--check-avg-response-time`, `--check-fail-ratio`)
|===

The loadr form, written the way the SLO document reads:

[source,yaml]
----
thresholds:
  http_req_duration:
    - "slo(99%) < 300ms"           # == p(99) < 300
    - "slo(99.9%) < 1s"
  "http_req_duration{name:checkout}":
    - "slo(95%) < 250ms"           # stricter gate for one endpoint
  http_req_failed: [ "rate<0.001" ]
----

=== Parameterization and feeders

Test data strategy is <<ch05>>; the mechanics:

[cols="1,4",options="header"]
|===
| Tool | Idiom
| loadr | Top-level `data:` block — `type: csv`/`json`, `mode: shared`/`per_vu`, `pick: shuffle`/`random`, `on_eof: recycle`/`stop`; reference as `${data.users.username}`
| k6 | `new SharedArray('users', () => ...)` + papaparse for CSV; index by VU/iteration yourself
| JMeter | CSV Data Set Config element; `${username}` in samplers
| Gatling | `csv("users.csv").circular()` (or `.random()`, `.shuffle()`); `#{username}` in the DSL
| Locust | Plain Python — read the file at module load, pick per task
|===

=== Correlation (extract from one response, use in the next)

The task that kills recorded scripts (<<ch06>>). Extractor per tool:

[cols="1,4",options="header"]
|===
| Tool | Idiom
| loadr | `extract:` on a request — `jsonpath`, `regex`, `css`, `header`, `boundary`; reuse as `${token}`
| k6 | `res.json('token')`, `res.html().find(...)`, regex on `res.body` — it's JavaScript
| JMeter | JSON Extractor, Regular Expression Extractor, Boundary Extractor, CSS Selector Extractor
| Gatling | `.check(jsonPath("$.token").saveAs("token"))`, then `#{token}`
| Locust | `resp.json()["token"]` — plain Python
|===

In loadr, the login-then-use shape:

[source,yaml]
----
flow:
  - request:
      name: login
      method: POST
      url: /login
      body: { form: { username: "${data.users.username}",
                      password: "${data.users.password}" } }
      extract:
        - { type: jsonpath, name: token, expression: "$.token" }
  - request:
      name: profile
      url: /me
      headers: { Authorization: "Bearer ${token}" }
      checks: [ { type: status, equals: 200 } ]
----

=== Outputs and CI integration

[cols="1,3,3",options="header"]
|===
| Tool | Live/streamed outputs | CI hook
| loadr | `outputs:` list — `prometheus` scrape endpoint, `json` (JSONL), `statsd`, `influxdb`, `otlp`; `--summary-export out.json`; `loadr report` for HTML | Exit 0/99; `loadr compare base.json current.json --assert --max-regression p95=10%` for baseline diffs; a GitHub Action (`setup-loadr`) exists
| k6 | `--out json=`, `--out csv=`, `--out experimental-prometheus-rw`, StatsD, cloud | Exit 0/99 on thresholds; official GitHub Action
| JMeter | JTL files; Backend Listener → InfluxDB/Graphite; `-e -o report/` HTML dashboard | Wrap with Taurus or the Jenkins Performance plugin for pass/fail
| Gatling | `simulation.log` + rich HTML report; Enterprise for live streaming | Assertions fail the Maven/Gradle build; JUnit-style output
| Locust | `--csv prefix`, `--html report.html`, web UI, event hooks | Nonzero exit on failures; locust-plugins checks for latency gates
|===

Exit codes are the contract that makes any of this automatable
(<<ch14>>). loadr and k6 share the convention deliberately: `0` all gates
passed, `99` the run finished but a threshold failed, other codes mean the
test itself broke. Wire your pipeline to distinguish "the system is too slow"
(99 — page the service owner) from "the test is broken" (1 — page whoever owns
the test), because the two failures have different on-call rotations.

Finally, the regression-gate one-liner that <<ch14>> builds a whole pipeline
around — run the same plan against baseline and candidate, then let the diff
fail the build:

[source,console]
----
$ loadr run plan.yaml --summary-export base.json      # baseline build
$ loadr run plan.yaml --summary-export current.json   # candidate build
$ loadr compare base.json current.json --assert \
    --max-regression p95=10% --max-regression error_rate=0.5
----

Whichever column of these tables you live in, the rows are the discipline.
The tool is the spelling; Parts II and IV of this book are the grammar.

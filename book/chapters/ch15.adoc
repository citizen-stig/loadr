== Performance Testing in CI/CD

A payments team I worked with ran a proper load test twice a year, before each
peak season. One spring, the test found checkout p95 had gone from 180ms to
2.1 seconds. Somewhere in the eleven weeks since the last test — somewhere in
roughly nine hundred merged commits — someone had added an innocent ORM
annotation that turned one query into one-per-row. Finding it took three
engineers most of a week: bisecting builds, redeploying candidates to staging,
re-running the test at each step. The fix, once found, was a one-line diff.

Now run the counterfactual. If a thirty-second performance smoke had run on the
pull request that introduced it, the regression would have been caught with a
blast radius of one commit and one author, while the change was still fresh in
their head. Ten minutes, not a week. The cost of a performance regression is
mostly a function of *detection latency* — the number of commits between the
mistake and the moment somebody notices. Everything in this chapter is about
driving that number toward one.

That's the easy part of the argument. The hard part is the engineering tension
underneath it: CI wants tests that are fast and deterministic, and performance
tests are naturally slow and noisy. A real load test takes thirty minutes;
nobody will accept a thirty-minute PR check. A real load test is sensitive to
its environment; CI runners are the noisiest environment in your estate. Teams
that ignore this tension end up in one of two failure modes — a gate so slow
that everyone bypasses it, or a gate so flaky that everyone re-runs it until it
passes, which is the same as not having one. The way out is not one heroic
test. It's a *tiered* set of tests, each sized to its trigger, plus some
unglamorous discipline about baselines, noise, and who gets to loosen a gate.

=== One test can't do it all: the tiers

"Shift left" gets said as if it meant "run the big test earlier." It doesn't,
and can't — the big test is too slow and too expensive to run nine hundred
times between seasons. What shifts left is a *question*, and different
questions afford different budgets. Recall the taxonomy from <<ch01>>: a smoke
test, an SLO-validating load test, and a capacity sweep are different
instruments. A working pipeline runs all three, on different triggers:

[cols="1,1,1,2,1",options="header"]
|===
| Tier | Trigger | Duration | The question | On failure
| *PR smoke gate* | every pull request | 30–90s | Is this change *grossly* worse — 2x latency, errors, a lost index? | block the merge
| *Merge baseline* | every merge to `main` | 3–10m | Produce the canonical baseline; did `main` regress? | alert the merging author
| *Nightly full run* | schedule | 30–60m | Does the system still meet its SLOs at target load? | ticket + owner, block release
| *Capacity sweep* | nightly or weekly schedule | hours | Where is the knee, and has it moved? | trend review, capacity ticket
|===

Each tier buys a more expensive question with a bigger budget, and each
catches what the tier below cannot. The PR gate catches
the accidental N+1 and the synchronous call someone added inside a loop; it
will never catch a 7 percent drift. The nightly run catches the drift and
validates the actual SLO at the actual target rate. The sweep catches the
regression neither of the others can see: the code that's just as fast at 50
VUs but falls over at 150, because someone shrank a connection pool.

[WARNING]
====
The classic mistake is building only the middle tier — a ten-minute test that's
too slow to run on PRs and too shallow to find the capacity ceiling — and then
running it "sometimes." A test that runs on no fixed trigger, gating nothing,
is a science-fair project (<<ch04>>). Pick the triggers first; size the tests
to fit them.
====

=== The PR gate: small, fast, ruthless

The PR gate's job is narrow: fail loudly on order-of-magnitude regressions, in
under two minutes, on every pull request, with near-zero false positives. That
last constraint dictates everything else. Modest load, short duration,
generous thresholds — because a gate that fails one PR in twenty for no reason
will be dead (socially, then literally) within a month.

Here is a complete PR gate plan in loadr — twenty virtual users, thirty
seconds, hitting the two endpoints that matter, with thresholds that *are* the
gate:

[source,yaml]
----
# perf-smoke.yaml — the test the CI job runs on every PR
name: perf-smoke
description: fast PR performance gate

defaults:
  http:
    base_url: "${secrets.target_url}"   # provided by the CI env (TARGET_URL)

secrets:
  target_url: { env: TARGET_URL }

scenarios:
  smoke:
    executor: constant-vus
    vus: 20
    duration: 30s
    flow:
      - request: { name: home,   url: /,           checks: [ { type: status, equals: 200 } ] }
      - request: { name: health, url: /api/health, checks: [ { type: status, equals: 200 } ] }

thresholds:
  http_req_failed:   [ "rate<0.01" ]
  http_req_duration: [ "p(95)<300", "p(99)<800" ]
  checks:            [ "rate>0.99" ]
----

If any threshold is breached, `loadr run` exits with code `99`, the job goes
red, and the merge is blocked. The wiring is one workflow file:

[source,yaml]
----
# .github/workflows/perf-gate.yml
name: perf-gate
on:
  pull_request:

jobs:
  loadr:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: loadr performance gate
        id: perf
        uses: levantar-ai/loadr@v1
        with:
          plan: perf/perf-smoke.yaml
          version: v1.25.0            # pin it — reproducible CI beats 'latest'
          junit: loadr-junit.xml
          summary: loadr-summary.json
          # fail-on-threshold defaults to 'true' → a breached SLO fails the job
        env:
          TARGET_URL: ${{ vars.STAGING_URL }}

      - name: Upload report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: loadr-report
          path: |
            loadr-junit.xml
            loadr-summary.json
----

Two details in that workflow earn their keep. The JUnit export puts each
threshold into the PR's test panel as a named pass/fail, right next to the unit
tests — the reviewer sees "p(95)<300: FAILED" in the pane where they already
look, not in a dashboard they don't. And the summary JSON is uploaded even on
failure (`if: always()`), because the failed runs are precisely the ones you'll
want to inspect and, later, to trend.

The action is convenience sugar; any CI system that can run a binary and read
an exit code works the same way:

[source,console]
----
$ TARGET_URL=https://staging.example.com loadr run perf/perf-smoke.yaml --junit out.xml
$ echo $?
99        # threshold breached → your job goes red, whatever the CI vendor
----

The contract also survives a change of load tool. In k6, the thresholds block
plays the same role and a breach produces the same exit code 99:

[source,javascript]
----
// smoke.js — the k6 spelling of the same gate
export const options = {
  vus: 20,
  duration: '30s',
  thresholds: {
    http_req_failed:   ['rate<0.01'],
    http_req_duration: ['p(95)<300', 'p(99)<800'],
    checks:            ['rate>0.99'],
  },
};
----

Gatling expresses it as assertions on the simulation
(`global.responseTime.percentile(95).lt(300)`,
`global.failedRequests.percent.lt(1)`) and fails the Maven or Gradle build
when they don't hold. Locust's headless mode has `--check-fail-ratio` and
`--check-avg-response-time` flags that make the process exit non-zero on
breach. JMeter has no native gate, which is why JMeter-in-CI almost always
means Taurus, whose `passfail` module wraps the JMX in exactly this kind of
threshold. Whatever the tool: *the gate is an exit code, evaluated by a
machine, on every PR.* Everything else is spelling.

==== What the PR gate can and cannot catch

Be honest about the instrument. Thirty seconds at 20 VUs against a shared
staging environment will reliably catch: an accidental N+1 query, a lost
database index, a synchronous external call added to a hot path, a
serialization bug that quadrupled payload size, an endpoint that started
returning 500s under mild concurrency. These are the regressions that turn a
40ms endpoint into a 900ms one, and they are *common* — in my experience they
are most of the performance regressions that matter.

It will not catch a 10 percent drift, a slow memory leak, or anything that
only appears at 10x this load. That's fine. Those belong to the nightly tiers,
and pretending the PR gate covers them is how teams end up with a ten-minute
PR check that still misses the leak. Set the smoke thresholds *loose* relative
to your SLO — if the SLO is p99 < 300ms at production load, a smoke gate of
p99 < 800ms at toy load is not weakness, it's calibration. You're not
validating the SLO here; you're tripping on catastrophe.

=== Regression detection: comparing builds, not just checking bounds

A fixed threshold has a blind spot: it can't see movement *within* bounds.
p95 goes from 120ms to 260ms — a 2.2x regression — and a `p(95)<300` gate
smiles and waves it through. The complementary instrument is a *diff against a
baseline*: the same plan, run against the previous known-good build, compared
field by field, gating on the delta rather than the absolute value.

The pattern needs two jobs. The merge-baseline tier from the table produces
the baseline — machine-made, boringly reproducible, refreshed only when it
passes (<<ch04>> covers baseline hygiene in detail). The PR job runs the same
plan, downloads the baseline, and diffs:

[source,yaml]
----
# PR job steps, after the smoke gate
- name: Run comparison plan
  run: loadr run perf/api.yaml --summary-export current.json

- name: Fetch baseline                        # the baseline job on main
  uses: dawidd6/action-download-artifact@v6   # publishes this; GitHub's own
  with:                                       # download-artifact only sees the
    workflow: perf-baseline.yml               # current run, hence the
    branch: main                              # cross-workflow action
    name: perf-baseline

- name: Compare against main
  run: |
    loadr compare baseline.json current.json \
      --max-regression p95=10% \
      --max-regression error_rate=0.5 \
      --markdown compare.md --assert

- name: Comment on the PR
  if: always()
  uses: marocchino/sticky-pull-request-comment@v2
  with: { path: compare.md }
----

`--assert` makes any regression beyond tolerance exit 99 — the same code, the
same job-failure wiring as a threshold breach. The comparison is
direction-aware (an improvement never fails the gate) and the markdown table
lands as a sticky PR comment with the regressed cells bolded, so the author
sees "p95 +33.3%, *regression*" in the conversation where the decision is
being made. A threshold that passed in the baseline but fails in the current
run is always flagged, regardless of tolerance — a newly failing gate is never
noise.

One honest note on portability: baseline comparison is where the tool
landscape thins out. k6 exports the same summary JSON (`--summary-export`) but
has no built-in diff; open-source teams script it with `jq` or a short Python
script, and the commercial tiers of k6, Gatling, and others sell the trend
analysis as a feature. The capability is what matters — if your tool doesn't
compare runs, build the twenty-line script, because "we check bounds but can't
see movement" is exactly the blind spot the ORM annotation walked through.

=== Sweeps: catching the regression that only exists at scale

The third kind of regression is invisible to both previous instruments: the
system is exactly as fast at smoke load, and exactly as fast at yesterday's
baseline load, but its *ceiling* moved. Someone shrank a connection pool,
introduced a lock, changed a GC setting — and the knee in the latency curve
slid from 200 VUs down to 120. You find knees by sweeping: one plan, run
repeatedly across a parameter matrix, tabulated side by side.

[source,console]
----
$ loadr sweep perf/api.yaml --var vus=25,50,100,200 --duration 10m

  combo    p50      p95       p99       error rate  rps
  -------  -------  --------  --------  ----------  --------
  vus=25   18.20ms  24.90ms   31.00ms   0.00%       137.9/s
  vus=50   19.10ms  28.40ms   40.20ms   0.00%       270.8/s
  vus=100  24.70ms  61.30ms   112.5ms   0.02%       509.2/s
  vus=200  71.40ms  412.80ms  1.28s     2.31%       541.0/s
----

The knee is one table read away: throughput nearly flat from 100 to 200 VUs
while p95 goes up 6.7x — the system saturated somewhere between those rows
(<<ch02>>'s Little's Law arithmetic, made visible). Each combination runs
sequentially as a full `loadr run`, so combos don't contend with each other
for load-generator resources; each drops an ordinary summary export into
`loadr-sweep/`, which means `loadr compare` works on any two cells of the
matrix.

Sweeps are long by construction — four combos at ten minutes is most of an
hour — so they run on a schedule, not on PRs:

[source,yaml]
----
# .github/workflows/perf-nightly.yml
name: Nightly perf matrix
on:
  schedule: [{ cron: "0 2 * * *" }]

jobs:
  sweep:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: levantar-ai/loadr/.github/actions/setup-loadr@v1
        with: { version: latest }

      - name: Sweep
        run: |
          loadr sweep perf/api.yaml \
            --var vus=25,50,100,200 --duration 10m \
            --out-dir sweeps --markdown sweep.md

      - name: Publish matrix
        if: always()
        run: cat sweep.md >> "$GITHUB_STEP_SUMMARY"

      - name: Keep the summaries
        if: always()
        uses: actions/upload-artifact@v4
        with: { name: perf-sweep-${{ github.run_id }}, path: sweeps/ }
----

The matrix appears on the workflow's summary page every morning, and the
per-combo JSONs are archived — so when the knee moves, running `loadr compare`
on last Tuesday's `sweep-vus-100.json` and today's shows exactly which
percentile shifted. A failing combo doesn't abort the sweep; it's reported, its row shows
what it produced, and the sweep exits 99 at the end if anything failed — so
the same red-job wiring applies. The number to put on a chart and watch week
over week is not any single latency; it's *where the knee is*. When the knee
drifts left, you've lost capacity, and you've lost it on a specific day whose
commits you can enumerate.

=== Ephemeral environments

Everything above assumed there's something to point the test at. On PRs,
increasingly, there isn't a shared staging environment — there's an ephemeral
one: a Docker Compose stack booted inside the runner, or a per-branch
namespace deployed to a shared cluster and torn down when the PR closes. For
performance work these are a genuine improvement on the traditional shared
staging box, for one big reason: *isolation*. Nobody else's test is hitting
your database mid-run, which removes the single largest source of
inter-run noise.

Two disciplines make them work. First, *warm up before you measure*. A
freshly booted environment has cold caches, empty connection pools, an
unwarmed JIT, and a database with no pages in memory; its first thirty seconds
are unrepresentative of everything after. Run a short warm-up phase and gate
only on the steady state — in loadr, a first ramp stage you exclude by
gating on named requests, or simply a plan whose duration is long enough that
warm-up is a small fraction. Second, *seed realistic data volume*. An
ephemeral database with forty rows will happily table-scan its way to
excellent latency; the missing index only costs something at forty million
rows. <<ch05>> covers data seeding in depth; the CI-specific point is that the
seed step belongs in the workflow, versioned, so every ephemeral environment
is the same environment.

And keep the conclusions honest. An ephemeral environment is a scale model —
one replica, small instance, local network. It supports *relative* claims
("this PR made checkout 2x slower than main, measured identically") and
gross-failure claims. It does not support absolute claims ("we can handle
5,000 RPS") — those belong to the nightly tier against a production-shaped
environment, and confusing the two is how a green PR gate becomes false
confidence.

=== Flakiness: the gate's mortal enemy

A performance gate dies the way trust dies: one unexplained failure at a
time. The first time the gate fails a PR that didn't cause a regression,
someone re-runs the job. The second time, re-running becomes the reflex. By
the third, the gate is decorative — every red is presumed noise, including the
one that isn't. So treat flakiness as a first-class engineering problem, not
an annoyance.

*Measure your noise floor before you set a tolerance.* Run the identical plan
five or ten times against the identical build and look at the spread of p95.
On a shared GitHub-hosted runner, run-to-run variation of 5–15 percent is
normal — the runner is a VM on contended hardware with contended egress. If
your spread is 12 percent, a 10 percent regression tolerance is a coin flip.
The tolerance must sit above the noise floor, and if the noise floor is too
high to catch what you care about, lower the noise, don't widen the gate:
longer runs (more samples shrink percentile jitter — <<ch12>>), self-hosted or
larger pinned runners for the perf jobs, ephemeral isolation as above.

*Make the generator prove its own health.* The sneakiest CI flake is the
load generator itself getting starved — a noisy-neighbor runner pegs the CPU,
timers drift, and the tool reports the *target* as slow. Make generator
health a threshold, so a sick run fails as invalid instead of failing as a
regression:

[source,yaml]
----
observe:
  - type: system
    metrics: [ cpu, memory ]
    interval: 1s

thresholds:
  # If the generator itself is unhealthy, no other number means anything.
  system_cpu:    [ "max < 0.90" ]
  system_memory: [ "max < 0.85" ]
----

This converts "mysterious latency spike, re-run it" into "runner was
saturated, run invalid" — a different failure class, routed differently, with
nobody learning to distrust the latency numbers.

*Quarantine new gates, with an expiry date.* When you introduce a gate, run it
report-only for two weeks: it runs, it comments, it does not fail the job.
You learn its false-positive rate before you spend the team's trust on it.
The critical word is *expiry* — a report-only gate with no promotion date is
the permanent science-fair project, and (as <<ch04>>'s retail story showed) a
40 percent regression will ship straight through it. Put the promotion date in
the PR that adds the gate.

[TIP]
====
Budget for flakiness explicitly: track the gate's false-positive rate like
you'd track a service SLO. If more than about one in fifty failures is noise,
stop adding gates and fix the environment. A gate the team trusts at 98
percent gets investigated; a gate trusted at 80 percent gets re-run.
====

=== Storing and trending: the regression no gate can catch

Every gate in this chapter has a shared blind spot: *slow drift*. A 2 percent
regression per week trips nothing — it's inside every tolerance you'd sanely
set — and compounds to roughly 67 percent in six months. The only instrument
that catches drift is a trend line, and a trend line needs history.

The raw material is already being produced: every tier above exports a
summary JSON. The discipline is to *keep* them somewhere with a longer
memory than CI artifact retention (GitHub's default is 90 days — enough for
diffs, not for trends). Options, in ascending order of effort:

* *Artifacts with long retention* — set `retention-days` high on the nightly
  jobs, and accept that trend analysis means downloading a folder of JSONs.
  Adequate for small teams; `loadr report` and `loadr compare` work on the
  files directly.
* *A results bucket* — the nightly job copies `summary.json` to object
  storage keyed by date and git SHA. A twenty-line script turns the bucket
  into a CSV of p95-per-build; a spreadsheet turns the CSV into the chart
  that catches the drift.
* *A metrics backend* — stream run results into Prometheus/Grafana or a
  time-series database, and put "checkout p95, nightly run, last 90 days" on
  the team dashboard next to production metrics. This is where mature teams
  land, because the chart answers the drift question passively — someone
  glances at it in standup and says "that's been creeping for a month."

Whichever store you choose, key every result by *git SHA, plan version, and
environment*. A trend line mixing two test plans, or staging-small with
staging-large runs, is a lie with an x-axis. And keep the failed runs too —
the run that breached its gate is a data point about the build that breached
it, and when someone asks "when did this start?", the archived failures are
the answer.

The weekly ritual that makes the storage worth it takes five minutes: look at
the trend, look at the knee from the sweep matrix, and ask one question — *is
anything moving in the wrong direction slower than our gates can see?*
<<ch18>> builds the organizational habit; the pipeline's job is to make the
five minutes possible.

=== Failing the build without crying wolf

Pull the threads together into a gate policy — the handful of rules that keep
a red performance check meaning something:

*Gates derive from the SLO, not from vibes.* The nightly gate is the SLO
under the target load shape (`slo(99%) < 300ms`, `rate<0.001` — <<ch04>>). The
PR smoke gate is deliberately looser, because it asks a coarser question at a
smaller scale. A gate someone made up ("let's say 200ms?") is a gate someone
will argue down the first time it's inconvenient.

*Tolerances sit above the measured noise floor.* Not the guessed one. If you
haven't run the same-build-five-times experiment, you don't know your noise
floor, and your comparison gate is set by superstition.

*Failure classes route differently.* Exit 99 (threshold or regression) means
"the system is too slow" — block the merge, notify the author. Exit 1 means
"the test never validly ran" — broken URL, unreadable baseline, saturated
generator — and it pages whoever owns the *test*, not the PR author. Collapse
these into one red X and every failure becomes an argument about whose
problem it is.

*Runaway tests stop themselves.* An abort threshold
(`abort_on_fail: true`, with a `delay_abort_eval` grace period for warm-up)
keeps a collapsing system from being pounded for the remaining twenty-nine
minutes of a scheduled run — and keeps your nightly job from taking staging
down for the team in another timezone.

*Loosening a gate is a diff.* Thresholds live in YAML in the repo, so
widening one is a pull request with a name, a review, and a blame entry.
That friction is the feature. The week before a deadline, someone will want
to bump `p(95)<300` to `p(95)<500` "temporarily" — and the PR that does it
should say so, and someone should have to approve it, and `git log` should
remember. Gates configured by hand in a vendor dashboard are gates that will
be quietly edited, and six months later nobody will know the standard ever
was different.

*Every gate has an owner and a removal path.* A gate nobody owns rots: the
endpoint it tests gets deprecated, the threshold drifts out of date, and it
fails PRs for reasons no one can explain. Gates are code; they need the same
maintenance, and deleting an obsolete one is as legitimate as adding it.

The sociology is the real system here. A performance gate is a machine for
saying *no* to your own colleagues, and it only keeps that authority while
its noes are almost always right. Spend the effort where this chapter spends
it — small honest PR gates, machine-made baselines, measured noise floors,
quarantine before enforcement — and the payoff is the counterfactual we
opened with: the nine-hundred-commit archaeology dig replaced by a red check
on the one PR that caused it, ten minutes after it was pushed.

.Key takeaways
****
* The cost of a regression is mostly detection latency. Shift the *question*
  left, not the big test: a tiered pipeline — PR smoke, merge baseline,
  nightly SLO run, scheduled capacity sweep — each sized to its trigger.
* A PR gate is a coarse instrument by design: 30–90 seconds, loose
  thresholds, near-zero false positives. It catches the N+1 and the lost
  index, not the 10 percent drift — the nightly tiers own those.
* Fixed thresholds can't see movement within bounds; compare each build
  against a machine-made baseline (`loadr compare --assert`) and gate on the
  delta. Sweeps catch the third kind of regression: a knee that moved.
* Flakiness is the gate's mortal enemy. Measure the noise floor before
  setting tolerances, make the generator prove its own health, and quarantine
  new gates report-only — with an expiry date.
* Keep every summary export, keyed by SHA and environment, and trend it.
  Slow drift trips no gate; 2 percent a week is 67 percent in six months, and
  only a chart with history will show it.
* A gate keeps its authority only while its noes are right: derive gates from
  SLOs, route "too slow" and "test broke" differently, and make loosening a
  threshold a reviewed diff with a name on it.
****

.Field Card — Performance gates in CI
****
Cost of a regression is detection latency. Reach for a *tiered* pipeline — each test sized to its trigger, each catching what the tier below can't.

Grossly worse on a PR?:: 20-VU/30s smoke gate, thresholds *loose* vs SLO (`p(99)<800` for a `300ms` SLO). Exit 99 blocks the merge.
Movement within bounds?:: fixed thresholds can't see it — `loadr compare` a machine-made `main` baseline, `--assert` on the delta.

----

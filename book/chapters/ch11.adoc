== Instrumentation and Observability

The run finishes at 14:32. The load test's summary says p99 latency was 2.4
seconds at 400 requests per second — a hard fail against the 500ms objective.
You walk over to the backend team with the bad news, and they pull up their
dashboard: p99, 180 milliseconds, flat as a table all afternoon. Now you're in
the worst meeting in performance engineering: two dashboards, two numbers, one
request, and everyone quietly convinced the other side's tooling is broken.

Both numbers were right. The client measured the full round trip. The server
measured the time its handler spent on each request — and under load, those
requests were spending over two seconds queued in the kernel's accept backlog
and the load balancer before any handler ever saw them. The server's histogram
was telling the truth about a question nobody was asking.

This chapter is about never having that meeting. A load test without
server-side observability tells you *that* something is slow; observability
without load tells you about a system nobody is stressing. The craft is wiring
them together: the same time axis, the same run identifiers, and a path from a
bad percentile to the exact span of the exact request where the time went. We
said in <<ch01>> that testing creates the conditions and observability explains
them. Here's the plumbing.

=== Two clocks, one request

Start with the disagreement itself, because it's not a bug — it's a
measurement, and often the most useful one you'll get from a run.

When your load generator reports a request duration, it's timing everything:
DNS resolution (usually cached), the TCP handshake, the TLS handshake,
transit to the load balancer, time queued *at* the load balancer, time queued
in the target's kernel listen backlog, time queued in the application's own
thread pool or event loop, the handler itself, and then streaming the response
body all the way back. When the server reports a request duration, it's
typically timing one slice of that: from "my framework's middleware saw the
request" to "I wrote the last byte of the response to my socket buffer."

Under light load the slices outside the handler are millisecond noise and the
two clocks agree. Under heavy load, queues form — and queues form precisely in
the places the server doesn't instrument by default:

[cols="2,3",options="header"]
|===
| Where the missing time hides | Why the server's histogram can't see it
| Load balancer queue | The LB hasn't forwarded the request yet
| Kernel accept backlog | The connection is accepted; no application code has run
| Thread pool / event-loop queue | The framework timer usually starts when a worker picks the request up
| TLS handshake under CPU pressure | Handshakes compete with handlers for the same cores
| Response streaming to a slow reader | Many frameworks stop the clock at first byte written
|===

The rule that falls out of this: *the client-side number is the truth about
user experience; the server-side number is a diagnostic.* When they diverge,
the divergence is itself a signal — it tells you the bottleneck is a queue
*between* the two clocks, and it points you at connection handling, backlog
sizes, and pool configuration rather than at the handler code the server-side
profile would have you optimizing.

Good load tools help you decompose their side of the clock. k6, for instance,
splits `http_req_duration` into `http_req_blocked`, `http_req_connecting`,
`http_req_tls_handshaking`, `http_req_sending`, `http_req_waiting`, and
`http_req_receiving` — if `blocked` and `connecting` grow while `waiting`
stays flat, the time is going into connection establishment, not the backend.
Whatever tool you use, learn its phase breakdown before you need it at 2 a.m.

[WARNING]
====
Before you blame the gap on the server, rule out the third clock: the load
generator itself. A saturated generator inflates client-side latency with
queueing that happens *inside your own tool*, and no amount of server-side
digging will ever find it because it isn't there. <<ch10>> covers this
failure mode in depth; later in this chapter we'll put generator CPU on the
same timeline as the latency it might be corrupting.
====

=== The three signals under load

Observability practice organizes around three signals — metrics, logs, and
traces. You've likely met them in an operations context. Under deliberate load
they behave differently enough to deserve a second introduction.

*Metrics* are pre-aggregated numbers on a clock: request rates, latency
histograms, CPU, queue depths. They're cheap, always on, and they answer *what
happened and when* — which makes them the backbone of everything in this
chapter, because "when" is the axis you correlate on. Their trap under load is
cardinality. A label like `user_id` or `session` that's harmless at ten users
becomes a few hundred thousand distinct time series during a test, and your
metrics backend falls over before the system under test does. Label by
*endpoint*, *status*, and *scenario* — categories, never identities.

*Logs* are discrete events with detail. They answer *what exactly happened at
that moment* — the query text, the exception, the order ID. Their trap is
volume: at 2,000 requests per second, one log line per request is 2,000
writes per second into your logging pipeline, plus disk I/O on the target that
wasn't in your workload model. Two consequences. First, sample or aggregate
during load tests; keep full logging for errors only. Second — and this one is
underrated — *a load test is also a load test of your observability stack*.
More than one team has discovered during a test that their real production
bottleneck was the synchronous log shipper. Better to learn it Tuesday
afternoon than during the incident.

*Traces* follow a single request across services, decomposing its latency into
a tree of timed spans. They answer *where inside the request the time went* —
the question metrics structurally cannot answer, because a histogram has
already thrown the individual request away. Their trap is cost: tracing every
request at load-test rates can melt a collector. We'll come back to sampling.

The division of labor during a run, then: metrics tell you *when* it went bad
and *how* bad; a trace tells you *where* in the request; logs tell you *what*
was happening at that spot. You'll use them in exactly that order.

=== Watching the system under test

For the system under test, the de facto standard stack is Prometheus for
metrics and Grafana for dashboards, and it's what we'll wire against — the
concepts port directly to Datadog, CloudWatch, or whatever your organization
runs.

Two mnemonics tell you what belongs on the dashboard you stare at during a
run. For each *service*: RED — rate, errors, duration. For each *resource*
(CPU, memory, pools, disks, queues): USE — utilization, saturation, errors.
In PromQL, the two service panels you'll build first:

[source,text]
----
# Rate: requests per second, by status class
sum by (status) (rate(http_server_requests_seconds_count[1m]))

# Duration: server-side p99 from a histogram
histogram_quantile(0.99,
  sum by (le) (rate(http_server_requests_seconds_bucket[1m])))
----

[WARNING]
====
`histogram_quantile` interpolates *within* buckets, so its answer is only as
honest as the bucket boundaries. With a default bucket layout that jumps from
1s to 2.5s, every p99 between those bounds gets smeared across the gap — the
dashboard can show "p99 = 1.8s" for a service whose real p99 is 2.4s, or
report a regression as noise. If tail latency is the thing you're gating on
(it is), define buckets densely around your SLO boundary, or use native
histograms where your Prometheus version supports them. <<ch12>> treats the
general problem of quantile estimation error; this is its server-side face.
====

Beyond RED, put the *saturation* signals for your specific architecture on the
same board before the test, not after: connection-pool utilization (database
and HTTP client pools both), thread-pool or event-loop queue depth, GC pause
time for managed runtimes, and database-side metrics. In the hunt of <<ch13>>,
nine times out of ten the smoking gun is one of these hitting a ceiling at
exactly the moment client-side p99 leaves the ground — but you can only see
"exactly the moment" if everything shares a time axis. Which brings us to the
step most teams skip.

=== One time axis: the load test on the same dashboard

Here is the move that changes how you debug: *export the load test's own
metrics into the same Prometheus, and graph them on the same Grafana
dashboard as the server's.* Offered RPS, client-side p99, error rate, and
active VUs in the top row; server CPU, GC, pools, and queue depths below —
one shared time cursor across all of it.

loadr exposes a Prometheus scrape endpoint for the duration of a run:

[source,yaml]
----
# loadr.yaml — expose live run metrics for Prometheus to scrape
scenarios:
  load:
    executor: constant-vus
    vus: 25
    duration: 10m
    think_time: { type: uniform, min: 200ms, max: 800ms }
    flow:
      - request: { name: list,   url: /api/items,   checks: [ { type: status, equals: 200 } ] }
      - request: { name: detail, url: /api/items/1, checks: [ { type: status, equals: 200 } ] }

outputs:
  - { type: prometheus, listen: 0.0.0.0:9091 }   # 0.0.0.0 so Prometheus-in-Docker can reach it

thresholds:
  http_req_duration: [ "p(95)<400" ]
  http_req_failed:   [ "rate<0.02" ]
----

Point a scrape job at it:

[source,yaml]
----
# prometheus.yml
scrape_configs:
  - job_name: loadr
    scrape_interval: 5s
    static_configs:
      - targets: ["host.docker.internal:9091"]
----

and the run appears in Prometheus as `loadr_http_reqs_total`,
`loadr_http_req_duration_milliseconds`, `loadr_checks_rate`, `loadr_vus`, and
any custom metric you've declared under the same prefix — a custom counter
`orders` arrives as `loadr_orders_total`. If your metrics
platform prefers push over scrape — Grafana Cloud, Mimir, Thanos receivers —
swap the output for remote-write:

[source,yaml]
----
outputs:
  - { type: prometheus, remote_write_url: "https://prom.internal/api/v1/write" }
----

The same concept exists across the tool landscape, because everyone eventually
learns they need it. In k6 it's the Prometheus remote-write output:

[source,console]
----
$ K6_PROMETHEUS_RW_SERVER_URL=http://prometheus:9090/api/v1/write \
    k6 run -o experimental-prometheus-rw script.js
----

JMeter ships a Backend Listener that streams live results to InfluxDB with a
well-worn community Grafana dashboard on top; Gatling and Locust have Graphite
and Prometheus integrations respectively. The spelling varies; the payoff is
identical: when client p99 jumps at 14:19:40 and the JVM's GC pause metric
jumps at 14:19:40 on the panel directly beneath it, the correlation does the
first half of the diagnosis for you.

[TIP]
====
Correlation on a shared time axis assumes shared time. Make sure the load
generator and the target both run NTP-disciplined clocks; a generator VM
drifting by eight seconds will have you "correlating" a latency spike with a
deploy that happened after it. When in doubt, check the offset before a run —
it's one command and it has saved me from at least two wrong conclusions.
====

=== Closing the loop: pulling server metrics into the run

Dashboards are for humans watching. For a run you want to *gate* — in CI, in
a nightly soak — the correlation should happen inside the test itself, so the
run can fail on server-side evidence without anyone watching a screen.

loadr's `observe` block is the inverse of an output: instead of pushing load
metrics out, it pulls foreign metrics *in* for the run window, overlays them
on the run timeline, and makes them thresholdable like any other metric:

[source,yaml]
----
observe:
  - type: system                 # the generator's own CPU/memory/disk/network
    metrics: [ cpu, memory ]
    interval: 1s
  - type: prometheus             # the target, via a PromQL range query
    name: api cpu
    source: http://prometheus:9090
    query: sum(rate(container_cpu_usage_seconds_total{pod=~"api-.*"}[1m]))
    as: target_cpu
    unit: ratio

thresholds:
  system_cpu:  [ "max < 0.90" ]  # generator saturated -> the numbers are lies
  target_cpu:  [ "max < 0.75" ]  # server has no headroom -> fail, even if latency passed
  http_req_duration: [ "p(99)<500" ]
----

Read those three thresholds together, because they encode this chapter's
argument. The first protects the *validity* of the run: if the generator's own
CPU pinned, the latency numbers are corrupted and the run should fail as
inconclusive rather than pass as flattering (<<ch10>>). The third is the
familiar client-side gate. The second is the interesting one: a run whose
latency passed *while the target ran at 90 percent CPU* is a pass with no
headroom — one marketing email away from a failure. Gating on server-side
utilization turns "we passed" into "we passed with margin," which is the only
kind of pass worth reporting to the capacity conversation in <<ch16>>.

No tool monopoly here either: the portable version is a post-run script that
queries the Prometheus HTTP API over the run window and asserts on the result.
Ten lines of shell gets you most of the value:

[source,console]
----
$ curl -sG http://prometheus:9090/api/v1/query_range \
    --data-urlencode 'query=max_over_time(node_load1[1m])' \
    --data-urlencode "start=${RUN_START}" \
    --data-urlencode "end=${RUN_END}" \
    --data-urlencode 'step=15' | jq '[.data.result[].values[][1] | tonumber] | max'
----

Wrap it in a threshold check and exit nonzero, and your CI pipeline (<<ch14>>)
now fails on server-side saturation regardless of which load tool produced the
traffic.

=== Tracing the slow request

Metrics got you here: p99 is 2.4 seconds, it started at 14:19, and it
correlates with database connection-pool utilization hitting 100 percent. But
a histogram is an aggregate — the individual slow requests are gone, and
"correlates with" is not "caused by." To finish the diagnosis you want to hold
one 2.4-second request in your hands and see where its time went. That's a
distributed trace: the request decomposed into a waterfall of spans, one per
operation, across every service it touched. The one I'm describing showed a
12ms handler, an 8ms query — and 2.3 seconds in a span named `pool.acquire`.
The database was never slow. The *pool was too small for the concurrency*,
and every aggregate in the building had been pointing vaguely at "the
database" for an hour.

The mechanics come from OpenTelemetry: each instrumented service records
spans, and context propagates between services in a `traceparent` HTTP header
(the W3C Trace Context standard), so the backend can stitch one request's
spans into one tree. Your services need to be instrumented for this — that's
an application concern, and OTel's auto-instrumentation agents make it cheap
for most mainstream stacks.

The load generator's job is smaller but load-bearing: *start the trace
context*, so every synthetic request is traceable end to end and
distinguishable from organic traffic. In loadr it's one line:

[source,yaml]
----
defaults:
  http:
    base_url: https://api.example.com
    tracing: true        # inject a fresh W3C traceparent header per request
----

Every request now carries a unique `traceparent`, and your tracing backend —
Jaeger, Tempo, Honeycomb — sees each one as a root trace. k6 does the
equivalent with its HTTP instrumentation library, and for any tool without
native support you can set the header yourself; it's just an HTTP header with
a well-specified format.

The workflow, once both sides are wired:

1. Run the test with tracing enabled.
2. In the tracing UI, search the run window for traces with
   `duration > 2s` — the p99 exemplars, by construction.
3. Open three or four. Under load, slow requests are rarely slow in unique
   ways; the same span will dominate all of them.
4. Read the offending span's attributes, and *now* go to the logs — filtered
   by that trace ID — for the exact statement, lock, or exception involved.

Metrics → trace → logs, each signal narrowing the search space for the next.
This is the systematic hunt of <<ch13>> in miniature.

Two practicalities. *Sampling*: a ten-minute run at 400 RPS is 240,000
requests, and tracing all of them will stress your collector and your storage
bill. Head-based sampling (keep a random 1 percent, decided at the start of
the request) is cheap but statistically likely to *discard* the rare slow
requests you ran the test to find. Tail-based sampling — the collector buffers
each trace and keeps it if it turned out slow or errored — keeps exactly the
interesting ones, at the price of a more complex collector pipeline. For load
testing, tail-based is worth the setup. *Exemplars*: Prometheus can attach
sampled trace IDs to histogram buckets, and Grafana renders them as dots on
the latency panel — click the dot on the p99 spike, land in the trace. It
collapses the four-step workflow above into one click, and it's the single
best quality-of-life upgrade in this whole chapter.

=== Tagging the run: which test did this to us?

Last piece, unglamorous and essential: six weeks from now, someone will look
at a CPU plateau on a Grafana dashboard and ask "was that a load test?" Every
synthetic sample and every server-side effect should be traceable to the run
that caused it — which run, which build, which scenario.

On the load-test side, tag every sample at the source:

[source,yaml]
----
defaults:
  http:
    base_url: https://api.example.com
  tags:
    run: nightly-soak
    build: ${env.GITHUB_SHA}     # added to every sample this run emits
----

Those tags flow through every output — Prometheus labels, JSON fields, CSV
columns — so the run's metrics are filterable in the same backend as the
server's. They also unlock *scoped* pass/fail gates, so one noisy endpoint
can't hide behind an aggregate:

[source,yaml]
----
thresholds:
  "http_req_duration{name:checkout}": [ "p(99)<800" ]   # gate the endpoint that matters
  "checks{scenario:browse}":          [ "rate>0.98" ]
----

On the server side, mark the run window where humans will look for it. The
Grafana annotations API takes one `curl` at run start and one at run end, and
every dashboard in the organization gets a labeled vertical band saying
`loadr nightly-soak, build 3f2c91a`:

[source,console]
----
$ curl -s -X POST http://grafana:3000/api/annotations \
    -H "Authorization: Bearer $GRAFANA_TOKEN" -H "Content-Type: application/json" \
    -d "{\"tags\":[\"loadtest\",\"$GITHUB_SHA\"],\"text\":\"loadr nightly-soak\"}"
----

And propagate the identity *into* the traffic itself: a header like
`x-load-test: nightly-soak-3f2c91a` on every synthetic request lets backend
teams filter their logs and — critically, when you start testing in production
in <<ch15>> — lets downstream systems recognize and, where necessary, divert
synthetic traffic. Finally, archive the client-side artifact under the same
identity: export the run summary (`loadr run --summary-export results.json
soak.yaml`) and store it keyed by the same build SHA, so the regression comparisons of
<<ch14>> and the capacity models of <<ch16>> can join load-test results to
server-side history without archaeology.

That's the full loop. The load test stamps its identity on everything it
touches; the servers' telemetry lands on the same time axis as the client's;
the run itself can pull server metrics back in and fail on them; and any bad
percentile is one click from the trace that explains it. The 14:32 meeting
still happens — but now it's short, and it ends with a pool-size change
instead of an argument.

.Key takeaways
****
* Client-side and server-side latency measure different slices of the same
  request. Divergence between them isn't a tooling bug — it's a finding, and
  it points at queues (backlogs, pools, load balancers) that neither side
  instruments by default.
* Metrics tell you *when* and *how bad*; traces tell you *where inside the
  request*; logs tell you *what exactly*. Use them in that order, and
  remember every load test is also a load test of the observability stack.
* Export the load tool's metrics into the same Prometheus/Grafana as the
  system under test. One shared time axis does the first half of every
  diagnosis.
* Close the loop in automation: pull server-side metrics into the run
  (loadr's `observe`, or a post-run Prometheus API query) and gate on them —
  a latency pass with no server headroom is a failure wearing a green badge.
* Under load, sample traces tail-based, not head-based: the slow one percent
  is the whole reason you're tracing.
* Tag everything with a run and build identity — samples, annotations, a
  header on the traffic itself — so every server-side effect is attributable
  to the test that caused it.
****

== Modeling the Workload

_A load test is a hypothesis about how the system will be used. This chapter is
about making that hypothesis realistic._

A European ticketing company once showed me a load test they were proud of. It
simulated 5,000 concurrent users against their booking flow, ran clean for an
hour, and produced a p95 of 180ms. Two weeks later a stadium tour went on sale
and the site collapsed inside four minutes — at a moment when their own
analytics showed fewer than 3,000 people on the site.

The test wasn't underpowered. It was *mis-modeled*. Their 5,000 simulated users
politely waited for each response before sending the next request, so when the
booking service slowed down, the simulated crowd slowed down with it and the
test's own load gently receded. The real crowd did the opposite: ticket buyers
at 10:00 a.m. on a Friday do not wait. They arrive, and they keep arriving, and
when the page hangs they open a second tab and arrive again. The test modeled a
patient queue; production delivered a firehose. Same endpoint, same
infrastructure, same nominal "user count" — completely different physics.

Everything in this chapter is downstream of that story. Before you argue about
tools, percentiles, or environments, you have to answer one question honestly:
*what does the demand on this system actually look like?* Get that right and a
mediocre tool will still tell you the truth. Get it wrong and the best tool in
the world will produce a confident, precise, irrelevant number.

=== Open versus closed: the choice that decides what your test means

Queueing theorists sorted this out decades ago, and the vocabulary is worth
adopting because it names the exact mistake the ticketing company made.

In a *closed* workload model, a fixed population of users cycles through the
system. Each user issues a request, waits for the response, maybe thinks for a
moment, then issues the next one. The key property: *a new request cannot start
until the previous one finishes.* The number of in-flight requests is capped by
the population size, no matter how slow the system gets.

In an *open* workload model, requests arrive according to some external
process — a rate — that does not care whether previous requests have completed.
The internet does not check whether your checkout service has caught up before
sending it the next customer.

Here is why the distinction is not academic. Recall Little's Law from <<ch02>>:
throughput equals population divided by round-trip time. Take a closed test
with 50 virtual users, each pausing an average of 3 seconds between actions,
against a service responding in 500ms. Each user completes a cycle every 3.5
seconds, so the whole population offers about 14 requests per second. Now
suppose a deploy makes the service four times slower — 2 seconds per response.
The cycle stretches to 5 seconds and the offered load *falls* to 10 requests
per second. The system degraded, and your test rewarded it by easing off. The
worse the system behaves, the gentler a closed test becomes. This is the
self-throttling feedback loop at the heart of coordinated omission (<<ch02>>,
and in gory detail in <<ch12>>), and it is how a test "passes" at load levels
production will never be lucky enough to see.

An open test has no such mercy. If you specify 100 arrivals per second and the
system slows down, requests stack up, concurrency climbs, queues grow — exactly
as they do in production. The slowdown is *visible in the results* instead of
absorbed by the load generator's good manners.

So which model is right? The one that matches reality:

* *Open* fits systems whose demand comes from a large, effectively unbounded
  population that doesn't coordinate: public websites, APIs consumed by other
  companies, mobile-app backends, anything behind a marketing campaign. Most
  internet-facing traffic is open. When in doubt, this is the default.
* *Closed* fits systems with a genuinely fixed concurrency: 40 call-center
  agents on an internal CRM, a batch pipeline with 16 workers, a partner
  integration that runs exactly 8 connections. Here the population cap is real,
  and modeling it as open would overstate the demand.

[WARNING]
====
Choosing the wrong model doesn't make your test noisy — it makes it answer a
different question than the one you asked. A closed test of an open system
measures "how does the system behave when demand politely adapts to its
capacity?" No real system enjoys that courtesy. This single modeling error
invalidates more load tests than every tooling bug combined.
====

=== Saying it in a tool: arrival rate versus virtual users

Every serious load tool exposes this choice, though rarely with a sign that
says "warning: physics decision." In loadr it's the `executor` field. A closed
model counts *virtual users*:

[source,yaml]
----
scenarios:
  browse:
    executor: ramping-vus        # closed: 50 users cycling through the flow
    start_vus: 0
    stages:
      - { duration: 2m, target: 50 }   # ramp up
      - { duration: 5m, target: 50 }   # hold
      - { duration: 1m, target: 0 }    # ramp down
    think_time: { type: uniform, min: 1s, max: 4s }
    flow:
      - request: { name: home, url: / }
      - request: { name: products, url: /products }
----

An open model specifies *arrivals per unit time*:

[source,yaml]
----
scenarios:
  steady_api_load:
    executor: constant-arrival-rate  # open: 100 starts/s, no matter what
    rate: 100
    time_unit: 1s
    duration: 10m
    pre_allocated_vus: 50            # worker pool created up front
    max_vus: 200                     # pool may grow to this under slowdown
    flow:
      - request: { name: health, url: /health }

thresholds:
  http_req_duration: [ "p(99)<300" ]
  dropped_iterations: [ "count<10" ]  # the saturation tripwire
----

Two details in that second file deserve a hard stare. First, `max_vus`: an open
generator still needs workers to carry the arrivals, and if responses slow to 3
seconds at 100 arrivals per second, Little's Law says you need about 300
in-flight workers — more than the pool allows. Second, `dropped_iterations`:
when the pool is exhausted, loadr (and k6, which uses the same executor
semantics) *drops* the arrivals it can't start and counts them, rather than
silently converting your open test back into a closed one. A nonzero
`dropped_iterations` is not an inconvenience to tune away by raising `max_vus`;
it's often the first honest signal that the system under test has saturated.
Put a threshold on it and let it fail the run.

The concept ports directly. Gatling has drawn this distinction sharply for
years in its injection profiles:

[source,scala]
----
setUp(
  scn.inject(
    rampUsersPerSec(0).to(100).during(30.seconds),  // open model
    constantUsersPerSec(100).during(10.minutes)
  ).protocols(httpProtocol)
)
----

k6 spells the closed model `constant-vus`/`ramping-vus` and the open model
`constant-arrival-rate`/`ramping-arrival-rate`, exactly mirroring the YAML
above. JMeter's classic thread group is closed — a fixed pool of threads
looping — and stays closed even when you bolt on a Constant Throughput Timer,
which merely paces the pool; genuinely open arrivals need the Open Model
Thread Group plugin. Locust is closed by construction (a population of
simulated users with `wait_time`), with open-ish behavior only via extensions.
None of these designs is wrong. But you must know which physics your tool is
giving you, because the default is usually closed and the internet is usually
open.

=== Think time and pacing: the silences between requests

Real users spend most of their session doing nothing you can see: reading,
scrolling, hunting for a credit card. A journey of six requests might span
ninety seconds of wall time, of which the server sees 900ms. If your test
replays those six requests back-to-back with no gaps, you haven't simulated one
user — you've simulated a loop that iterates as fast as your service can
answer, which at 150ms per request is roughly a hundred humans' worth of
demand wearing one VU's name tag.

This is why "we tested with 500 users" is a nearly information-free sentence.
Five hundred VUs with zero think time might offer 3,000 requests per second;
five hundred VUs with realistic ten-second think times might offer fifty. Both
are legitimate tests. They are not the same test, and only one of them
resembles your production traffic.

In loadr, think time is a step or a default, with a distribution:

[source,yaml]
----
defaults:
  think_time: { type: uniform, min: 1s, max: 4s }  # after every request

scenarios:
  reader:
    executor: constant-vus
    vus: 50
    duration: 10m
    flow:
      - request: { name: article, url: /articles/42 }
      - think_time: { type: gaussian, mean: 8s, std_dev: 2s }  # reading
      - request: { name: next, url: /articles/43 }
----

Use a distribution (`uniform` or `gaussian`), not a constant. Fifty VUs pausing
exactly 3 seconds fall into lockstep and hammer the server in synchronized
pulses with dead air between them — a pattern no human crowd produces. Jitter
is realism.

*Pacing* is think time's stricter sibling: instead of "pause this long between
steps," it means "hold the scenario to this iteration rate," stretching the
gaps as needed. It's JMeter's Constant Throughput Timer by another name:

[source,yaml]
----
scenarios:
  steady:
    executor: constant-vus
    vus: 20
    duration: 10m
    pacing: { iterations_per_second: 10 }  # ~10 it/s across all 20 VUs
    flow:
      - request: { url: / }
----

Pacing gives a closed model a rate *ceiling*, which is useful when porting old
JMeter plans or modeling a fixed worker pool with a quota. But notice what it
can't do: if the system slows until 20 VUs can no longer sustain 10 iterations
per second, the rate silently sags. When the *offered* rate is the thing you're
asserting, use an arrival-rate executor and let `dropped_iterations` tell you
when reality diverged from the plan.

=== Poisson arrivals: why "100 per second" never means 100 every second

Independent users don't coordinate their clicks, and the sum of many
uncoordinated sources is well approximated by a Poisson process. The
practical consequence is variance. A Poisson stream
averaging 100 arrivals per second has a standard deviation of 10 per
second — so roughly one second in six will carry more than 110 arrivals, and
you'll see a 120-arrival second every minute or two, all without anything
"unusual" happening. Averages smooth; reality clumps.

Most arrival-rate generators, loadr and k6 included, space iteration starts
*evenly*: 100 per second means one start every 10 milliseconds, like a
metronome. That is smoother than any real traffic and therefore slightly
kinder to the system than the Poisson stream it stands in for — evenly spaced
arrivals never stack up by chance, so queues that would flicker into existence
under real traffic stay empty under the test. For most capacity questions the
difference is small and the determinism is worth it. But be aware of the bias,
and compensate in two ways.

First, leave headroom: if the busy hour averages 100 requests per second,
validating at exactly 100 proves the average and nothing about the clumps.
Testing at 120–130 covers the routine statistical bursts for free.

Second, know that real traffic is often *burstier than Poisson*. Poisson
models independent arrivals, and plenty of real arrivals aren't independent: a
push notification lands on a million phones inside a minute; a retry storm
synchronizes clients; a cron fleet fires at :00; a TV ad airs. Traffic studies
since the 1990s have found self-similar burstiness in web traffic at every
timescale. You don't handle that with a fudge factor on a steady test — you handle
it with an explicitly bursty profile, which is what the spike test at the end
of this chapter is for.

=== Deriving the model from production telemetry

Everything so far tells you what shape the model should take. Production tells
you the numbers. If the system already exists, the workload model is not a
brainstorm — it's an extraction job, and an hour with the access logs beats a
day of stakeholder guessing.

You are mining for five things:

1. *Rate*: requests per second, at the busy hour, not the daily average.
2. *Mix*: which endpoints, in what proportions.
3. *Journeys*: how requests chain into sessions (entry pages, typical paths,
   session length).
4. *Think time*: the gaps between a session's requests.
5. *Data shape*: payload sizes, cache-buster diversity, the ratio of new to
   returning users. (This becomes <<ch05>>'s problem, but capture it now.)

Two lines of shell against an nginx-style access log get you the first two:

[source,console]
----
# requests per minute across the day — find the busy hour
$ awk -F'[' '{ print substr($2,1,17) }' access.log | uniq -c | sort -rn | head

# endpoint mix during that hour (strip query strings)
$ grep '06/Jul/2026:19' access.log | awk '{ print $7 }' | sed 's/?.*//' \
    | sort | uniq -c | sort -rn | head
----

If you have an APM or a metrics stack, so much the better: a Prometheus query
like `sum(rate(http_requests_total[5m])) by (handler)` over the last month, or
the equivalent in Datadog or CloudWatch, gives you rate and mix with less awk.
Session journeys come from analytics tools or from grouping log lines by
session cookie; think-time distributions fall out of the timestamp gaps within
those groups.

[TIP]
====
Don't chase precision past the first significant figure. A model that says
"roughly 70 percent browse, 20 percent search, 10 percent buy, about 300 rps
at peak" and is honest about its provenance beats a beautifully precise model
of last year's traffic. Revisit the extraction quarterly — workloads drift,
and a workload model has a shelf life.
====

And if the system is new, with no telemetry to mine? Then say so out loud.
Derive a demand estimate from first principles — marketing's launch forecast,
comparable products, the size of the invited beta — write down every
assumption, and label the test results with them. A load test of a new product
is a test of the *assumptions* as much as the system; the discipline is making
them visible enough to be argued with.

=== The busy hour, and the peaks beyond it

Whatever number your telemetry produces, resist the daily average. Traffic to
almost any human-facing system concentrates: a typical consumer product does
2–4x its daily-average rate at the evening peak; a B2B product spikes when the
workday starts across its biggest time zone; a food-delivery platform lives
and dies by ninety minutes around dinner. Capacity planners have called this
the *busy hour* since the telephone era, and it — not the average — is your
baseline design load.

Above the busy hour sit the *seasonal* peaks: Black Friday retail at 5–10x
normal, tax-deadline week, results day, the annual enrollment window. And
above those, the *event* peaks that arrive on a schedule you don't control —
the press mention, the viral post, the competitor's outage dumping their
users onto you. A defensible testing program states its targets in these
terms, each with provenance:

* *Baseline*: current busy-hour load (measured, from telemetry).
* *Design peak*: baseline x seasonal multiplier x twelve months of forecast
  growth (each factor written down and attributed).
* *Resilience target*: 2–3x design peak, where you're no longer asking "does
  it meet the SLO?" but "does it degrade without falling over?" — a stress
  test in the <<ch01>> taxonomy, with different pass criteria (<<ch04>>).

=== User journeys and scenario weighting

A workload is not a pile of independent endpoints; it's a population of
*journeys* in proportion. Your telemetry said 70 percent of sessions browse
and bounce, 20 percent search, 10 percent buy. The mix matters more than it
looks: browsing is cache-friendly reads, checkout is uncacheable writes with
payment calls and inventory locks. A test that hits endpoints in equal
proportions overstates cache hit rates, understates lock contention, and
produces a p95 for a workload that doesn't exist.

In loadr, one way to express the mix is a weighted branch inside a single
scenario — each iteration rolls the dice and becomes one kind of user:

[source,yaml]
----
scenarios:
  mixed_traffic:
    executor: constant-vus
    vus: 50
    duration: 2m
    flow:
      - random:
          strategy: weighted
          choices:
            - weight: 70
              name: browse
              steps:
                - request: { name: home, url: / }
                - think_time: { type: uniform, min: 1s, max: 3s }
                - request: { name: category, url: "/c/widgets" }
            - weight: 20
              name: search
              steps:
                - request: { name: search, url: "/search?q=widget" }
            - weight: 10
              name: checkout
              steps:
                - request: { name: cart, url: /cart }
                - request:
                    name: checkout
                    method: POST
                    url: /checkout
                    checks: [ { type: status, equals: 200 } ]
----

Each branch is named, so per-journey latency falls out of the results without
extra work — and it should, because "the site is fine, checkout is dying" is
exactly the finding an aggregate p95 hides.

Locust expresses the identical idea with task weights, and it reads almost
like the telemetry that produced it:

[source,python]
----
class ShopUser(HttpUser):
    wait_time = between(1, 4)

    @task(7)
    def browse(self):
        self.client.get("/")
        self.client.get("/c/widgets")

    @task(2)
    def search(self):
        self.client.get("/search?q=widget")

    @task(1)
    def checkout(self):
        self.client.get("/cart")
        self.client.post("/checkout", json={"sku": "A-1", "qty": 1})
----

(Gatling's `randomSwitch` and Artillery's scenario weights are the same shape
again.) Weighted branches suit journeys that share a model and a schedule.
When the journey *classes* differ structurally — human browsers that are
closed-ish and think, plus a partner API that is pure open arrivals — give
each its own scenario with its own executor and run them concurrently. That
composition is <<ch06>>'s subject.

=== Steady, ramp, or spike: the shape of load over time

The final modeling axis is time. The same scenario, driven by a different
load-over-time profile, answers a different question from <<ch01>>'s taxonomy:

*Steady state* — ramp to the target, hold long enough for warmup to wash out
and for the metrics to stabilize, and measure. This is the workhorse profile
for "do we meet the SLO at design load?", and the hold must dominate the run:
2 minutes of ramp and 10 minutes of hold, not the reverse. (Held for hours,
the same profile becomes a soak test hunting leaks and drift.)

*Ramp* — climb through the target and beyond in stages, watching where latency
leaves linearity. This is how you find the knee for capacity work (<<ch17>>).
A ramp that's too steep smears the answer: give each level time to settle or
you'll never know which load level caused the degradation you're seeing.

*Spike* — the shape everything above has been warning you about: the push
notification, the flash sale, the failover that redirects a region's traffic.
A spike test is a step function, and the question is less "did latency stay
low during the surge?" (it won't, not entirely) than "did the system shed or queue
gracefully, and did it *recover* when the surge passed?" Which is why the
profile must include the aftermath:

[source,yaml]
----
scenarios:
  spike:
    executor: ramping-arrival-rate   # open model — a spike is never polite
    start_rate: 20
    time_unit: 1s
    pre_allocated_vus: 100
    max_vus: 500
    stages:
      - { duration: 2m,  target: 20 }    # baseline
      - { duration: 10s, target: 200 }   # spike: 10x in ten seconds
      - { duration: 1m,  target: 200 }   # sustained surge
      - { duration: 10s, target: 20 }    # release
      - { duration: 2m,  target: 20 }    # verify recovery

thresholds:
  http_req_failed:
    - { threshold: "rate<0.05", abort_on_fail: true, delay_abort_eval: 30s }
----

Note the closing act. The final baseline stage is the most diagnostic part of
the run: connection pools that never drain, caches evicted and thrashing,
autoscalers still flapping, a queue that's still digesting the surge five
minutes later. A system that survives the spike but never returns to its
baseline latency didn't pass. And note `abort_on_fail` — when errors explode
past 5 percent, the run stops rather than spending twenty minutes bludgeoning
a system that has already answered the question.

A workload model, then, is the product of every choice in this chapter: open
or closed, rate or users, the silences between requests, the burstiness on
top of the average, numbers mined from the busy hour, journeys in telemetry-
derived proportions, and a load shape matched to the question. Write it down —
half a page is plenty — and attach it to every result the test produces.
When someone questions your numbers in a year, the model is the part you'll
need to defend. It's also, if you've done this chapter's work, the part you
can.

.Key takeaways
****
* Open versus closed is the workload decision. Closed models self-throttle
  when the system slows; open models keep arriving, like the internet does.
  Model an open system as closed and the test quietly eases off exactly when
  production wouldn't.
* Specify open load as an *arrival rate* and treat `dropped_iterations` (or
  your tool's equivalent) as a saturation alarm, not a nuisance.
* "N users" is meaningless without think time. Use jittered distributions,
  and remember that zero think time makes one VU impersonate a hundred humans.
* Averages smooth; arrivals clump. A 100 rps average routinely delivers
  120 rps seconds — test with headroom, and test genuine bursts with an
  explicit spike profile that includes the recovery.
* Derive rate, mix, journeys, and think time from production telemetry when
  it exists; when it doesn't, write your assumptions down where they can be
  argued with. Design for the busy hour and the seasonal peak, never the
  daily average.
* Weight journeys to match reality — the browse/search/checkout mix changes
  cache behavior, write contention, and every percentile you'll report.
* Steady, ramp, and spike profiles answer different questions from <<ch01>>'s
  taxonomy. Choose the shape for the question, and give every result its
  workload model as provenance.
****

[.fieldcard]
.Field Card — Modeling the Workload
****
Before you argue tools or percentiles, pin down what demand actually looks like — the model is the part you'll defend in a year.

Internet-facing, unbounded population:: use an arrival-rate executor (`constant-arrival-rate`/`ramping-arrival-rate`); it keeps arriving when the system slows. This is the default.
Genuinely fixed concurrency (40 agents, 16 workers):: closed `constant-vus`/`ramping-vus` is honest; open would overstate demand.
Asserting the *offered* rate:: threshold on `dropped_iterations` — nonzero means saturated, not "raise `max_vus`".
Modeling a session:: jittered think time (`uniform`/`gaussian`), never a constant — zero think time makes one VU impersonate ~100 humans.

[source,console]
----
# mine the busy hour, then the endpoint mix during it
awk -F'[' '{ print substr($2,1,17) }' access.log | uniq -c | sort -rn | head
# open-model spike with a recovery tail, errors gated
loadr run examples/04-spike-test.yaml
----

*Remember:* busy hour is 2–4x the daily average — never test the average. Design peak = baseline x seasonal x growth; resilience target 2–3x that. A 100 rps average delivers 120+ rps seconds, so validate at 120–130. A spike profile MUST include the post-surge baseline stage — no recovery, no pass.
****

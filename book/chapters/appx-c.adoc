== Checklists

Pilots with twenty thousand hours still run the pre-flight checklist, and not
because they've forgotten how flaps work. They run it because the failure mode
of experts is not ignorance — it's skipping a step they've done a thousand times,
on the one day it mattered. Performance testing has the same shape: every trap
in this book is one a competent engineer walked into while being competent.

These four checklists mark the moments where you are most likely to fool
yourself: before you design, before you press Enter, after the run finishes,
and in the thirty seconds before you repeat a number to someone who will act on
it. Print them. Put them in the PR template. The point of a checklist is that
you don't get to decide, in the moment, that today you don't need it.

=== The test-plan checklist

Run this before writing a single line of scenario YAML. Most tests are lost
here, not at execution (<<ch03>> through <<ch06>> are the long version).

*Objectives*

* [ ] The test answers *one named question* (load? stress? spike? soak?
  capacity?) — and everyone involved agrees which one. See the taxonomy in
  <<ch01>>.
* [ ] Someone will make a *decision* based on the result. If no decision hangs
  on it, don't run it.
* [ ] You know what "pass" looks like *before* the run, in writing.

*Workload model*

* [ ] Load is expressed in the right model — open (arrival rate) when you're
  simulating independent users arriving from the internet, closed (fixed VUs)
  only when real-world concurrency is genuinely bounded (<<ch03>>).
* [ ] Target rates come from production data or a defensible forecast, not a
  round number that felt impressive.
* [ ] The request *mix* reflects reality — reads and writes in production
  proportion, not 100 percent `GET /health`.
* [ ] Think time and pacing are modeled where they exist in reality.

*Data*

* [ ] Test data has production-like cardinality: thousands of distinct users
  and records, not one user ID hammered a million times into a warm cache
  (<<ch05>>).
* [ ] Data-dependent hot spots (the one customer with 400,000 orders) are
  represented.
* [ ] Writes are handled: the database either resets between runs or the test
  tolerates accumulation — and you know which.

*Environment*

* [ ] The environment's delta from production is *written down*: instance
  sizes, replica counts, cache tiers, feature flags, data volume.
* [ ] Downstream dependencies are either real, or stubbed with realistic
  latency — not stubbed with 0ms and forgotten.
* [ ] TLS, CDN, auth, and rate limiters match production's posture.

*Success criteria*

* [ ] Every objective is machine-checkable — a threshold the tool evaluates,
  not a vibe a human squints at afterward (<<ch04>>).
* [ ] Criteria are percentile-based (p95/p99), never average-based.
* [ ] An error-rate gate exists. A fast 500 is not a success.

=== The pre-run readiness checklist

Two minutes before you start the run. The theme is: will this run produce a
number worth analyzing, and can it hurt anyone?

* [ ] *Warm-up planned.* JITs, connection pools, and caches need a ramp before
  the measurement window; you know which portion of the run you'll discard.
* [ ] *Monitoring is up on both sides.* Server-side dashboards (CPU, memory,
  GC, connection pools, downstream latency) are open and recording *now*, not
  reconstructable later. A load test without server-side telemetry tells you
  _that_, never _why_ (<<ch11>>).
* [ ] *Clocks and run IDs correlate.* You can line up load-generator timestamps
  with server metrics afterward; the run is tagged or logged somewhere findable.
* [ ] *Generator headroom verified.* The load generator has CPU, sockets, file
  descriptors, and bandwidth to spare — and the test itself watches for the
  telltale of a starved generator (<<ch10>>):

[source,yaml]
----
thresholds:
  http_req_duration: [ "p(99)<300" ]
  dropped_iterations: [ "count<10" ]   # a starved generator drops arrivals
----

If `dropped_iterations` climbs, the open model silently degraded into a closed
one and your latency numbers are fiction. k6 exposes the same counter under the
same name; watch it there too.

* [ ] *Blast radius bounded.* You know what shares infrastructure with the
  target — databases, rate limiters, third-party APIs with billing meters — and
  the people who own it know you're running. For anything near production,
  <<ch15>> applies.
* [ ] *A circuit breaker is armed.* The test aborts itself when the target is
  clearly in distress, instead of kicking it for another 55 minutes:

[source,yaml]
----
thresholds:
  http_req_failed:
    - { threshold: "rate<0.02", abort_on_fail: true, delay_abort_eval: 2m }
----

* [ ] *You can stop it by hand.* You know the kill command and it's one
  keystroke away, not a runbook away.
* [ ] *Results will be captured.* `--summary-export` (or your tool's
  equivalent) is on, so the run produces an artifact — not just scrollback.

=== The results-review checklist

The run finished and the summary looks fine. This is the moment of maximum
danger, because "fine" is exactly what a lying test looks like. <<ch12>> is the
full treatment.

* [ ] *You looked at distributions, not averages.* p50, p95, p99 — and the gap
  between them. A tidy average with a p99 ten times the p50 is a finding, not a
  pass.
* [ ] *Coordinated omission is ruled out.* Open model confirmed;
  `dropped_iterations` near zero; if closed-model, latency during stalls was
  corrected or the caveat travels with the number (<<ch12>>).
* [ ] *Errors were read, not just counted.* 0.4 percent errors that are all
  timeouts on one endpoint in the last two minutes is a different story from
  0.4 percent spread evenly.
* [ ] *The whole timeline was inspected.* Latency-over-time, not just the
  end-of-run aggregate. Degradation that starts at minute 40 of a soak vanishes
  into a summary statistic.
* [ ] *Server-side telemetry corroborates.* The story the load tool tells
  (latency rose at 800 rps) matches a story the server tells (connection pool
  saturated at 800 rps). One-sided results are hypotheses.
* [ ] *Run-to-run noise is accounted for.* You know this scenario's normal
  variance from repeated runs; a 6 percent swing means nothing if the noise
  floor is 8 percent. Compare against a baseline mechanically, not by memory:

[source,console]
----
$ loadr run perf/api.yaml --summary-export current.json
$ loadr compare baseline.json current.json --assert --max-regression p95=10%
----

`compare --assert` fails on direction-aware deltas — a p95 that got 33 percent
worse fails even when every absolute threshold still passes. That gap between
"thresholds passed" and "we just got 20 percent slower" is where regressions
hide in CI (<<ch14>>).

* [ ] *The result is reproducible.* If it drives a real decision, you ran it at
  least twice and the runs agree within the noise floor.

=== The "is this number a lie?" checklist

The thirty-second sniff test before a number leaves your keyboard and enters a
slide deck. Any "no" below means the number ships with a caveat, or doesn't
ship.

* [ ] Is it a *percentile*, not an average?
* [ ] Do I know the *load model* (open or closed) that produced it?
* [ ] Do I know the *error rate* alongside it? (Fast failures flatter latency.)
* [ ] Was the system *warm* and the measurement window past the ramp?
* [ ] Was the generator *healthy* — no dropped iterations, no saturated client?
* [ ] Does *server-side data* agree with it?
* [ ] Is the *environment delta* from production attached to it?
* [ ] Would a *second run* have produced the same number, within noise?
* [ ] Can someone else *reproduce* it from what I've recorded — scenario file,
  version, environment, data set?

[TIP]
====
The fastest liars are the flattering numbers. When a result is worse than
expected, engineers instinctively audit it; when it's better than expected,
they instinctively believe it. Run this checklist *hardest* on the results you
like most.
====

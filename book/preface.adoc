== Preface

I have watched a lot of load tests lie.

Not maliciously — nobody sets out to deceive themselves about how their system
behaves under pressure. But it happens constantly, and it happens to good
engineers. A team spends a week building a test, runs it the night before a big
launch, sees a reassuring "average response time: 42ms," ships with confidence,
and then watches the site fall over at a tenth of the load the test supposedly
cleared. The test wasn't wrong because the team was careless. It was wrong
because performance testing is a discipline full of quiet traps, and almost
nobody is ever taught the discipline. They're taught a tool.

That's the gap this book exists to close.

There are excellent books about the systems side of performance — how to read a
flame graph, how to reason about a CPU that's saturated. There are excellent
books about reliability culture — SLOs, error budgets, the whole vocabulary that
SRE gave us. And there is accurate, current documentation for every load-testing
tool worth using. What there isn't, as far as I can find, is a single book that
takes an engineer from "why does any of this matter" all the way to "here is a
correct, automated, production-representative test whose numbers I can defend in
a room full of skeptical people" — and does it without assuming you already know
which questions to ask.

=== What this book is

This is a practitioner's field guide. It is organized the way you actually do
the work: understand *why* performance matters and learn the vocabulary; *design*
a test that reflects reality; *execute* it with the right tool at the right
scale; *observe and analyze* the results honestly; and finally *operationalize*
the whole thing so performance becomes a property you measure on every change
instead of a heroic effort you perform once a quarter.

The book is opinionated about method and deliberately even-handed about tools.
You will see the same idea expressed in several load testers, because the idea is
what matters and you should be able to carry it to whatever your team already
runs.

=== A note on the examples

Most of the runnable examples in this book use *loadr*, an open-source,
single-binary load-testing tool. I chose it for two honest reasons and one I'll
own up to.

The honest reasons: its scenario model maps cleanly onto the concepts each
chapter is teaching, so the example doesn't fight the lesson; and it ships with
the analysis machinery — proper percentiles, histogram merging, regression
comparison — that this book argues you should demand from any tool, which makes
it convenient for showing you what "good" looks like.

The one I'll own: I wrote loadr. So wherever a concept could be misread as a sales
pitch, I've made a point of showing the equivalent in *k6*, *JMeter*, *Gatling*,
or *Locust* alongside it. The goal is never "use my tool." The goal is "understand
the thing, then use whatever tool you like." If you finish this book and go run
better k6 tests, I'll consider it a success.

Every example lives in a companion Git repository, and every listing is exercised
in CI so the code can't quietly rot as the tools evolve. When you see a command,
you can run it.

=== Who should read this

If you can write a bit of code, read a dashboard, and you've ever been the person
who has to answer "will it hold up?" — this book is for you. That's engineers,
SREs, performance and QA engineers, and the tech leads and managers who have to
turn "fast enough" into a number and then defend it. I assume you're comfortable
on a command line and can read YAML and a little JavaScript. I do not assume you
know any performance testing, and I very much do not assume you know statistics —
this book teaches the handful of statistical ideas that matter and refuses to
drown you in the rest.

=== How to read it

The five parts build on each other, and if performance testing is new to you,
read them in order at least once — Part I and the statistics in Chapter 2 are
load-bearing for everything after. If you're already practicing and want to plug
specific gaps, each chapter is written to stand on its own, and Chapter 12
("Reading the Results Without Fooling Yourself") is the one I'd send a colleague
to first.

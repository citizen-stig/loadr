== Continuous and Production Performance

In the spring of 2008, Facebook shipped Chat to roughly seventy million users,
and launch day was — from the infrastructure's point of view — boring. That was
the achievement. For weeks beforehand, ordinary Facebook pages had been quietly
opening connections to the not-yet-visible chat backend and exchanging invisible
messages, the volume turned up gradually by a configuration flag, until the
system was carrying production-scale chat traffic while no user could see a
single pixel of the feature. By the time the UI appeared, the load test had
already been running — in production, on real traffic, for weeks. They called it
a *dark launch*, and it remains the clearest expression of this chapter's
thesis: the most truthful load test you will ever run is the one that runs in
production.

Everything before this point in the book has treated the load test as something
that happens *before* the change ships: model the workload (<<ch03>>), define
the objectives (<<ch04>>), run the test, gate the pipeline (<<ch15>>). All of
that is necessary, and none of it is sufficient, because every pre-production
test is a model of production, and models leave things out. Production has the
real data volume and the real data skew — the customer with 400,000 order lines
that your seeded database doesn't. It has the real cache hit ratios, the real
traffic mix drifting week by week, the real noisy neighbors, the real
configuration that diverged from staging eleven deploys ago. <<ch05>> was about
narrowing that gap. This chapter is about admitting it never fully closes, and
building the practices that keep validating performance *after* the artifact
leaves the pipeline: canaries, safe load against production, traffic replay,
chaos under load, autoscaling validation, and the feedback loop into capacity
planning.

Nothing here replaces the pre-production gates of <<ch15>> — production
techniques catch what staging can't *represent*, and staging catches what
production can't *tolerate finding*. You want both ends of the pipeline armed.

=== Canaries: the smallest honest load test

A canary deployment is a load test you didn't have to write. You route a small
slice of real traffic — 1 percent, then 5, then 25 — to the new version, and
real users generate a perfectly representative workload against it, for free.
The performance question at each step is the same one your CI gate asks: is the
new version measurably worse than the old one? The difference is that here the
evidence is real.

Two subtleties decide whether a canary tells you the truth.

*Compare cohorts, not time periods.* The naive analysis compares the canary's
p99 this hour against the fleet's p99 last hour, and time confounds everything —
traffic mix shifts, a marketing email lands, a batch job wakes up. The honest
design, which Netflix baked into its Kayenta automated-canary system, is to run
a *baseline* instance alongside the canary: same instance size, same count, same
moment in time, running the *old* code. Canary versus baseline is a controlled
experiment; canary versus "the rest of the fleet" quietly confounds instance
count, cache warmth, and load-balancer behavior. Progressive-delivery
controllers like Argo Rollouts and Flagger automate the loop: shift traffic,
query Prometheus for the canary and baseline cohorts, compare against a
threshold, promote or roll back.

*Respect the sample size.* A 1 percent canary sees 1 percent of traffic. If the
service does 500 requests per second overall, the canary sees 5; a ten-minute
analysis window gives you 3,000 requests, and its p99 is being judged on the
slowest 30 of them. That is exactly the small-sample trap from <<ch12>>: the
canary will catch a regression that doubles your latency and will happily wave
through the 15 percent p99 regression that your CI comparison gate would have
flagged. Canaries catch gross regressions with real traffic; pre-production
gates catch subtle regressions with controlled traffic. You need both.

You can also point synthetic load at the cohorts directly, which is useful when
the canary slice is too small to accumulate samples, or when you want to probe
an endpoint real traffic rarely exercises at this hour. Run the same plan
against each cohort and diff the summaries:

[source,console]
----
$ BASE_URL=https://stable.internal  loadr run perf/checkout.yaml --summary-export stable.json
$ BASE_URL=https://canary.internal  loadr run perf/checkout.yaml --summary-export canary.json
$ loadr compare stable.json canary.json --assert \
    --max-regression p95=10% \
    --max-regression error_rate=0.5
----

The plan reads its target from the environment (`base_url:
"${env.BASE_URL}"`), so the two runs are identical in every respect except the
code they hit, and the comparison exits non-zero if the canary is worse beyond
tolerance — a machine-checkable promotion gate, same discipline as <<ch15>>,
different environment.

=== Rules of engagement for load against production

Sooner or later the canary isn't enough and you need to generate *synthetic*
load against production itself — before a launch, after a migration, to validate
a capacity claim nobody quite believes. This is a legitimate, mainstream
practice. It is also how you become the outage you were trying to prevent, so it
runs under rules of engagement, and the first rule generates all the others:
*the test must not be able to cause more damage than it prevents.*

*Carry a circuit breaker in the plan itself.* Every plan aimed at production
gets abort thresholds — guardrails that kill the run the moment the target shows
distress, without waiting for a human to notice the dashboard:

[source,yaml]
----
# Every production-facing plan carries these. The delay stops a cold-start
# blip from tripping the breaker; after that, sustained errors end the run.
thresholds:
  http_req_failed:
    - { threshold: "rate<0.02", abort_on_fail: true, delay_abort_eval: 30s }
  http_req_duration:
    - { threshold: "p(99)<1500", abort_on_fail: true, delay_abort_eval: 60s }
----

*Mark your traffic.* Send a distinctive header (`X-Synthetic-Test: <run-id>`)
and user-agent on every request. It lets you filter the traffic out of business
analytics, lets on-call distinguish your load from an attack, and gives
downstream systems something to shed *first* if they get into trouble.

*Mind the side effects.* Read paths are cheap to test; write paths are where
production load testing earns its scars. Synthetic users, synthetic SKUs, and a
plan for cleanup are table stakes. The failure modes are always the
integrations: I have watched a synthetic checkout flow fire real fulfilment
webhooks at a warehouse, and the industry is full of stories of test runs that
sent tens of thousands of very real emails. Walk the full downstream graph of
every write before you turn the dial — payments, notifications, search indexing,
data pipelines that feed the finance report.

*Ramp like you mean it, and tell people.* Start at a few percent of measured
headroom, off-peak, with on-call informed, a change record filed, and a kill
switch that more than one person knows how to throw. Then ramp in deliberate
steps, watching the guardrails. The goal is a test that an SRE two teams away
would describe as "careful," not "brave."

==== Shadowing and dark launches

There is a family of techniques that get you production-truth with even less
user risk, because no user ever sees a synthetic response.

*Traffic shadowing* (or mirroring) duplicates real production requests to the
new system and discards its responses. Envoy and Istio can mirror a percentage
of traffic natively, NGINX has a `mirror` directive, and GoReplay does it at the
network level with the useful extra trick of replaying at a *multiple* of live
rate — 2x, 5x — which turns a mirror into an amplifier. Shadowing gives the new
system a perfectly realistic workload: real headers, real payloads, real
burstiness. Its limitation is writes. A mirrored `POST /orders` executed twice
is a duplicated order, so the shadow target needs stubbed side effects,
idempotency by design, or a read-only scope.

*Dark launches* are the Facebook Chat pattern: the new code path runs for real —
real reads, real writes, real work — but its output is hidden behind a flag. The
system carries the full production load of the feature while the blast radius of
a mistake is "we wasted some compute," not "users saw it." If you have a
feature-flag system, you already own the machinery; the discipline is deciding
to use it for load, weeks before launch, instead of the night after.

The structural limit of both techniques: mirrored and dark traffic can never
exceed the traffic you actually have today. For "will we survive three times
this?" you need synthetic load shaped like production — which is the next
section.

=== Production-shaped load: replaying real traffic

<<ch03>> made the argument that most load tests fail before they run, because
the workload model is a guess. In production you are holding the antidote: your
access logs *are* the workload model — real endpoints, real mix, real arrival
rate, recorded to the second. The job is to turn them back into load.

loadr does this with `convert`, which parses standard combined-format access
logs into a runnable plan:

[source,console]
----
$ loadr convert access.log -o replay.yaml
----

The output reproduces the observed traffic shape: a `constant-arrival-rate`
scenario at the log's average rate, for the log's duration, with a weighted
`random` block over the top endpoints, weighted by how often each appeared:

[source,yaml]
----
# loadr convert access.log (abridged)
name: access log replay
description: 'Imported from an access log by `loadr convert`: 18240 requests
  over 600s (avg 30.40 req/s, peak 77 req/s).'
variables:
  id: "1"                        # placeholder — see warnings
scenarios:
  replayed_traffic:
    executor: constant-arrival-rate
    rate: 30.4
    duration: 600s
    pre_allocated_vus: 77
    max_vus: 154
    flow:
      - random:
          strategy: weighted
          choices:
            - weight: 9120
              name: GET /api/items
              steps: [ { request: { method: GET, url: /api/items } } ]
            - weight: 4560
              name: GET /api/users/${vars.id}
              steps: [ { request: { method: GET, url: "/api/users/${vars.id}" } } ]
            - weight: 1824
              name: POST /api/orders
              steps: [ { request: { method: POST, url: /api/orders } } ]
----

Treat the output as a strong first draft, not a finished test, and read the
converter's warnings like a checklist — every one of them marks a place where
the log couldn't tell the whole truth. Logs record neither scheme nor host, so
you set the base URL. Numeric IDs and UUIDs are normalized into a placeholder
variable, which you replace with a feeder of real identifiers, or every request
hits the same warm cache line and your test flatters the system (<<ch05>>'s
cache-warmth trap, again). Logs don't record request bodies, so `POST` payloads
need to be reconstructed. And — the one that bites — the generated rate is the
observed *average*. In the example above the average was 30.4 requests per
second but the peak *second* hit 77. Replaying the average replays a smoothed,
gentler version of the day; for a worst-case test, re-shape the plan around the
reported peak. An average is a lie of omission here, just as it was in
<<ch01>>.

The concept ports directly. k6 converts recorded HAR files into scripts;
JMeter has shipped an Access Log Sampler for years that replays common-format
logs directly; Gatling's recorder
builds scenarios from captured browser sessions. And when you need byte-exact
fidelity rather than a re-shaped model — same headers, same bodies, same
inter-request gaps — GoReplay's capture-and-replay sits closer to the wire than
any scenario file can.

Whichever tool: re-derive the model on a schedule. Traffic mix drifts, and the
replay file you generated in March is a museum piece by September. A quarterly
re-convert, diffed against the current test plan, is a cheap way to catch your
workload model going stale — and it regularly surprises teams about what their
top endpoints even are.

=== Chaos under load

Chaos engineering without load is theater. Killing a pod at three requests per
second proves the orchestrator restarts pods; killing it at three thousand —
while connection pools are full, queues have depth, and retries have something
to amplify — is the experiment that predicts production. Netflix understood this
when Chaos Monkey started terminating instances in production during business
hours in 2011: the point was never the termination, it was the termination
*under real traffic*. If you run chaos experiments against an idle staging
cluster, you are testing the part of the system that was never going to fail.

The practical consequence: a chaos experiment needs a load test running through
it, both to create the conditions and to *measure the blast radius*. There are
two complementary places to inject the faults.

*Client-side fault injection* degrades the traffic the load generator itself
produces. In loadr this is the `faults:` block — added latency jitter and
dropped requests, injected in the generator, nothing installed on the target:

[source,yaml]
----
faults:
  latency:
    jitter: 200ms              # extra client-side delay per request
    distribution: uniform      # uniform | gaussian
  drop_rate: 0.02              # drop 2% of requests
  drop_mode: before_send       # dropped before they leave the client

thresholds:
  http_req_duration: [ "slo(95%) < 800ms" ]  # SLO holds despite the jitter
  checks: [ "rate>0.90" ]                    # graceful degradation, not collapse
  faults_injected: [ "count>0" ]             # fail if chaos silently no-oped
----

Notice that a chaos run *flips the intent of your gates*. A clean run asserts
"nothing goes wrong." A chaos run asserts "when things go wrong, the system
degrades the way we promised" — retries fire, timeouts bound the damage, the SLO
holds at a degraded-but-promised level. The `faults_injected: [ "count>0" ]`
threshold is the tell of a well-designed experiment: it fails the run if the
chaos never actually happened, because a chaos test that silently no-ops is
worse than no test — it manufactures false confidence. Turning the whole thing
off is deleting four lines of YAML, which makes this the cheap, repeatable,
CI-able end of chaos engineering.

*A nemesis in the plan* goes one step further: a second scenario whose only job
is to make the backend hurt while the first scenario represents the users you
care about. Scenario-scoped thresholds then judge the blast radius on the
steady traffic only — the injected faults are *supposed* to fail:

[source,yaml]
----
scenarios:
  steady_traffic:              # the experience we actually care about
    executor: constant-vus
    vus: 20
    duration: 2m
    flow:
      - request:
          name: browse
          url: /get
          checks:
            - { type: status, equals: 200 }
            - { type: duration, name: within slo, max: 800ms }
      - think_time: { type: uniform, min: 200ms, max: 800ms }

  fault_injection:             # the nemesis: error bursts and stalls
    executor: constant-arrival-rate
    rate: 5
    time_unit: 1s
    duration: 2m
    pre_allocated_vus: 10
    max_vus: 40
    flow:
      - random:
          strategy: weighted
          choices:
            - weight: 70
              name: error burst
              steps: [ { request: { name: inject 500, url: /status/500 } } ]
            - weight: 30
              name: stall
              steps: [ { request: { name: inject stall, url: /delay/3 } } ]

thresholds:
  # Judge only the steady traffic; abort if the blast radius explodes.
  "http_req_failed{scenario:steady_traffic}":
    - { threshold: "rate<0.05", abort_on_fail: true, delay_abort_eval: 20s }
  "http_req_duration{scenario:steady_traffic}": [ "p(95)<800" ]
  "checks{scenario:steady_traffic}": [ "rate>0.95" ]
----

*Infrastructure-level chaos* — real network partitions, killed nodes, disk
pressure, clock skew — comes from a dedicated fault-injection tool: Chaos Mesh
or LitmusChaos on Kubernetes, AWS Fault Injection Service in that cloud,
Gremlin commercially, Toxiproxy for a single dependency's network path. The k6
ecosystem has a particularly clean expression of the pairing in xk6-disruptor,
which injects faults into Kubernetes services from *inside the k6 script*, so
the same test that generates the load schedules the fault. Whatever the pairing,
the division of labor is constant: the chaos tool creates the fault; the load
tool creates the traffic and *renders the verdict*, because its thresholds are
where "the system survived" becomes a checkable claim instead of a vibe.

[WARNING]
====
Never feed a chaos run into your baseline history. A run with injected faults
that lands in the corpus your CI comparison gate uses as "normal" will make
every future regression look like an improvement. Keep chaos plans in separate
files, exported to separate result stores, and never hand one to
`loadr compare` as a baseline.
====

=== The autoscaler exam

"It autoscales" is the modern version of "it's fine" — a claim everyone makes
and almost nobody has tested. An autoscaler is a control loop, and every control
loop has lag. Add up a Kubernetes HPA's chain: the metrics pipeline samples on
an interval (the HPA sync period defaults to 15 seconds), the decision passes
through stabilization windows, the scheduler places pods, images pull,
containers start, readiness probes gate traffic, and then the application itself
warms up — JIT compilation, connection pools, caches. Call it 30 to 90 seconds
for a warm-image pod, and three to eight *minutes* for a fresh VM behind a cloud
auto-scaling group. During that entire window, your *existing* capacity absorbs
the surge alone. Autoscaling does not protect you from spikes; it protects you
from sustained growth. The spike protection is your headroom, and the size of
headroom you need is exactly "how much traffic can arrive during the lag."

The spike test from <<ch01>>'s taxonomy is the autoscaler's exam, and its shape
is designed to answer four questions: does it scale at all, how long is the lag,
does the SLO hold *during* the lag, and does scale-down behave on the way back:

[source,yaml]
----
name: spike-test
description: baseline -> 10x spike -> recovery

scenarios:
  spike:
    executor: ramping-arrival-rate     # open model — arrivals don't slow down
    start_rate: 20                     #   just because the target does (ch03)
    time_unit: 1s
    pre_allocated_vus: 100
    max_vus: 500
    stages:
      - { duration: 2m, target: 20 }    # baseline: steady-state replicas
      - { duration: 10s, target: 200 }  # spike: 10x, faster than any scaler
      - { duration: 8m, target: 200 }   # sustained: outlast the scaling lag
      - { duration: 10s, target: 20 }   # recover
      - { duration: 2m, target: 20 }    # verify recovery: catch flapping
    flow:
      - request:
          name: checkout
          method: POST
          url: /checkout
          body:
            json: { sku: "A-1", qty: 1 }
          checks:
            - { type: status, one_of: [200, 201] }

thresholds:
  http_req_failed:
    - { threshold: "rate<0.05", abort_on_fail: true, delay_abort_eval: 30s }
  http_req_duration: [ "p(95)<2000" ]
----

The same shape in k6, because the concept is tool-independent:

[source,javascript]
----
export const options = {
  scenarios: {
    spike: {
      executor: 'ramping-arrival-rate',
      startRate: 20, timeUnit: '1s',
      preAllocatedVUs: 100, maxVUs: 500,
      stages: [
        { target: 20,  duration: '2m'  },  // baseline
        { target: 200, duration: '10s' },  // spike
        { target: 200, duration: '8m'  },  // sustained — outlast the lag
        { target: 20,  duration: '10s' },  // recover
        { target: 20,  duration: '2m'  },  // verify recovery
      ],
    },
  },
  thresholds: {
    http_req_failed:   ['rate<0.05'],
    http_req_duration: ['p(95)<2000'],
  },
};
----

Three details decide whether this test means anything.

First, the *sustained window must outlast the lag*. Hold the spike for eight
minutes when scale-up takes two, and you observe both phases: the lag (where
your pre-spike replicas take the hit and the latency curve tells you whether the
SLO survived) and the post-scale steady state. Hold it for only 60 seconds
against a two-minute lag and you have tested your buffer, not your autoscaler —
the new capacity arrives after the traffic has left.

Second, use the *open model*. This whole test is meaningless with a closed,
fixed-VU workload, because the moment the undersized fleet slows down, a closed
model politely slows its arrivals to match, and the backlog that would have
triggered scaling never forms. <<ch03>>'s distinction is never more load-bearing
than here.

Third, *watch the way down*. The recovery stages exist because scale-in has its
own failure modes: connections dropped by terminating pods, and flapping — the
scaler oscillating replicas up and down around the threshold. Kubernetes
defaults its scale-down stabilization window to five minutes for exactly this
reason. Overlay the replica count on the latency timeline (the correlation
habit from <<ch11>>) and the whole story is one chart: spike, lag, latency bulge,
new replicas, recovery — and any post-spike wobble.

Run this against the real scaling *policies* — an environment with different
HPA settings, instance types, or warm-pool configuration validates someone
else's autoscaler. And when you find the lag is four minutes and the SLO only
survives ninety seconds of it, the fixes are the usual menu: more baseline
headroom, faster-starting instances, predictive or scheduled scaling for the
spikes you can see coming. Which is capacity planning — the last loop to close.

=== Closing the loop: capacity planning from both kinds of data

Capacity planning gets a full treatment in <<ch17>>; what belongs here is the
shape of the loop, because it's the "continuous" in continuous performance.
Test data and production data answer different halves of the question. A
capacity test tells you where the knee is — the load beyond which latency
degrades non-linearly. Production telemetry tells you where you *are* and how
fast you're moving toward it. Neither alone is a plan.

The test half is a parameter sweep — one plan, stepped across a load matrix,
results tabulated so the knee is a table read away:

[source,console]
----
$ loadr sweep perf/api.yaml --var vus=10,50,100,200 --duration 10m

  combo    p50      p95       p99       error rate  rps
  -------  -------  --------  --------  ----------  --------
  vus=10   18.20ms  24.90ms   31.00ms   0.00%       55.1/s
  vus=50   19.10ms  28.40ms   40.20ms   0.00%       270.8/s
  vus=100  24.70ms  61.30ms   112.5ms   0.02%       509.2/s
  vus=200  71.40ms  412.80ms  1.28s     2.31%       541.0/s
----

Read the last two rows together: doubling the offered load bought 6 percent
more throughput and a 6.7x worse p95. The knee sits between those rows — the
system's useful capacity is roughly 500 requests per second, and everything
past it is queueing, not work.

Now the production half. Say telemetry puts today's peak at 300 requests per
second, growing 8 percent month over month. Compounding forward, you cross 500
in a bit over six months. That single sentence — "at current growth we hit the
knee in Q1" — is the entire point of the exercise, and it is only possible
because both numbers exist: the knee from the sweep, the trajectory from
production. Re-run the sweep nightly or weekly (<<ch15>>'s scheduled matrix)
and the knee itself becomes a trend line; when a deploy moves it left, you find
out from a diff of two sweep rows, not from an incident review.

And the loop closes where the chapter started: production feeds the tests.
Re-derive the workload model from fresh access logs each quarter, re-shape the
sweep around the current traffic mix, re-run, re-plan. Test, ship, observe,
re-model, test. The test never really ends — it just changes which environment
it's learning from. Making an organization actually *do* this, sustainably, is
<<ch18>>.

.Key takeaways
****
* Every pre-production test is a model; production is the territory. Keep the
  CI gates of <<ch15>> *and* validate in production — each catches what the
  other can't.
* A canary is a free load test with real traffic. Compare canary against a
  same-sized, same-time *baseline* cohort, and respect the sample size: a small
  canary catches gross regressions, not subtle ones.
* Synthetic load against production runs under rules of engagement: abort
  thresholds as a circuit breaker in the plan, marked traffic, audited write
  paths, deliberate ramps, informed humans. Shadowing and dark launches buy
  production truth with near-zero user risk — but can't exceed today's traffic.
* Your access logs are the ground-truth workload model. Replay them — but fix
  what the log can't say: base URL, IDs, request bodies, and above all the
  peak-versus-average rate.
* Chaos without load is theater. Pair a fault injector with a load test whose
  flipped assertions define graceful degradation — and prove the faults
  actually fired.
* Autoscaling is a control loop with lag, and your headroom must cover the lag.
  Test it with an open-model spike whose sustained phase outlasts the scaling
  time, and watch scale-down as closely as scale-up.
* Capacity planning needs both halves: the knee from a sweep, the growth curve
  from production telemetry. Together they yield the only sentence management
  needs: "at current growth, we hit the wall in N months."
****

[.fieldcard]
.Field Card — Validating in Production
****
When staging can't represent it, test against production — carefully, with the guardrails in the plan.

Judging a canary:: diff it against a same-sized, same-time *baseline* cohort — never against last hour's fleet.
Synthetic load on prod:: carry the circuit breaker in the plan (`abort_on_fail`), mark every request (`X-Synthetic-Test`), audit write paths.
Replaying access logs:: `loadr convert` is a first draft — fix base URL, IDs, request bodies, and re-shape around the *peak* second, not the average.
Testing the autoscaler:: open-model spike; the sustained phase must outlast the scaling lag, and watch scale-down for flapping.

[source,console]
----
# canary vs baseline: same plan, diff the summaries
BASE_URL=https://canary.internal loadr run perf/checkout.yaml --summary-export canary.json
loadr compare stable.json canary.json --assert \
    --max-regression p95=10% --max-regression error_rate=0.5

loadr convert access.log -o replay.yaml            # logs -> runnable plan
loadr sweep perf/api.yaml --var vus=10,50,100,200 --duration 10m   # find the knee
----

*Remember:* abort guard `rate<0.02, abort_on_fail, delay_abort_eval: 30s`. Scaling lag: 30-90s warm pod, 3-8min fresh VM; scale-down stabilizes over 5min. Add `faults_injected: ["count>0"]` so a no-op chaos run fails. Never hand a chaos run to `loadr compare` as a baseline.
****

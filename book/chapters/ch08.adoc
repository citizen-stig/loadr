== Your First Real Load Test

The first load test almost everyone runs looks like this: point a tool at
`localhost`, crank the concurrency to something impressive, hit go, and admire
the throughput number. I've done it. You've probably done it. That test is
wrong in at least three independent ways — the target is wrong, the load model
is wrong, and the reading of the result is wrong — and each of those ways is
invisible from the output. The numbers come back looking perfectly plausible.
That's the trap: a broken load test doesn't crash. It just quietly measures
something other than what you think it's measuring.

So this chapter builds a first test the slow way, on purpose. We'll install a
tool, write the smallest scenario that's honestly useful, run it, and read
every line of the output. Then we'll make it progressively more realistic —
think time, ramps, checks, a multi-step flow — and translate the finished test
into k6 and JMeter so you can see that everything here is concept, not tool
trivia. We close with the mistakes that ambush nearly every first-timer,
including the three from the opening paragraph.

Everything from Part II is quietly at work here: the workload model from
<<ch03>>, the pass/fail gates from <<ch04>>, the environment discipline from
<<ch05>>. If you skipped ahead to this chapter to get hands-on — fine, that's
what it's for — but when a design question surfaces, I'll point you back.

=== Ten minutes to a running tool

The worked examples use loadr (see the preface for why, and <<ch07>> for the
full tooling survey). It's a single static binary — no JVM, no Node, no system
dependencies — which makes the install honest to describe:

[source,console]
----
$ curl -sSL https://github.com/levantar-ai/loadr/releases/latest/download/loadr-x86_64-unknown-linux-gnu.tar.gz | tar xz
$ sudo mv loadr-*/loadr /usr/local/bin/
$ loadr version
----

macOS (Intel and Apple Silicon) and Windows builds live on the same releases
page, or build from source with `cargo install --git
https://github.com/levantar-ai/loadr loadr-cli`. Whatever tool you choose,
apply the same standard: if installing it takes longer than writing your first
test, that's friction you'll pay every time you onboard a teammate.

[TIP]
====
Generate the tool's JSON Schema (`loadr schema > loadr.schema.json`) and point
your editor at it. Autocomplete and inline validation on scenario files turn
"why is my YAML silently ignored" — a genuine, hour-eating class of bug in
every YAML-configured tool — into a red squiggle. k6 users get the same effect
from TypeScript types; JMeter users get it from the GUI, at the cost of the
GUI.
====

=== A target worth testing

You need something to aim at, and it should not be production (yet — <<ch15>>
covers testing in production deliberately, which is different from doing it by
accident). The book's companion repository ships a small catalog service — a
Go API in front of Postgres, with a product list, a product detail endpoint,
and an order endpoint — that deploys with one `docker compose up`. It's built
to be interesting under load: it has a cache, a connection pool, and one
deliberately slow code path, so your tests will find real things.

Two rules about the target, both from <<ch05>>, both non-negotiable even for a
learning exercise:

* *Run the target and the load generator on different machines* (or at minimum
  different containers with separate CPU limits). If they share a CPU, they
  fight for it, and you can no longer tell whether the target is slow or your
  generator is starving.
* *Know what you're pointing at.* Write the base URL once, at the top of the
  file, and make it screamingly obvious. Every performance engineer eventually
  meets someone who ran a stress test against the wrong environment. Some of
  them ran it against production. One of them was me.

For the listings below I'll use `https://shop.example.com` as the target;
substitute wherever you deployed the sample service.

=== Anatomy of a scenario file

Here is the first test, complete. Read it top to bottom before running
anything — the structure is the lesson:

[source,yaml]
----
# first.yaml — the smallest honestly useful load test
name: first-test
description: 10 VUs against the product list for 30s

defaults:
  http:
    base_url: https://shop.example.com
    timeout: 10s

scenarios:
  hit_products:
    executor: constant-vus
    vus: 10
    duration: 30s
    flow:
      - request:
          name: list products
          method: GET
          url: /products
          checks:
            - { type: status, equals: 200 }
            - { type: duration, name: fast enough, max: 800ms }

thresholds:
  http_req_duration: [ "p(95)<1000" ]
  http_req_failed: [ "rate<0.01" ]
  checks: [ "rate>0.95" ]
----

Four blocks, and every declarative load-testing tool has some version of each:

`defaults`:: Things every request inherits — the base URL, a timeout, shared
headers. The timeout deserves a second look: it's not boilerplate. A load test
without timeouts will happily wait forever on a hung server and report it as
"zero errors." Ten seconds is a deliberate statement: _slower than this is a
failure, not a data point_.

`scenarios`:: Who is doing what. Each scenario names an *executor* — the load
model — plus its parameters, and a `flow`: the ordered steps one virtual user
performs per iteration. `constant-vus` with `vus: 10` means exactly ten
concurrent users looping for the duration. Hold that thought; the choice of
executor is the single most consequential line in the file, and we'll return
to it in the mistakes section.

`flow`:: The steps. Here it's one GET, with two *checks* attached: the status
must be 200, and the response must arrive within 800ms. Checks record pass or
fail into a metric without aborting anything — the request "succeeds" at the
HTTP level either way. That separation matters: a 200 that took four seconds
is a functional success and a performance failure, and you want both facts
recorded.

`thresholds`:: The pass/fail gate — the difference between a test and a number
generator (<<ch04>>). This file asserts three things: 95 percent of requests
complete under a second, fewer than 1 percent fail outright, and at least 95
percent of checks pass. If any of these is false, the run *fails*, with a
nonzero exit code a CI pipeline can act on (<<ch14>>).

Validate before you run — a schema-checked config with line-precise errors is
one of the cheapest quality gates in existence:

[source,console]
----
$ loadr validate first.yaml
✓ first.yaml is valid (1 scenario(s), 1 request(s))
----

=== Run it, and read every line

[source,console]
----
$ loadr run first.yaml

  first-test — 1 scenario(s), 30.0s

  checks.....................: 99.31% — ✓ 4445 ✗ 31
    ✓ status is 200 (2238 / 2238)
    ✗ fast enough (2207 / 2238)
  http_req_duration..........: avg=134.02ms min=41.87ms med=118.55ms
                               max=2104.90ms p(90)=201ms p(95)=287ms p(99)=1310ms
  http_req_failed............: 0.00% — ✓ 0 ✗ 2238
  http_reqs..................: 2238 (74.6/s)
  iterations.................: 2238 (74.6/s)
  vus........................: value=10 min=10 max=10

  thresholds:
    ✓ http_req_duration: p(95)<1000 (observed: 287.44)
    ✓ http_req_failed: rate<0.01 (observed: 0.00)
    ✓ checks: rate>0.95 (observed: 0.99)
----

The run passed. Now read it the way a practitioner does, which is mostly a
list of refusals:

*Refuse the average.* `avg=134ms` is the least informative number on the
screen, for all the reasons <<ch01>> and <<ch02>> hammered on. The story is in
the spread: median 118ms, p95 287ms, p99 1.3 seconds, max 2.1 seconds.
Something in this service occasionally takes ten to fifteen times its typical
latency. At 30 seconds of data you can't say what — but you now have a
question, and a first load test that produces a *question* is doing its job.

*Read the failed checks, not just the rate.* Thirty-one requests blew the
800ms check. 99.31 percent sounds fine; "thirty-one requests took over 800ms
in half a minute at trivial load" sounds like what it is — a lead. Notice that
`http_req_failed` is zero: every one of those slow responses was a perfectly
valid 200. If you'd only asserted on errors, this run would have looked
flawless.

*Distrust the throughput.* 74.6 requests/second is *not* the service's
capacity. It's the arithmetic consequence of 10 users in a closed loop whose
iterations took ~134ms each — Little's Law wearing a trench coat
(<<ch02>>). Change nothing about the server and double the VUs, and
throughput will roughly double. This number describes your test, not your
system. Capacity is a different test with a different shape (<<ch16>>).

*Note the exit code.* All thresholds passed, so the process exited `0`. Had
any failed, it would exit `99`. That single convention is the seed of every
automated performance gate you'll build in <<ch14>> — make sure whatever tool
you use has an equivalent, and that you've tested it fails when it should.

=== Making it realistic: think time, ramps, and a flow

Ten robots requesting the same URL in a zero-delay loop is not a workload,
it's a metronome. Real users pause to read, navigate between pages, and — most
importantly — arrive gradually. Three changes, in ascending order of
importance:

*Think time* inserts a human pause between steps. Without it, each VU
generates traffic at a rate no human ever would, and your connection reuse,
cache hit rates, and session concurrency all skew (<<ch03>> covers how to get
the numbers from production traffic; a uniform 1–4 seconds is a defensible
placeholder for browsing behavior).

*A ramp* replaces "instant full load" with a climb, a hold, and a descent. The
climb doubles as a warm-up and shows you at what point degradation begins; the
hold gives you a clean measurement window; the descent shows whether the
system *recovers*, which is a question people forget to ask until an incident
answers it for them.

*A flow* replaces the single URL with the sequence a user actually performs —
including *correlation*: pulling a value out of one response and using it in
the next request, the way a browser carries a product ID from the listing page
to the detail page.

Here's the upgraded test, all three changes in place:

[source,yaml]
----
# browse.yaml — ramp to 50 users, browse like humans, gate on the tail
name: browse-flow
description: ramp to 50 VUs, hold 5m, ramp down

defaults:
  http:
    base_url: https://shop.example.com
    timeout: 10s
    headers:
      User-Agent: loadr/1.0 (browse test)

scenarios:
  browse:
    executor: ramping-vus
    start_vus: 0
    stages:
      - { duration: 2m, target: 50 }   # ramp up
      - { duration: 5m, target: 50 }   # hold — the measurement window
      - { duration: 1m, target: 0 }    # ramp down — watch recovery
    graceful_ramp_down: 15s
    think_time: { type: uniform, min: 1s, max: 4s }
    flow:
      - request:
          name: home
          url: /
          checks: [ { type: status, equals: 200 } ]

      - request:
          name: list products
          url: /products
          checks: [ { type: status, equals: 200 } ]
          extract:
            # correlation: grab a product id to use in the next step
            - chain: product_id
              jsonpath: "$.items[0].id"
              as: int
              check: { min: 1 }

      - request:
          name: product detail
          url: "/products/${product_id}"
          checks:
            - { type: status, equals: 200 }
            - { type: duration, name: detail under 500ms, max: 500ms }

thresholds:
  http_req_duration: [ "p(95)<500", "p(99)<1200" ]
  http_req_failed: [ "rate<0.01" ]
  checks: [ "rate>0.99" ]
----

A few things worth pausing on. The scenario-level `think_time` applies between
every step, so each VU behaves like a person clicking through, not a script
sprinting. The `extract` block on the listing request pulls `$.items[0].id`
out of the JSON, coerces it to an integer, sanity-checks it, and saves it as
`product_id` for interpolation in the next URL — if the extraction fails, you
find out from the check, not from a mysterious wave of 404s three steps later.
And the thresholds now gate *two* points on the distribution, because a
comfortable p95 can coexist with a p99 your users will absolutely notice.

[WARNING]
====
Adding think time changes the arithmetic of your test, and people are
routinely surprised by how much. Fifty VUs with 1–4 seconds of think time and
a three-request flow generate on the order of 20–30 requests/second — not the
hundreds they imagined. That's correct behavior: it means your "50 users" now
approximates 50 *users* instead of 50 infinite loops. If what you actually
want to control is requests per second, more VUs isn't the answer — a
different load model is. The mistakes section picks this up.
====

=== Watch the shape, not just the summary

The end-of-run summary aggregates everything into one table, which means it
averages *across time* — and a test with a ramp has interesting time structure
by design. Was p99 elevated all along, or only above 40 VUs? Did errors appear
gradually or at a cliff? Did latency recover when the ramp descended? The
summary can't say. A time series can:

[source,console]
----
$ loadr run --summary-export results.json browse.yaml
$ loadr report results.json -o report.html
$ open report.html
----

The exported summary carries a `timeline` array — one point per snapshot
interval — and the HTML report charts throughput, latency percentiles, active
VUs, and error rate against elapsed time. (For live viewing during a run,
`loadr run --ui` serves the same charts as they happen; k6 pairs with Grafana
for the equivalent; JMeter has listener plugins.) The habit to build is
looking at the *shape* first and the aggregates second. A latency line that
tilts upward through a constant-load hold is a leak or a queue growing; a
clean step when the ramp passes a threshold is a saturation point. Both are
invisible in the summary table, and both are the actual findings. <<ch12>>
formalizes this.

=== The same test in k6 and JMeter

The concepts — executor, stages, think time, checks, thresholds, correlation —
are portable. Here's the browse test in *k6*, where the scenario is
JavaScript instead of YAML:

[source,javascript]
----
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  scenarios: {
    browse: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [
        { duration: '2m', target: 50 },
        { duration: '5m', target: 50 },
        { duration: '1m', target: 0 },
      ],
      gracefulRampDown: '15s',
    },
  },
  thresholds: {
    http_req_duration: ['p(95)<500', 'p(99)<1200'],
    http_req_failed: ['rate<0.01'],
    checks: ['rate>0.99'],
  },
};

const BASE = 'https://shop.example.com';

export default function () {
  check(http.get(`${BASE}/`), { 'home 200': (r) => r.status === 200 });
  sleep(1 + Math.random() * 3);                      // think time

  const list = http.get(`${BASE}/products`);
  check(list, { 'list 200': (r) => r.status === 200 });
  const productId = list.json('items.0.id');         // correlation
  sleep(1 + Math.random() * 3);

  const detail = http.get(`${BASE}/products/${productId}`);
  check(detail, {
    'detail 200': (r) => r.status === 200,
    'detail under 500ms': (r) => r.timings.duration < 500,
  });
  sleep(1 + Math.random() * 3);
}
----

Line for line, it's the same test. The trade is explicitness for flexibility:
in k6 the think time, correlation, and flow logic are ordinary code, which is
wonderful when the logic gets complicated and slightly noisy when it doesn't.
Pick based on who maintains the tests, not on ideology — <<ch07>> has the
longer argument.

In *JMeter*, the same intent is a Thread Group (50 threads, 120-second
ramp-up, 8-minute duration), a Uniform Random Timer (1000ms constant delay +
3000ms random) for think time, three HTTP Request samplers, Response
Assertions for the status checks, a Duration Assertion (500ms) on the detail
sampler, and a JSON Extractor on the listing sampler writing `$.items[0].id`
into a variable referenced as `${product_id}` — the identical correlation
move, assembled in a GUI and saved as XML. The `.jmx` file is not something
you'd write by hand, and that's the honest cost of JMeter: enormous
capability, painful diffs. If you inherit a pile of them, converters exist —
`loadr convert legacy-plan.jmx -o converted.yaml` imports the sampler tree,
timers, assertions, and extractors into the YAML shape above.

One genuine difference to note: JMeter's classic Thread Group ramp-up spaces
out thread *starts* but then holds thread count constant; it has no native
equivalent of a multi-stage profile without plugins (Ultimate Thread Group
fills the gap). When you translate a test between tools, translate the *load
model* first and the requests second — the requests are the easy part.

=== The ways first tests go wrong

Everything above is mechanics. What follows is the judgment — the mistakes I
see in essentially every team's first month of load testing, in descending
order of how expensive they are.

==== Accidental closed model

This is the big one, and it's subtle enough that <<ch03>> spends pages on it,
so here is just the operational warning. Every test in this chapter used
VU-based executors: N users in a loop, where a new iteration starts only when
the previous one finishes. That's a *closed* load model, and it has a
built-in anesthetic: *when the system slows down, the load backs off.* Each VU
is stuck waiting on the slow response, so it isn't sending the next request,
so the arrival rate drops exactly when the system is struggling. Real traffic
does not do this. The users arriving at your API at 2 p.m. don't politely
pause because the ones from 1:59 are still waiting.

Closed models are right when the thing you're modeling really is a bounded
population — a fixed worker pool, a batch pipeline, N kiosks in N stores. For
"traffic from the internet," you usually want an *open* model, where requests
arrive at a rate you control regardless of how the target is coping:

[source,yaml]
----
scenarios:
  steady_api_load:
    executor: constant-arrival-rate
    rate: 100            # iterations per time_unit — arrivals, not users
    time_unit: 1s
    duration: 10m
    pre_allocated_vus: 50
    max_vus: 200
    flow:
      - request:
          name: list products
          url: /products

thresholds:
  http_req_duration: [ "p(99)<300" ]
  dropped_iterations: [ "count<10" ]
----

Note `dropped_iterations` in the thresholds. In an open model, if the target
slows down and all `max_vus` workers are busy waiting, the tool cannot start
the iterations the schedule demands — and it *tells you*, by counting them as
dropped. That counter is your saturation alarm. A closed model in the same
situation reports nothing; it just quietly measures a lighter workload than
you intended. If you take one sentence from this chapter into every future
test: *know which model you're running, and be able to say why.*

==== Testing localhost

Running generator and target on one machine corrupts the test twice over:
they compete for CPU (so both look slower than they are), and the traffic
rides the loopback interface — no real network stack, no latency, no TLS
handshake cost, effectively infinite bandwidth. I've watched a service
demonstrate 9,000 req/s on a laptop and deliver 800 from a real client across
a real network, and *both numbers were honest measurements* — of completely
different systems. Localhost results are fine for smoke-testing the scenario
file itself. They are not evidence about capacity, and presenting them as
such is how load tests get their bad reputation.

==== No warm-up

The first thirty seconds of load hit a cold system: empty connection pools,
unpopulated caches, an unwarmed JIT (JVM and .NET services can be several
times slower pre-JIT), lazy initializers firing for the first time. If your
measurement window includes the cold start, your percentiles are a blend of
two regimes and describe neither. The ramp stage in `browse.yaml` mostly
solves this for free — steady-state numbers come from the hold, after two
minutes of climbing. If you run flat profiles, either prepend a short warm-up
stage or discard the head of the data. And decide *consciously* whether cold
start is in scope: for a service that autoscale spawns cold into live
traffic, the cold seconds are the finding, not the noise.

==== No gate, one run

Two smaller ones, briefly. A test without thresholds produces numbers that
mean whatever the reader wants them to mean — usually "looks fine." Write the
gate before you run, even if the first version is a guess; <<ch04>> shows how
to turn guesses into defensible objectives. And a single run is an anecdote:
run-to-run variance on identical configs is routinely 5–15 percent on shared
infrastructure. Before you conclude anything — especially "the new build is
2 percent faster" — you need repeats and the comparison discipline from
<<ch12>>.

=== Where this leaves you

You now have the complete skeleton of the craft: a declarative scenario with
an explicit load model, human-shaped behavior, correlation, machine-checkable
gates, and a time-series view of the result — plus the same test portable
across three tools. Every remaining chapter is a deepening of one of those
elements. Next, <<ch09>> takes the flow beyond plain HTTP — WebSockets, gRPC,
GraphQL, and the queue-and-database traffic that most "web" load actually
turns into.

.Key takeaways
****
* A broken load test doesn't error — it returns plausible numbers about the
  wrong thing. Correctness lives in the design, not the output.
* A scenario file has four load-bearing parts: shared defaults (including
  timeouts — always timeouts), a load model, a flow with checks, and
  thresholds. A test without thresholds is a number generator.
* Read summaries as a practitioner: ignore the average, read the percentile
  spread, treat failed checks as leads, and remember that closed-model
  throughput describes your test, not your system's capacity.
* Think time, ramps, and correlation are what separate a workload from a
  metronome — and adding them will (correctly) slash your request rate.
* The classic first-timer traps: testing localhost, measuring the cold start,
  and — subtlest of all — running a closed model against an open-world
  question, where the load politely backs off exactly when the system starts
  to fail.
* The concepts port cleanly across loadr, k6, and JMeter. Translate the load
  model first; the requests are the easy part.
****
